# 产品级开发清单发现

## 目标

这个 skill 用于在阅读真实项目后推荐“接下来还要做什么”的产品级开发清单。它不是代码审查、技术方案、执行任务拆解，也不是模块实现指导；它的产物应该接近产品负责人可以拿来排期和验收的一组事项。

除非用户明确要求阶段拆分，不要主动拆成试用版、正式版、MVP、商业化版等阶段。阶段必须来自用户目标或项目文档，而不是执行者从发布渠道过度推导。

## 核心问题

先回答四个问题，再列清单：

1. 这是什么业务：从文档、页面文案、导航、功能命名、数据模型和服务名判断产品的核心用户、核心玩法、核心资产和用户价值。
2. 代码做到什么程度：从源码、配置、服务、持久化、UI 状态、资源、entitlements、占位文案、模拟逻辑、测试和构建文件判断哪些能力是真的闭环，哪些只是半成品或占位。
3. 上线目标是什么：优先采用用户明确给出的目标；如果没有，就给出一个合理建议，并明确这是推断。
4. 还要做哪些工作：只列增量事项；如果已有功能不完整，用“完成某某闭环/优化某某体验/接入某某正式能力”的方式描述增量。

## 读取顺序

优先读取高信号材料：

- `README*`、`docs/`、`arckit/`、`spec*`、`prd*`、`tasks*`、`plan*`、release/app-store 相关材料。
- App 入口、主要页面、Feature 目录、服务层、数据模型、资源、entitlements、配置文件、测试。
- 用户可见文案，尤其是“模拟”“占位”“敬请期待”“coming soon”“API Key”“本地”“分享”“删除”“同步”“积分”“登录”“导出”“导入”等关键词。

不要只按目录结构看模块。先识别产品里的“核心资产”，再沿这个资产的完整生命周期读代码。

常见生命周期模板：

```text
创建/采集 -> 编辑/加工 -> 播放/查看/消费 -> 保存 -> 同步/备份 -> 分享/导出 -> 删除/恢复 -> 账号/权限/付费 -> 发布合规
```

不同业务要替换成自己的生命周期，例如内容创作、交易、学习、工具处理、协作、设备管理、数据分析等，但原则一样：检查用户真正完成一件事所经过的完整路径。

## 证据口径

每个候选事项先判断来源，再决定是否进入主清单：

- `源码可发现`：代码或配置直接暴露缺口，例如模拟支付、占位 UI、假数据、本地-only 持久化、缺少登录、用户手填服务密钥、分享只输出文本、同步配置未完成、能力入口存在但没有真实闭环。
- `源码支持但需要产品判断`：源码能看出体验或闭环不完整，但目标形态需要产品判断，例如列表管理体验、播放/预览体验、页面视觉层级、失败恢复、空状态和引导。
- `业务输入`：只能来自用户或产品决策，例如选择哪种后端、是否先用平台同步、是否参考某个既有产品形态、是否沉淀为团队 skill、上线阶段边界。
- `证据不足`：既不能从代码看出，也没有用户输入支撑。不要放进主清单，可以放进开放问题。

最终回答中不需要给每条事项都贴标签；默认在清单后用一小段汇总哪些来自源码、哪些来自产品判断、哪些需要业务输入。

## 事项粒度

一条事项应该是一个可以独立排期和验收的产品/业务交付块，不是技术步骤。

判断标准：

- 用户或业务方能理解这一条交付后会带来什么变化。
- 这一条通常对应一个完整闭环、一个正式能力替换、一个关键体验面、一个平台级能力，或一个明确的上线阻塞项。
- 它可以被验收，但不需要把验收所需的每个技术动作都写进标题。
- 多个技术动作只服务同一个用户可感知能力时，合成一条。
- 一句话里混入不同用户场景、不同 owner、不同上线风险时，拆成多条。

合适的抽象形态：

```text
1. 将当前模拟/占位能力替换为正式可用能力，并完成对应用户闭环
2. 补齐核心资产的分享闭环，让接收方能够直接体验或打开结果
3. 接入账号、服务端和统一授权能力，让本地临时状态升级为可运营状态
4. 优化核心操作页面的视觉层级和使用体验，使其达到目标上线版本质量
5. 为用户核心资产提供跨设备存储和同步能力，并明确当前同步方案与后续迁移提示
```

不要输出这种粒度：

```text
1. 新增一个 service
2. 改一个按钮样式
3. 写资源加载逻辑
4. 增加 loading
5. 写单元测试
```

也不要输出这种空泛表述：

```text
1. 优化体验
2. 完善后端
3. 做上线准备
```

## 反推方法

面对一堆源码、没有用户给现成清单时，按这个顺序反推：

1. 找核心业务承诺：入口页面、主路径、空状态、设置页、按钮文案、模型名、服务名通常会暴露产品真正想解决的问题。
2. 找核心资产：用户到底在创建、上传、保存、消费、购买、分享或管理什么；围绕这个资产检查完整生命周期。
3. 找伪闭环：能点但不能完成、能展示但是假数据、能保存但不能同步、能分享但对方不能体验、能生成但失败恢复差、能购买但只是模拟。
4. 找平台级缺口：账号、权限、Token、后端、支付、额度、同步、Universal Link/Deep Link、隐私声明、发布配置。
5. 找体验债：已有功能但形态明显不适合目标版本，例如主路径视觉层级混乱、核心资产列表难管理、播放/预览反馈不足。只有影响核心闭环时才进入主清单。
6. 做事项合并：把技术实现细节合并成产品事项，把泛化口号落到具体闭环或体验面。
7. 标注边界：把源码能证明的、需要产品判断的、必须问用户的部分区分开。

## 输出格式

默认输出：

```text
我判断这个项目的核心业务是……；当前代码已经做到……，但距离目标上线状态还缺这些产品级事项。

1. ……
2. ……
3. ……

证据口径：……可以从源码或配置直接发现；……属于源码支持但需要产品判断；……属于业务输入，源码本身无法推导。

开放问题：如果没有会影响清单准确性的关键问题，再列。没有就不列。
```

不要默认输出“标题 + 子列表”。如果需要组织信息，优先把范围和约束揉进单条事项，而不是展开成实现步骤。

## 避免事项

- 不要把用户提供的某个项目案例原样写成通用模板；案例只能用于抽象方法，不能绑死具体业务。
- 不要把已经开发完成的功能写进“还要做什么”；只写增量。
- 不要把技术方案、测试清单、重构建议、性能优化自动提升为产品清单，除非它们阻断上线目标。
- 不要因为 App Store、TestFlight、MVP 等词自动拆阶段；阶段必须来自用户目标。
- 不要先编译或跑测试来代替阅读代码。只有为了确认可运行性或用户明确要求时，才把编译作为辅助证据。
- 不要为了显得完整而补泛化事项，例如“埋点”“崩溃监控”“客服反馈”“多语言”，除非源码或上线目标显示它们是当前版本的必要缺口。
