# 代码质量审查维度

---

## 一、正确性维度

### 逻辑错误

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 条件反转 | `if (!isAdmin)` 写成 `if (isAdmin)` | 逻辑推演 + 测试覆盖 | 添加断言注释说明条件含义 |
| 短路求值误用 | `a && b()` 本意是 `a \|\| b()` | 代码审查 + 边界测试 | 明确意图，必要时拆分 |
| 运算符优先级 | `a + b * c` 本意是 `(a + b) * c` | 静态分析(lint) | 加括号明确优先级 |
| 类型隐式转换 | `[] == false` 为true | strict mode + TypeScript | 使用 `===`，禁止隐式转换 |

### 边界遗漏

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 空值未处理 | `user.name.length` user为null | TypeScript strictNullChecks | 可选链 `user?.name?.length` |
| 数组越界 | `arr[arr.length]` | 边界测试 | `arr[arr.length - 1]` |
| 整数溢出 | 大数计算精度丢失 | BigInt / decimal.js | 使用精确数值库 |
| 并发竞态 | 两个请求同时修改同一数据 | 并发测试 | 乐观锁/悲观锁/原子操作 |

### 竞态条件

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| check-then-act | 先检查余额再扣款，中间可被插入 | 并发测试 | 原子操作/数据库事务 |
| 读后写不一致 | 读取后基于旧值写入 | 版本号/时间戳校验 | 乐观锁 |
| 双重检查锁失效 | 单例模式在多线程下失效 | 并发压测 | 使用语言内置单例机制 |

## 二、可维护性维度

### 命名

| 指标 | 阈值 | 检测方法 |
|------|------|---------|
| 变量名单词数 | ≥2个有意义的词 | `grep -E '\b[a-z]\b'` 查单字母变量 |
| 函数名动词开头 | 是 | 代码审查 |
| 布尔值is/has/can开头 | 是 | 代码审查 |
| 常量全大写 | 是 | lint规则 |
| 无缩写 | 除广为人知的(id/url/http)外 | 代码审查 |

### 复杂度

| 指标 | 阈值 | 检测方法 |
|------|------|---------|
| 圈复杂度 | ≤10 | `eslint complexity` / `lizard` |
| 认知复杂度 | ≤15 | SonarQube |
| 函数行数 | ≤30 | lint规则 |
| 类行数 | ≤300 | lint规则 |
| 文件行数 | ≤500 | lint规则 |

### 重复

| 指标 | 阈值 | 检测方法 |
|------|------|---------|
| 重复代码块 | ≤3% | SonarQube / jscpd |
| 复制粘贴代码 | 0 | git blame分析 + 代码审查 |

### 耦合

| 指标 | 阈值 | 检测方法 |
|------|------|---------|
| 传入耦合(Ce) | 越低越好 | 结构分析工具 |
| 传出耦合(Ca) | 越低越好 | 结构分析工具 |
| 不稳定性(I) | Ce/(Ce+Ca)，核心模块应<0.5 | 手动评估 |
| 依赖数量 | 单个模块≤5个直接依赖 | 代码审查 |

## 三、可读性维度

### 注释必要性

| 场景 | 是否需要注释 | 示例 |
|------|-------------|------|
| 解释"为什么" | ✅ 需要 | `// 使用乐观锁而非悲观锁，因为读多写少` |
| 解释"做什么" | ❌ 不需要，应让代码自解释 | `// 检查用户是否成年` → `isAdult(user)` |
| 标记TODO | ✅ 需要（但需限期清理） | `// TODO: 替换为批量查询 [2026-06-30]` |
| 解释算法 | ✅ 需要 | `// 快速排序的分区逻辑` |
| 法律/合规声明 | ✅ 需要 | `// 版权声明` |

### 代码自解释程度

| 检查项 | 好的做法 | 坏的做法 |
|--------|---------|---------|
| 函数名 | `calculateMonthlyPayment()` | `calc()` |
| 变量名 | `daysSinceLastLogin` | `d` |
| 常量名 | `MAX_RETRY_ATTEMPTS = 3` | `3` |
| 参数名 | `formatDate(date, format)` | `formatDate(d, f)` |
| 返回值 | `const isValid = validateEmail(email)` | `const r = check(e)` |

### 函数长度

| 行数 | 评价 | 建议 |
|------|------|------|
| 1-10 | ✅ 优秀 | 保持 |
| 11-20 | ✅ 良好 | 可以接受 |
| 21-30 | ⚠️ 一般 | 考虑拆分 |
| 31-50 | ❌ 过长 | 必须拆分 |
| >50 | ❌❌ 严重 | 立即拆分 |

### 参数数量

| 参数数 | 评价 | 建议 |
|--------|------|------|
| 1-2 | ✅ 优秀 | 保持 |
| 3 | ✅ 良好 | 可以接受 |
| 4 | ⚠️ 一般 | 考虑封装为对象 |
| ≥5 | ❌ 过多 | 必须封装为对象 |

## 四、效率维度

### 不必要的计算

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 循环内重复计算 | `for(i) { compute(x) }` compute(x)不变 | profiling | 提取到循环外 |
| 未使用memo | React组件每次渲染都重新计算 | React DevTools Profiler | useMemo/useCallback |
| 过度序列化 | 每次请求都JSON.stringify相同对象 | profiling | 缓存序列化结果 |

### 内存泄漏

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 未清理事件监听 | addEventListener未remove | Chrome Memory面板 | useEffect cleanup / AbortController |
| 闭包引用大对象 | 定时器回调引用了大对象 | Heap Snapshot对比 | 弱引用 / 及时clearInterval |
| 全局缓存无上限 | Map/Object只增不减 | 长时间运行后内存持续增长 | LRU缓存 / 设置上限 |

### N+1查询

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 循环内查询 | `for(id of ids) { db.find(id) }` | SQL日志/ORM debug | `db.findAll({ where: { id: ids } })` |
| 关联数据逐条加载 | 显示订单列表，每个订单单独查用户 | 查询次数=N+1 | eager loading / JOIN |

### 大O问题

| 检查模式 | 描述 | 检测方法 | 修复建议 |
|---------|------|---------|---------|
| 嵌套循环 | O(n²)排序/查找 | 数据量>1000时明显变慢 | 使用Map/Set O(1)查找，或排序后二分 |
| 递归无记忆化 | Fibonacci/树遍历重复计算 | 递归深度>30时性能骤降 | 记忆化 / 改为迭代 |
| 字符串拼接 | `for(s += x)` O(n²) | 长字符串拼接慢 | 数组join / StringBuilder |
