# 架构决策记录（ADR）模板

## ADR编号规则

### 格式

```
ADR-{序号}-{简短标识}
```

- **序号**：从 `001` 开始，三位数字，连续递增，不跳号
- **简短标识**：英文小写，短横线分隔，2-5个词，描述决策核心内容

### 示例

| ADR编号 | 说明 |
|---------|------|
| ADR-001-use-postgresql-as-primary-db | 选择PostgreSQL作为主数据库 |
| ADR-002-adopt-event-driven-architecture | 采用事件驱动架构 |
| ADR-003-split-order-service | 拆分订单服务 |
| ADR-004-use-redis-for-session-cache | 使用Redis做会话缓存 |

### 编号管理规则

1. **不回收**：即使ADR被废弃，编号也不复用
2. **不跳号**：按顺序递增，不预留号段
3. **唯一性**：简短标识不可重复，即使状态为"已废弃"
4. **索引文件**：在仓库根目录维护 `docs/adr/README.md`，列出所有ADR编号、标题、状态

## 各Section填写指南

### 上下文（Context）

**目的**：让6个月后的自己或新团队成员能理解"当时为什么要做这个决策"。

**填写要点**：
- 背景要写清楚"问题是什么"，而不是"方案是什么"
- 约束条件必须是客观事实，不是主观偏好
- 驱动力要按优先级排序，最重要的放前面
- 避免省略"显而易见"的信息——对你显而易见的，对别人未必

**示例**：
> **背景**：当前系统使用单体MySQL架构，日均订单量从1万增长到50万，高峰期数据库CPU持续>90%，读写延迟>500ms。
>
> **约束条件**：团队无分布式数据库运维经验；3个月内必须支撑双十一（预估峰值200万/日）；合规要求订单数据不可丢失。
>
> **驱动力**：1. 系统可用性（双十一不可宕机）；2. 读写性能；3. 团队可维护性；4. 成本控制

### 决策（Decision）

**目的**：一句话说清楚"我们决定做什么"。

**填写要点**：
- 必须是明确的、可执行的陈述，不是模糊的方向
- 避免写"我们决定考虑X"——决策就是决策，不是考虑
- 如果决策有条件限定，明确写出条件

**示例**：
> 我们决定采用PostgreSQL作为主数据库，配合Redis做热点数据缓存，暂不引入分库分表中间件。

### 可选方案（Alternatives）

**目的**：展示你认真考虑过其他选项，而不是一拍脑袋。

**填写要点**：
- 至少列出2个方案（含最终选择的方案）
- 每个方案的优势/劣势要客观，不要故意弱化未选方案
- "什么都不做"也是一个合法方案，不要遗漏

**示例**：
> **方案A：保持MySQL+读写分离**
> - 优势：零迁移成本，团队熟悉
> - 劣势：主库写瓶颈无法解决，读写分离有复制延迟
>
> **方案B：迁移到PostgreSQL**
> - 优势：更强的并发处理能力，JSONB支持灵活schema，分区表原生支持
> - 劣势：迁移成本中等，团队需要学习
>
> **方案C：引入TiDB分布式数据库**
> - 优势：水平扩展，MySQL协议兼容
> - 劣势：运维复杂度高，团队无经验，成本高

### 权衡分析（Tradeoff Analysis）

**目的**：用结构化方式展示各方案在不同维度上的表现差异。

**填写要点**：
- 六个维度（开发效率/运行性能/团队适配/长期维护/社区生态/可替代性）是必填项
- 可根据决策特点增加额外维度
- 使用相对评价（优/中/差 或 高/中/低），不要只写好话
- 关键权衡点要解释"为什么接受这个代价"

### 结论（Conclusion）

**目的**：用2-3句话把权衡分析的结果翻译成决策理由。

**填写要点**：
- 要引用权衡分析中的具体维度
- 说明"虽然X方案在Y维度更好，但因为Z原因我们选择了当前方案"

**示例**：
> 综合权衡，PostgreSQL在运行性能和长期维护上优于MySQL，在团队适配上优于TiDB。虽然迁移有短期成本，但PostgreSQL的学习曲线平缓，且分区表能力可支撑中期增长。

### 后果（Consequences）

**目的**：诚实地记录决策带来的所有影响，尤其是负面后果。

**填写要点**：
- 正面后果不要夸大，负面后果不要回避
- 风险必须附带缓解措施，否则就是不负责任
- 行动项要具体到可执行，不要写"后续优化"

## ADR生命周期管理

### 状态流转

```
提议 ──→ 已接受 ──→ 已废弃
  │                   ↑
  │                   │
  └─────→ 已替代 ─────┘
              │
              ↓
         指向新ADR
```

### 各状态说明

| 状态 | 含义 | 允许的后续状态 |
|------|------|----------------|
| **提议** | 决策正在讨论中，尚未最终确认 | 已接受、已废弃 |
| **已接受** | 决策已被采纳并开始执行 | 已废弃、已替代 |
| **已废弃** | 决策不再适用，但保留记录 | 无（终态） |
| **已替代** | 决策被新的ADR取代 | 无（终态，必须指向替代ADR） |

### 状态变更规则

1. **提议→已接受**：需决策者签字确认，更新决策日期
2. **已接受→已废弃**：需在ADR中记录废弃原因和废弃日期
3. **已接受→已替代**：必须创建新ADR，并在旧ADR中记录"被ADR-XXX替代"
4. **不可逆**：已废弃/已替代的ADR不可恢复为已接受（如需恢复，创建新ADR引用旧ADR）

### ADR文件管理

```
docs/
  adr/
    README.md              # ADR索引
    ADR-001-use-postgres.md
    ADR-002-event-driven.md
    ADR-003-split-order.md
    template.md            # 本模板
```

### 索引文件示例（README.md）

```markdown
# 架构决策记录索引

| 编号 | 标题 | 状态 | 决策日期 | 替代ADR |
|------|------|------|----------|---------|
| ADR-001 | 选择PostgreSQL作为主数据库 | 已替代 | 2024-03-15 | ADR-005 |
| ADR-002 | 采用事件驱动架构 | 已接受 | 2024-04-01 | - |
| ADR-003 | 拆分订单服务 | 已接受 | 2024-05-10 | - |

```

## 常见ADR示例

### 示例1：技术选型——前端框架选择

```markdown
# ADR-006: 选择React作为前端框架

## 元数据

| 字段 | 值 |
|------|-----|
| 状态 | 已接受 |
| 提出日期 | 2024-09-01 |
| 决策日期 | 2024-09-05 |
| 决策者 | 张三/技术负责人 |
| 影响范围 | 前端团队、全栈团队 |

## 上下文

### 背景
公司启动新的SaaS管理平台项目，需要选择前端框架。项目预期生命周期5年+，需要支持复杂表单、数据可视化、权限管理和多租户定制。

### 约束条件
- 前端团队8人中6人有React经验，2人有Vue经验
- 需要与现有Ant Design组件库生态兼容
- 项目需要SSR支持以优化首屏加载

### 驱动力
1. 团队开发效率（快速交付早期验证版本）
2. 长期可维护性（5年+生命周期）
3. 生态丰富度（组件库、工具链）
4. 招聘市场人才供给

## 决策

我们决定使用React 18作为前端框架，配合Next.js实现SSR，使用Ant Design 5作为组件库。

## 可选方案

### 方案A：React + Next.js
- **描述**：使用React 18配合Next.js 14框架
- **优势**：团队熟悉度高，SSR原生支持，生态最丰富，人才市场最大
- **劣势**：运行时包体积较大，学习曲线对新人不低

### 方案B：Vue 3 + Nuxt
- **描述**：使用Vue 3配合Nuxt 3框架
- **优势**：上手更快，模板语法直观，包体积更小
- **劣势**：团队仅2人有经验，企业级组件库生态不如React，人才市场较小

### 方案C：Svelte + SvelteKit
- **描述**：使用Svelte配合SvelteKit框架
- **优势**：无虚拟 DOM，运行时轻量，开发体验极佳
- **劣势**：生态不成熟，企业级UI库匮乏，团队零经验，招聘困难

## 权衡分析

| 维度 | React+Next.js | Vue+Nuxt | Svelte+SvelteKit |
|------|---------------|----------|-------------------|
| 开发效率 | 高 | 高 | 中 |
| 运行性能 | 中 | 中 | 高 |
| 团队适配 | 高 | 中 | 低 |
| 长期维护 | 高 | 高 | 低 |
| 社区生态 | 高 | 中 | 低 |
| 可替代性 | 中 | 中 | 低 |

### 关键权衡点
1. 团队适配：React有6/8人熟悉，Vue仅2/8人，Svelte零——这是决定性因素
2. 生态：Ant Design是企业级项目的成熟选择，Vue/Svelte无同等替代
3. 接受React包体积较大的代价，通过代码分割和SSR缓解

## 结论

React在团队适配和生态维度显著领先，这是5年+项目的决定性因素。虽然Svelte运行性能更好，但生态和人才供给不足以支撑长期维护。

## 后果

### 正面后果
- 团队可立即进入开发，无需学习期
- Ant Design提供开箱即用的企业级组件
- 招聘池大，人员流动风险低

### 负面后果
- 包体积较大，需重视性能优化
- React函数组件+Hooks对部分老成员有适应成本

### 风险与缓解
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| React新版本破坏性变更 | 低 | 中 | 锁定版本，渐进升级 |
| 包体积影响首屏性能 | 中 | 中 | SSR + 代码分割 + 懒加载 |
| 团队Hooks使用不规范 | 中 | 中 | 制定编码规范，Code Review把关 |

### 需要的行动
- [x] 创建Next.js项目脚手架
- [ ] 制定React编码规范（含Hooks使用规范）
- [ ] 配置代码分割和懒加载策略
```

### 示例2：系统拆分——订单服务拆分

```markdown
# ADR-007: 从单体中拆分订单服务

## 元数据

| 字段 | 值 |
|------|-----|
| 状态 | 已接受 |
| 提出日期 | 2024-10-01 |
| 决策日期 | 2024-10-10 |
| 决策者 | 李四/架构师、王五/后端负责人 |
| 影响范围 | 后端团队、运维团队、QA团队 |

## 上下文

### 背景
当前电商系统为单体架构（Java Spring Boot），代码量15万行。订单模块与用户、商品、支付模块紧耦合，导致：1）订单相关变更需要全量部署，风险高；2）订单高峰影响其他模块可用性；3）团队协作冲突频繁。

### 约束条件
- 订单模块日均50万单，双十一预估200万/日
- 不能停机迁移，需要灰度切换
- 团队有Spring Cloud经验但无大规模微服务运维经验
- 数据库为共享MySQL实例

### 驱动力
1. 独立部署——降低变更风险
2. 故障隔离——订单高峰不影响其他模块
3. 团队自治——减少协作冲突
4. 弹性伸缩——按订单负载独立扩缩容

## 决策

我们决定将订单模块从单体中拆分为独立服务，采用渐进式拆分策略：先API拆分→再数据拆分→最后完全独立部署。使用Spring Cloud作为微服务框架，暂不引入服务网格。

## 可选方案

### 方案A：渐进式拆分（API先拆→数据再拆→独立部署）
- **描述**：分三个阶段逐步拆分，每个阶段可独立验证和回滚
- **优势**：风险可控，每步可回滚，团队可逐步适应
- **劣势**：周期较长，中间状态有额外复杂度

### 方案B：一次性拆分
- **描述**：直接将订单模块完整拆出为独立服务
- **优势**：一步到位，无中间态
- **劣势**：风险极高，难以回滚，团队一次性适应成本大

### 方案C：暂不拆分，优化现有单体
- **描述**：通过模块化改造、异步化等手段优化单体
- **优势**：零架构变更成本
- **劣势**：核心问题（独立部署、故障隔离）无法解决

## 权衡分析

| 维度 | 渐进式拆分 | 一次性拆分 | 优化单体 |
|------|-----------|-----------|---------|
| 开发效率 | 中 | 低 | 高 |
| 运行性能 | 高 | 高 | 中 |
| 团队适配 | 高 | 低 | 高 |
| 长期维护 | 高 | 高 | 低 |
| 社区生态 | 高 | 高 | 高 |
| 可替代性 | 中 | 低 | 高 |

### 关键权衡点
1. 风险控制：双十一前不可出故障，渐进式拆分允许每步验证和回滚
2. 团队能力：无大规模微服务经验，渐进式让团队有时间学习
3. 接受周期较长的代价，换取风险可控

## 结论

渐进式拆分在风险控制和团队适配上显著优于一次性拆分，在解决核心问题上优于优化单体。虽然周期较长，但每阶段可独立验证，符合业务对稳定性的要求。

## 后果

### 正面后果
- 订单服务可独立部署和扩缩容
- 订单高峰不再影响其他模块
- 订单团队可自治开发

### 负面后果
- 引入服务间通信复杂度（网络延迟、序列化、重试）
- 需要分布式事务处理（订单+库存+支付）
- 运维监控复杂度增加

### 风险与缓解
| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 数据拆分时数据不一致 | 中 | 高 | 双写+校验脚本+灰度切换 |
| 服务间调用超时 | 中 | 中 | 设置合理超时+重试+熔断 |
| 分布式事务处理复杂 | 高 | 高 | 优先采用最终一致性+Saga模式 |

### 需要的行动
- [ ] 定义订单服务API边界
- [ ] 实现API网关路由切换
- [ ] 设计数据拆分和双写方案
- [ ] 建立服务监控和告警
```

### 示例3：数据存储策略——日志数据存储

```markdown
# ADR-008: 使用ClickHouse存储分析型日志数据

## 元数据

| 字段 | 值 |
|------|-----|
| 状态 | 已接受 |
| 提出日期 | 2024-11-01 |
| 决策日期 | 2024-11-08 |
| 决策者 | 赵六/数据平台负责人 |
| 影响范围 | 数据团队、后端团队、运维团队 |

## 上下文

### 背景
当前使用MySQL存储业务日志（操作日志、访问日志、错误日志），日志表已积累5亿行，单表查询耗时>30s。业务需要实时分析能力（秒级响应），且日志量以每日2000万行速度增长。

### 约束条件
- 需要支持SQL查询（分析师已熟悉SQL）
- 日志数据保留周期：热数据90天，冷数据1年
- 预算有限，不能使用商业数据仓库
- 需要与现有Grafana对接做可视化

### 驱动力
1. 查询性能（秒级响应分析查询）
2. 存储成本（日志量大，成本敏感）
3. SQL兼容性（分析师无需学习新查询语言）
4. 写入吞吐（支撑2000万/日写入）

## 决策

我们决定使用ClickHouse作为日志分析存储，热数据（90天）存SSD，冷数据（>90天）自动迁移至HDD。MySQL继续存储业务数据，不再存储日志数据。

## 可选方案

### 方案A：ClickHouse
- **描述**：列式OLAP数据库，专为分析场景设计
- **优势**：分析查询极快（10-100x MySQL），压缩比高（10:1），SQL兼容
- **劣势**：不支持事务，UPDATE/DELETE性能差，运维需要学习

### 方案B：Elasticsearch
- **描述**：全文搜索引擎，常用于日志存储（ELK栈）
- **优势**：全文检索强，生态成熟（Kibana），实时索引
- **劣势**：存储成本高（压缩比低），聚合分析慢于ClickHouse，资源消耗大

### 方案C：Doris（Apache Doris）
- **描述**：国产MPP分析数据库
- **优势**：兼容MySQL协议，社区活跃（国内），支持实时和批量写入
- **劣势**：社区相对年轻，生产案例少于ClickHouse，运维经验少

## 权衡分析

| 维度 | ClickHouse | Elasticsearch | Doris |
|------|-----------|---------------|-------|
| 开发效率 | 高 | 高 | 高 |
| 运行性能 | 高 | 中 | 高 |
| 团队适配 | 中 | 中 | 低 |
| 长期维护 | 中 | 中 | 中 |
| 社区生态 | 高 | 高 | 中 |
| 可替代性 | 中 | 低 | 中 |

### 关键权衡点
1. 查询性能：ClickHouse在聚合分析场景比Elasticsearch快10-100倍，这是决定性因素
2. 存储成本：ClickHouse压缩比10:1，Elasticsearch约3:1，2000万/日写入量下成本差异显著
3. 接受ClickHouse不支持事务和UPDATE/DELETE的代价——日志场景本身就是追加写入

## 结论

ClickHouse在日志分析场景的查询性能和存储成本上显著优于Elasticsearch，SQL兼容性优于Doris的运维风险。日志追加写入的特性正好规避了ClickHouse不支持事务的短板。
```
