# 技术权衡对比矩阵

> 本文档提供常见技术选型的结构化对比，帮助架构师快速做出有据可依的决策。
> 评分标准：★★★★★（极优）★★★★☆（优）★★★☆☆（中）★★☆☆☆（差）★☆☆☆☆（极差）

---

## 评价维度说明

| 维度 | 含义 | 评估标准 |
|------|------|----------|
| **开发效率** | 从0到可交付功能的速度 | 脚手架/工具链/调试体验/默认约定 |
| **运行性能** | 吞吐量/延迟/资源利用率 | 基准测试/QPS/内存占用/冷启动 |
| **团队适配** | 现有团队的学习和适应成本 | 学习曲线/心智模型/调试难度 |
| **长期维护** | 3-5年后的维护成本 | 升级路径/破坏性变更频率/代码可读性 |
| **社区生态** | 第三方库/工具/文档/回答的丰富度 | npm/crates/pip包数/StackOverflow/文档质量 |
| **可替代性** | 如果需要迁移，锁入程度有多高 | 标准协议/数据格式/迁移工具/厂商绑定 |

## 一、前端框架对比

### 对比矩阵

| 维度 | React | Vue | Svelte | Angular |
|------|-------|-----|--------|---------|
| **开发效率** | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| **运行性能** | ★★★☆☆ | ★★★★☆ | ★★★★★ | ★★★☆☆ |
| **团队适配** | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| **长期维护** | ★★★★☆ | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| **社区生态** | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| **可替代性** | ★★★☆☆ | ★★★☆☆ | ★★☆☆☆ | ★★☆☆☆ |

### 详细说明

#### React
- **开发效率**：JSX灵活但心智模型较重，Hooks需要理解闭包和依赖数组。脚手架（Vite/CRA/Next.js）成熟。
- **运行性能**：虚拟DOM带来额外开销，大量列表渲染需手动优化（memo/useMemo/useCallback）。React 19编译器有所改善。
- **团队适配**：国内使用最广，招聘池最大。但Hooks规则和并发模式对初学者不友好。
- **长期维护**：Meta主导，版本迭代稳定但升级成本不低（Class→Hooks→Server Components）。
- **社区生态**：npm包数量最多，UI组件库（Ant Design/MUI/Chakra）选择最丰富。
- **可替代性**：JSX非Web标准，Hooks心智模型独特，迁移到其他框架需重写。

#### Vue
- **开发效率**：模板语法直观，响应式系统自动追踪依赖无需手动优化，Vite原生支持启动极快。
- **运行性能**：Vue 3响应式基于Proxy，精确更新，性能优于React虚拟DOM diff。但不如Svelte编译时优化。
- **团队适配**：学习曲线最平缓，中文文档完善，国内社区活跃。适合小团队快速上手。
- **长期维护**：尤雨溪主导，版本迭代稳定。Vue 2→3迁移有破坏性变更但官方提供迁移构建。
- **社区生态**：UI库选择丰富（Element Plus/Vuetify/Naive UI），但企业级方案少于React。
- **可替代性**：模板语法接近标准HTML，迁移相对容易。但Composition API有Vue特定心智模型。

#### Svelte
- **开发效率**：语法最简洁，无虚拟DOM概念，状态管理零样板代码。但生态不成熟导致"造轮子"场景多。
- **运行性能**：编译时生成命令式DOM操作代码，无运行时框架开销，bundle最小，性能最优。
- **团队适配**：上手极快（1-2天可上手），但高级用法（Svelte Store/Context）文档和案例较少。
- **长期维护**：版本迭代快但破坏性变更多（Svelte 4→5 runes模式大幅改变），生态库版本兼容性是风险。
- **社区生态**：npm包数量远少于React/Vue，企业级UI库（Carbon/Skeleton）成熟度不足。
- **可替代性**：Svelte语法和编译模式独特，迁移到其他框架基本等于重写。

#### Angular
- **开发效率**：完整框架自带路由/表单/HTTP/测试，大项目少做选择。但模板语法繁琐， boilerplate多。
- **运行性能**：Zone.js变更检测性能较差，大量组件时需手动优化（OnPush/Signal）。Angular 17+ Signals改善明显。
- **团队适配**：学习曲线最陡峭（RxJS/依赖注入/模块系统），需要Java/C#背景更容易适应。
- **长期维护**：Google主导，6个月一个大版本，升级工具（ng update）成熟，向后兼容性好。
- **社区生态**：企业级组件库（Angular Material/PrimeNG）完善，但整体社区活跃度不如React/Vue。
- **可替代性**：TypeScript深度绑定，RxJS心智模型独特，迁移成本极高。

### 推荐场景

| 场景 | 推荐 | 理由 |
|------|------|------|
| 大型企业SaaS（5年+） | React | 生态最成熟，招聘池最大，Ant Design等企业级UI库完善 |
| 中小型管理后台 | Vue | 上手快，开发效率高，Element Plus开箱即用 |
| 高性能/轻量应用（H5/小程序） | Svelte | 包体积最小，运行时零开销，适合性能敏感场景 |
| 大型企业内部系统（强类型/规范约束） | Angular | 完整框架+强约束，适合大团队协作和长期维护 |
| 创业公司MVP | Vue 或 React | 招聘容易，生态丰富，快速出活 |
| 开源组件库/设计系统 | React 或 Svelte | React生态最大用户基数；Svelte封装最干净 |

## 二、后端框架对比

### 对比矩阵

| 维度 | Node.js | Go | Python | Java | Rust |
|------|---------|-----|--------|------|------|
| **开发效率** | ★★★★★ | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★☆☆☆ |
| **运行性能** | ★★★☆☆ | ★★★★★ | ★★☆☆☆ | ★★★★☆ | ★★★★★ |
| **团队适配** | ★★★★☆ | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★☆☆☆ |
| **长期维护** | ★★★☆☆ | ★★★★★ | ★★★☆☆ | ★★★★★ | ★★★★☆ |
| **社区生态** | ★★★★★ | ★★★★☆ | ★★★★★ | ★★★★★ | ★★★☆☆ |
| **可替代性** | ★★★☆☆ | ★★★★☆ | ★★★★☆ | ★★★☆☆ | ★★★★☆ |

### 详细说明

#### Node.js
- **开发效率**：前后端统一语言，npm生态最丰富，Express/Fastify/NestJS选择多。但异步回调/Promise链对复杂业务逻辑的心智负担大。
- **运行性能**：单线程事件循环，I/O密集型场景表现好，CPU密集型任务阻塞事件循环。Worker Threads可缓解但有额外复杂度。
- **团队适配**：前端工程师转后端成本最低，全栈团队首选。但缺乏静态类型（需TypeScript补强）。
- **长期维护**：npm依赖地狱是最大痛点，node_modules黑洞，安全漏洞频发。TypeScript+pnpm可缓解但无法根治。
- **社区生态**：npm包数量最多（200万+），几乎所有功能都能找到库。但质量参差不齐。
- **可替代性**：JavaScript/TypeScript通用性高，但npm生态依赖深度锁入。

#### Go
- **开发效率**：编译快，部署简单（单二进制），并发模型（goroutine）直觉友好。但泛型支持较晚（1.18），错误处理冗长。
- **运行性能**：goroutine轻量（2KB初始栈），可轻松创建百万并发。GC延迟亚毫秒级。编译为原生机器码，性能接近C。
- **团队适配**：语法简单（25个关键字），1-2周可上手。但接口隐式实现、error处理模式需要适应。
- **长期维护**：编译型+强类型+向后兼容承诺，升级几乎无痛。标准库足够用，外部依赖少。
- **社区生态**：云原生生态核心语言（Docker/K8s/Prometheus均为Go编写），Web框架选择多（Gin/Echo/Fiber）。
- **可替代性**：标准HTTP库+标准库即可构建服务，框架锁入风险低。编译为单二进制，部署可替代性极高。

#### Python
- **开发效率**：语法最简洁，原型开发最快。Django/FastAPI/Web框架选择丰富。但运行时类型检查弱，大型项目重构困难。
- **运行性能**：GIL限制多线程并行，CPU密集型任务性能差。异步IO（asyncio）可提升I/O性能但心智模型复杂。PyPy可加速但兼容性有限。
- **团队适配**：学习曲线最平缓，非程序员也能上手。AI/ML领域无可替代。
- **长期维护**：动态类型导致大型项目重构风险高，依赖管理混乱（pip/poetry/conda并存）。类型注解（Type Hints）改善但非强制。
- **社区生态**：PyPI包数量仅次于npm，AI/ML/数据科学生态无可匹敌。Web生态不如Java/Node.js。
- **可替代性**：Python代码可读性强，但动态类型使自动重构困难。迁移时需要重写类型系统。

#### Java
- **开发效率**：Spring Boot脚手架成熟，但XML/注解配置繁琐，启动慢。JVM预热导致冷启动延迟。GraalVM Native Image可缓解但有限制。
- **运行性能**：JIT编译后性能接近C++，GC调优可控制延迟到亚毫秒级。但内存占用高（JVM开销），冷启动慢（秒级）。
- **团队适配**：国内使用最广，招聘池最大。但Spring体系学习曲线不低，过度封装导致调试困难。
- **长期维护**：强类型+成熟框架+向后兼容，大型项目维护成本最低。Spring版本升级路径清晰。
- **社区生态**：Maven Central包数量庞大，企业级中间件（消息队列/数据库连接池/分布式事务）生态最完善。
- **可替代性**：JVM生态锁入深，Spring框架绑定强。但JVM语言互通（Kotlin/Scala）提供一定灵活性。

#### Rust
- **开发效率**：编译慢（增量编译改善中），学习曲线最陡峭（所有权/借用/生命周期）。但编译通过后运行时bug极少。
- **运行性能**：零成本抽象，无GC，性能等于C/C++。内存安全保证，无数据竞争。适合系统级编程。
- **团队适配**：学习曲线极陡峭，从其他语言转Rust需要1-3个月。国内Rust工程师稀缺。
- **长期维护**：编译器保证内存安全和线程安全，重构信心极高。但编译器版本更新可能引入新警告。
- **社区生态**：crates.io包数量增长快但不及npm/PyPI。Web框架（Actix/Axum）成熟度中等。
- **可替代性**：无运行时依赖，编译为原生二进制。FFI可与其他语言互操作。但Rust代码迁移到其他语言等于重写。

### 推荐场景

| 场景 | 推荐 | 理由 |
|------|------|------|
| API网关/代理/高并发服务 | Go | goroutine天然并发，性能接近C，部署简单 |
| AI/ML/数据科学后端 | Python | 生态无可替代，FastAPI可满足API性能需求 |
| 企业级大型系统（百人团队） | Java | Spring生态+强类型+招聘池，大团队协作最稳 |
| 全栈小团队 | Node.js | 前后端统一语言，招聘一个全栈即可 |
| 系统级/高性能/安全关键 | Rust | 零成本抽象+内存安全，适合数据库/网络/加密 |
| 微服务基础设施 | Go | 云原生生态核心语言，K8s/Docker/Prometheus同源 |
| 快速原型/MVP | Python 或 Node.js | 开发速度最快，快速验证业务假设 |
| 实时音视频/游戏服务端 | Go 或 Rust | 低延迟+高并发，Go上手快，Rust性能极致 |

## 三、数据库对比

### 对比矩阵

| 维度 | PostgreSQL | MySQL | MongoDB | Redis | DynamoDB |
|------|-----------|-------|---------|-------|----------|
| **开发效率** | ★★★★☆ | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| **运行性能** | ★★★★☆ | ★★★★☆ | ★★★☆☆ | ★★★★★ | ★★★★☆ |
| **团队适配** | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★★☆ | ★★☆☆☆ |
| **长期维护** | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★★☆ | ★★★☆☆ |
| **社区生态** | ★★★★★ | ★★★★★ | ★★★★☆ | ★★★★☆ | ★★★☆☆ |
| **可替代性** | ★★★☆☆ | ★★★☆☆ | ★★☆☆☆ | ★★★★☆ | ★☆☆☆☆ |

### 详细说明

#### PostgreSQL
- **开发效率**：JSONB支持灵活schema，分区表/CTE/窗口函数功能强大，无需引入额外中间件。但配置项多，调优学习曲线较陡。
- **运行性能**：复杂查询优化器成熟，并行查询支持好。写入性能在高并发场景略逊于MySQL（MVCC实现差异）。分区表性能优秀。
- **团队适配**：国内使用增长快但DBA资源少于MySQL。SQL标准兼容性好，有Oracle经验者迁移容易。
- **长期维护**：MVCC无废弃问题（无需VACUUM清理回滚段），版本升级稳定。pg_upgrade大版本升级风险可控。
- **社区生态**：扩展生态丰富（PostGIS/TimescaleDB/pgvector/Citus），单一数据库可覆盖多种场景。
- **可替代性**：SQL标准兼容，迁移到其他SQL数据库成本中等。但PostgreSQL特有功能（JSONB/扩展）迁移需改造。

#### MySQL
- **开发效率**：InnoDB引擎稳定可靠，SQL语法简单。但JSON支持弱于PostgreSQL，窗口函数支持晚（8.0+），无CTE递归。
- **运行性能**：简单查询（主键/索引查找）极快。但复杂JOIN和子查询优化不如PostgreSQL。无并行查询（8.0有限支持）。
- **团队适配**：国内使用最广，DBA资源最丰富，运维工具链成熟。几乎所有云厂商提供托管服务。
- **长期维护**：主从复制成熟，但大版本升级（5.7→8.0）有破坏性变更。InnoDB无MVCC回滚段膨胀问题，但DDL操作锁表（8.0 Instant DDL改善）。
- **社区生态**：运维工具（ProxySQL/Vitess/Canal）成熟。但扩展生态不如PostgreSQL。
- **可替代性**：SQL标准兼容，但MySQL方言（LIMIT/AUTO_INCREMENT）迁移需改造。Vitess/TiDB提供兼容协议的替代方案。

#### MongoDB
- **开发效率**：无schema约束，迭代速度快。聚合管道功能强大。但复杂事务（4.0+）性能差，JOIN能力有限。
- **运行性能**：单文档读写快。但聚合管道和跨文档事务性能差。内存占用大（WiredTiger缓存默认50%内存）。
- **团队适配**：文档模型直觉友好，Mongoose ODM降低使用门槛。但聚合管道语法学习成本不低。
- **长期维护**：无schema是一把双刃剑——开发快但数据治理难。索引/查询优化需要经验。升级有兼容性风险。
- **社区生态**：Atlas云服务体验好，Drivers齐全。但社区分片/运维方案不如MySQL生态成熟。
- **可替代性**：BSON文档模型独特，迁移到关系数据库需要重新设计schema。迁移到其他文档数据库（如CouchDB）成本也不低。

#### Redis
- **开发效率**：数据结构丰富（String/Hash/List/Set/ZSet/Stream），单命令原子操作。但Lua脚本调试困难，集群模式下多Key操作受限。
- **运行性能**：单线程内存操作，10万+QPS。但大Key操作阻塞，集群模式跨Slot操作性能下降。Redis 7.0多IO线程改善网络IO。
- **团队适配**：API简单，学习曲线平缓。但集群模式和持久化配置需要深入理解。
- **长期维护**：RDB/AOF持久化方案成熟，Sentinel/Cluster高可用方案完善。但内存成本高，需要容量规划。
- **社区生态**：模块生态（RediSearch/RedisJSON/RedisTimeSeries）丰富。但部分模块需要Redis Stack或企业版。
- **可替代性**：协议简单，KeyDB/DragonflyDB/Valkey等兼容替代方案多。但Redis模块API不完全兼容。

#### DynamoDB
- **开发效率**：全托管，无需运维。但数据模型受限（仅Key-Value+二级索引），查询灵活性差。Partition Key+Sort Key设计需要提前规划。
- **运行性能**：单数位毫秒延迟，自动扩缩容。但Scan操作慢且贵，复杂查询需要精心设计二级索引。
- **团队适配**：国内使用少，学习资源少。需要理解DynamoDB特有的数据建模方式（单表设计）。
- **长期维护**：全托管，零运维。但成本随数据量线性增长，冷数据归档需要DynamoDB Streams+S3方案。
- **社区生态**：AWS深度绑定，IAM/VPC/DAX集成完善。但跨云迁移需要重新设计数据模型。
- **可替代性**：AWS专属服务，无标准协议，迁移成本极高。DynamoDB Local仅用于开发测试。自建替代（ScyllaDB/Cassandra）数据模型不同。

### 推荐场景

| 场景 | 推荐 | 理由 |
|------|------|------|
| 通用业务数据（订单/用户/商品） | PostgreSQL | 功能最全面，JSONB+分区表+扩展覆盖多场景 |
| 简单业务/团队MySQL经验丰富 | MySQL | 团队熟悉，运维资源丰富，简单场景足够 |
| 内容管理/日志/灵活schema | MongoDB | 无schema约束，迭代快，适合文档型数据 |
| 缓存/会话/排行榜/限流 | Redis | 内存级性能，数据结构丰富，缓存场景标配 |
| AWS全托管/超大规模Key-Value | DynamoDB | 自动扩缩容，单数位毫秒延迟，但仅限AWS |
| 地理位置服务 | PostgreSQL + PostGIS | PostGIS是地理信息领域的事实标准 |
| 时序数据/监控指标 | TimescaleDB（PostgreSQL扩展）或 InfluxDB | 时序优化，自动分区压缩 |
| 全文搜索 | Elasticsearch | 全文检索能力无可替代，但不要当主数据库用 |
| 混合场景（关系+文档+搜索） | PostgreSQL | JSONB+pgvector+分区表，一个数据库搞定 |

## 四、部署方案对比

### 对比矩阵

| 维度 | 自建机房 | Kubernetes | Serverless | 边缘计算 |
|------|---------|-----------|-----------|---------|
| **开发效率** | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★☆☆ |
| **运行性能** | ★★★★★ | ★★★★☆ | ★★★☆☆ | ★★★★☆ |
| **团队适配** | ★★☆☆☆ | ★★★☆☆ | ★★★★☆ | ★★☆☆☆ |
| **长期维护** | ★★☆☆☆ | ★★★☆☆ | ★★★★★ | ★★★☆☆ |
| **社区生态** | ★★☆☆☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| **可替代性** | ★★★★★ | ★★★☆☆ | ★★☆☆☆ | ★★☆☆☆ |

### 详细说明

#### 自建机房
- **开发效率**：从采购服务器到上线周期长（周级别），环境配置繁琐，无自动化基础设施。但硬件完全可控。
- **运行性能**：独占硬件资源，无虚拟化开销，性能可预测。GPU/高性能存储可按需配置。网络延迟可控（同城/跨城）。
- **团队适配**：需要专职运维团队（网络/存储/服务器），招聘和培养成本高。小团队无法承担。
- **长期维护**：硬件故障/网络割接/电力中断需要自行处理。3-5年硬件更换周期。安全补丁需手动管理。
- **社区生态**：无统一生态，依赖商业软件（VMware/F5）或自建开源方案（Proxmox/OpenStack）。
- **可替代性**：无厂商绑定，可随时迁移到云。但自建运维工具和流程的沉没成本高。

#### Kubernetes
- **开发效率**：Helm/Kustomize/ArgoCD提供声明式部署，环境一致性高。但K8s概念多（Pod/Service/Ingress/CRD），学习曲线陡。初始搭建1-2周。
- **运行性能**：容器开销极小（<5%），接近裸机性能。但etcd性能限制集群规模，Pod调度有毫秒级延迟。
- **团队适配**：需要1-2名K8s专家，开发团队需学习容器化和K8s基础概念。国内K8s人才增长快但高级人才稀缺。
- **长期维护**：K8s版本每4个月发布，需要持续升级。集群运维（etcd备份/证书轮换/节点维护）需要经验。托管K8s（EKS/AKS/ACK）可降低运维成本。
- **社区生态**：CNCF生态最丰富（Prometheus/Grafana/Istio/Argo），几乎所有基础设施都有K8s原生方案。
- **可替代性**：K8s已成为事实标准，但YAML配置和CRD产生锁入。迁移到其他编排系统（Nomad）成本不低。云厂商托管K8s有差异化。

#### Serverless
- **开发效率**：无需管理基础设施，专注业务代码。框架（Serverless Framework/SAM/CDK）成熟。但本地调试和测试体验差。
- **运行性能**：冷启动延迟（Java/Node.js 100ms-1s，Go/Rust 10-50ms）。预热策略可缓解但增加成本。执行时间受限（Lambda 15分钟上限）。
- **团队适配**：开发者无需关心运维，上手快。但调试/监控/排错思维需要转变（分布式/无状态）。
- **长期维护**：零运维，自动扩缩容，自动高可用。但成本随调用量线性增长，高频场景可能比自建更贵。
- **社区生态**：AWS Lambda/Azure Functions/Cloudflare Workers各厂商方案，跨厂商标准（OCI/Serverless Framework）存在但迁移有差异。
- **可替代性**：严重绑定云厂商，迁移成本高。Lambda Layer/Step Functions/EventBridge等组合使用后迁移更难。使用Serverless Framework可部分缓解。

#### 边缘计算
- **开发效率**：Cloudflare Workers/Deno Deploy/Vercel Edge提供快速部署。但运行时受限（无完整Node.js API），调试困难。代码体积受限（Workers 10MB）。
- **运行性能**：全球分布式部署，用户就近访问，延迟极低（<50ms全球）。但单次执行资源受限（CPU/内存/时间）。
- **团队适配**：前端工程师最容易上手（Cloudflare Workers用JS/Vite），但分布式一致性、缓存失效等概念需要理解。
- **长期维护**：全托管，无需运维。但边缘节点的数据一致性、缓存策略、灰度发布比中心化复杂。
- **社区生态**：Cloudflare Workers生态最成熟（Workers AI/R2/D1/KV），但各厂商方案差异大。
- **可替代性**：边缘运行时API不统一，Cloudflare Workers Service Worker API vs Fetch API。迁移到其他边缘平台需要改写代码。

### 推荐场景

| 场景 | 推荐 | 理由 |
|------|------|------|
| 大型企业核心系统（金融/政务） | 自建机房 或 私有云K8s | 合规要求+数据安全+性能可控 |
| 中大型互联网公司（10-100微服务） | Kubernetes | 弹性伸缩+服务治理+声明式部署 |
| 创业公司/低频API | Serverless | 零运维+按调用量付费+快速上线 |
| 全球化/低延迟/CDN增强 | 边缘计算 | 就近访问，全球亚50ms延迟 |
| AI/ML训练推理 | 自建GPU集群 或 K8s+GPU | GPU资源需求大，Serverless成本不可控 |
| 数据密集/合规敏感 | 自建+K8s混合 | 数据留本地，弹性计算上云 |
| 事件驱动/异步处理 | Serverless | 按调用付费，自动扩缩容，天然适配事件驱动 |
| IoT/实时数据 | 边缘计算+K8s中心 | 边缘实时处理+中心聚合分析 |

## 使用指南

### 如何使用本矩阵做决策

1. **明确约束**：先列出你的硬性约束（团队技能/预算/合规/时间），这些是不可变的
2. **确定优先级**：根据项目特点确定6个维度的优先级排序，不是所有维度都同等重要
3. **缩小范围**：用优先级最高的2-3个维度筛掉明显不适合的方案
4. **深入比较**：对剩余2-3个方案，结合具体项目场景做POC验证
5. **记录决策**：用ADR模板记录最终决策和权衡理由

### 评分调整建议

本矩阵的评分基于通用场景。实际使用时，需要根据你的具体情况调整：

- **团队已有深度经验**：对应框架的"团队适配"评分 +1
- **项目规模极小（<3人）**：开发效率权重调高，长期维护权重调低
- **项目规模极大（>50人）**：长期维护和社区生态权重调高
- **合规/安全要求极高**：可替代性权重调高（避免厂商绑定）
- **性能是核心竞争力**：运行性能权重调到最高

### 常见误区

| 误区 | 正确做法 |
|------|----------|
| 选"最好的"技术 | 没有最好的技术，只有最适合当前约束的技术 |
| 只看运行性能 | 6个维度综合评估，运行性能不是唯一标准 |
| 追新追热 | 新技术的"团队适配"和"长期维护"评分通常偏低，需要折现 |
| 忽略团队现状 | 团队0经验的技术，开发效率评分要打5折 |
| 过度选型 | 不要为3年后的"可能需求"牺牲当前的开发效率 |
