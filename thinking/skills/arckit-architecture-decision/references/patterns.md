# 常见架构模式库

> 本文档整理常见架构模式，提供结构化的适用性分析，帮助架构师选择合适的架构模式。
> 每个模式包含：ASCII架构图、适用场景、优势、代价、关键决策点、何时不用。

---

## 一、单体架构（Monolithic Architecture）

### 架构图

```
┌──────────────────────────────────────────────┐
│                  单体应用                      │
│                                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │  用户模块  │ │  订单模块  │ │  商品模块  │     │
│  │          │ │          │ │          │     │
│  │  控制器   │ │  控制器   │ │  控制器   │     │
│  │  服务层   │ │  服务层   │ │  服务层   │     │
│  │  数据访问  │ │  数据访问  │ │  数据访问  │     │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘     │
│       │            │            │            │
│  ─────┴────────────┴────────────┴──────      │
│            共享数据库（单一数据库）             │
│  ─────────────────────────────────────      │
│            共享进程/内存/部署单元              │
└──────────────────────────────────────────────┘
        │
        ▼
   ┌─────────┐
   │  数据库   │
   └─────────┘
```

### 适用场景

- 创业公司早期验证阶段，需要快速验证业务假设
- 团队规模<10人，模块边界清晰
- 业务领域相对简单，模块间耦合度低
- 预算有限，无法承担分布式系统的运维成本
- 内部管理系统、工具类应用

### 优势

| 优势 | 说明 |
|------|------|
| **开发简单** | 无分布式复杂度，本地调试方便，一个进程跑全部逻辑 |
| **部署简单** | 一个制品部署，无服务间协调问题 |
| **事务简单** | 本地事务即可保证一致性，无需分布式事务 |
| **调试方便** | 调用栈完整，断点一跟到底，日志集中查看 |
| **性能好** | 进程内调用，无网络开销，延迟低 |
| **成本低** | 单台服务器即可运行，运维成本低 |

### 代价

| 代价 | 说明 |
|------|------|
| **代码膨胀** | 随业务增长，代码量膨胀导致编译慢、IDE卡顿 |
| **部署耦合** | 一个模块变更需要全量部署，变更风险扩散 |
| **技术锁入** | 全局技术选型（语言/框架/数据库）难以局部更换 |
| **扩展性差** | 无法按模块独立扩缩容，资源利用率低 |
| **团队冲突** | 多团队修改同一代码库，合并冲突频繁 |
| **故障扩散** | 一个模块的内存泄漏/OOM可能导致整个应用不可用 |

### 演进路径

```
单体应用 ──→ 模块化单体 ──→ API拆分 ──→ 数据拆分 ──→ 微服务

具体步骤：
1. 模块化单体：在代码层面强制模块边界（包结构/模块系统）
2. API拆分：模块间通过API通信而非直接调用（仍在同一进程）
3. 数据拆分：每个模块独立数据库schema（仍在同一数据库实例）
4. 进程拆分：将模块拆分为独立进程（即微服务）
```

### 关键决策点

1. **模块边界是否清晰？**——如果不清晰，拆分前先做模块化，否则拆分会制造分布式大泥球
2. **当前痛点是什么？**——如果痛点不是"独立部署/独立扩缩容/团队自治"，单体可能就是正确答案
3. **团队是否做好了分布式准备？**——服务发现、链路追踪、熔断降级等基础设施是否就绪

### 何时不用

- 团队>30人且模块间变更冲突频繁——考虑微服务
- 不同模块有截然不同的负载特征（如订单高峰不影响用户查询）——考虑拆分
- 不同模块需要不同技术栈——考虑拆分
- 已有明确的独立部署需求——考虑拆分
- **最重要**：不要因为"微服务是趋势"而拆分单体——如果你的单体运行良好，保持单体

## 二、微服务架构（Microservices Architecture）

### 架构图

```
                    ┌──────────────┐
                    │   API 网关    │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼─────┐  ┌──────▼─────┐  ┌──────▼─────┐
    │  用户服务   │  │  订单服务   │  │  商品服务   │
    │            │  │            │  │            │
    │  用户API   │  │  订单API   │  │  商品API   │
    │  用户逻辑  │  │  订单逻辑  │  │  商品逻辑  │
    │  用户数据  │  │  订单数据  │  │  商品数据  │
    └──────┬─────┘  └──────┬─────┘  └──────┬─────┘
           │               │               │
    ┌──────▼─────┐  ┌──────▼─────┐  ┌──────▼─────┐
    │  用户数据库 │  │  订单数据库 │  │  商品数据库 │
    └────────────┘  └────────────┘  └────────────┘

    ┌──────────────────────────────────────────┐
    │           基础设施层                       │
    │  ┌────────┐ ┌─────────┐ ┌──────────────┐  │
    │  │ 服务发现 │ │ 配置中心 │ │ 链路追踪/监控 │  │
    │  └────────┘ └─────────┘ └──────────────┘  │
    │  ┌────────┐ ┌─────────┐ ┌──────────────┐  │
    │  │ 消息队列 │ │ 熔断降级 │ │  日志聚合    │  │
    │  └────────┘ └─────────┘ └──────────────┘  │
    └──────────────────────────────────────────┘
```

### 适用场景

- 大型系统，多团队并行开发，需要团队自治
- 不同模块有差异化的扩展需求（如订单需要弹性伸缩，用户不需要）
- 不同模块有差异化的技术栈需求（如推荐服务用Python，核心交易用Go）
- 业务边界清晰，领域模型成熟（已完成领域驱动设计）
- 团队有分布式系统运维能力

### 优势

| 优势 | 说明 |
|------|------|
| **独立部署** | 服务间独立发布，变更不扩散，部署频率可不同 |
| **独立扩缩容** | 按服务负载独立扩缩容，资源利用率高 |
| **技术异构** | 不同服务可选不同语言/框架/数据库 |
| **故障隔离** | 单服务故障不会级联到整个系统（配合熔断） |
| **团队自治** | 每个团队 owning 一个或多个服务，减少协作摩擦 |
| **渐进演进** | 可以逐个服务替换/重写，不需要整个系统推翻重来 |

### 代价

| 代价 | 说明 |
|------|------|
| **分布式复杂度** | 网络延迟、超时、重试、幂等性——单体不存在的问题 |
| **数据一致性** | 跨服务事务需要Saga/最终一致性，实现和调试复杂 |
| **运维负担** | N个服务=N倍的部署、监控、告警、排错工作量 |
| **调试困难** | 一次请求跨多个服务，需要链路追踪才能还原调用链 |
| **测试复杂** | 集成测试需要多个服务联调，环境管理成本高 |
| **性能开销** | 服务间网络通信替代进程内调用，延迟增加 |

### 常见陷阱

| 陷阱 | 表现 | 避免方式 |
|------|------|----------|
| **分布式单体** | 服务间强耦合，改一个服务需要同时改多个 | 严格API版本管理，事件驱动解耦 |
| **过细拆分** | 一个CRUD拆成一个服务，导致"纳米服务" | 一个服务=一个业务领域，不是一个表 |
| **共享数据库** | 多个服务访问同一个数据库表 | 每个服务独占数据库，通过API/事件共享数据 |
| **忽略幂等性** | 重试导致重复扣款/重复下单 | 所有写操作必须幂等，使用唯一请求ID |
| **同步链路过长** | A→B→C→D串行调用，延迟叠加 | 长链路改为事件驱动异步，或合并服务 |
| **缺少可观测性** | 故障时无法定位是哪个服务的问题 | 上线前必须完成：链路追踪+日志聚合+指标监控 |
| **过早拆分** | 业务边界未稳定就开始拆，频繁调整 | 先模块化单体，边界稳定后再拆 |

### 关键决策点

1. **服务边界怎么划？**——按业务领域（DDD限界上下文）划分，不要按技术层或数据库表划分
2. **数据怎么共享？**——服务间通过API或事件共享数据，禁止跨服务直接访问数据库
3. **一致性怎么保证？**——优先最终一致性+Saga，只有核心链路考虑强一致性
4. **基础设施准备好了吗？**——服务发现/配置中心/链路追踪/日志聚合/消息队列必须先于业务服务就绪

### 何时不用

- 团队<10人——运维微服务的基础设施成本高于业务开发成本
- 业务边界不清晰——先做模块化单体，边界稳定后再拆
- 没有分布式运维经验——微服务的问题（网络/一致性/可观测性）比单体复杂10倍
- 系统调用量低（<1000 QPS）——单体完全可以承载
- **最重要**：不要因为"大公司都在用微服务"而选择微服务——微服务的代价只有在大规模时才划算

## 三、事件驱动架构（Event-Driven Architecture）

### 架构图

```
  ┌──────────┐     事件        ┌──────────────────────┐     事件        ┌──────────┐
  │          │ ──────────────→ │                      │ ──────────────→ │          │
  │  订单服务  │   订单已创建     │    事件总线/消息队列    │   订单已创建     │ 库存服务  │
  │          │                 │                      │                 │          │
  │  发布事件  │ ──────────────→ │  Kafka/RabbitMQ/     │ ──────────────→ │  消费事件  │
  │          │   订单已支付     │  RocketMQ/Pulsar      │   订单已支付     │          │
  └──────────┘                 │                      │                 └──────────┘
                                │  ┌────────────────┐  │
                                │  │  事件存储/日志   │  │     事件        ┌──────────┐
                                │  │  (持久化+回放)   │  │ ──────────────→ │          │
                                │  └────────────────┘  │   订单已创建     │ 通知服务  │
                                └──────────────────────┘                 │          │
                                                                         └──────────┘

  特点：
  - 生产者和消费者完全解耦
  - 事件是不可变的事实记录
  - 消费者可以独立扩展和失败
  - 事件可以被多个消费者消费
```

### 适用场景

- 多个系统需要对同一业务事件做出响应（如下单后：扣库存、发通知、积分、统计）
- 需要异步处理以提高响应速度（如用户注册后异步发送欢迎邮件）
- 系统间需要解耦，生产者不需要知道消费者的存在
- 需要事件回放能力（如重建读模型、数据修复）
- 实时数据流处理（如风控、推荐特征计算）

### 优势

| 优势 | 说明 |
|------|------|
| **松耦合** | 生产者不依赖消费者，增减消费者不影响生产者 |
| **异步处理** | 耗时操作异步执行，不阻塞主流程，用户体验好 |
| **可扩展** | 消费者可以独立扩容，不同消费者的消费速度可以不同 |
| **事件回放** | 事件持久化后可以重新消费，用于数据修复/模型重建 |
| **审计追踪** | 事件是不可变的历史记录，天然提供完整的审计链 |
| **时间解耦** | 生产者和消费者不需要同时在线，消息队列作为缓冲 |

### 代价

| 代价 | 说明 |
|------|------|
| **最终一致性** | 数据不会立即一致，需要业务接受延迟 |
| **调试困难** | 事件链路是隐式的，出问题时需要跨服务追踪事件流 |
| **事件顺序** | 分布式环境下事件顺序无法保证，需要业务层处理乱序 |
| **重复消费** | 至少一次投递意味着消费者可能收到重复事件，必须幂等 |
| **架构复杂** | 引入消息队列增加运维成本和故障点 |
| **调试/测试** | 事件驱动的流程难以在单元测试中完整覆盖 |

### 实现要点

1. **事件设计**
   - 事件名使用过去时：`OrderCreated`、`PaymentSucceeded`（描述已发生的事实）
   - 事件包含足够的上下文，消费者不需要回查生产者
   - 事件schema版本化，支持向后兼容的演进

2. **幂等消费**
   - 每个消费者维护消费位移（offset），且所有写操作必须幂等
   - 使用唯一事件ID + 消费者去重表防止重复处理
   - 数据库操作使用 upsert 而非 insert

3. **事件顺序**
   - 同一聚合根的事件必须发到同一分区（Kafka用key路由）
   - 跨聚合根的顺序无法保证，业务需要容忍乱序或设计补偿逻辑
   - 使用时间戳+序列号辅助排序

4. **失败处理**
   - 死信队列（DLQ）存储无法处理的事件，人工介入
   - 重试策略：指数退避 + 最大重试次数
   - 关键事件不能丢失：消息队列配置为持久化 + 确认机制

5. **可观测性**
   - 每个事件携带trace_id，串联生产者-队列-消费者的完整链路
   - 监控消息积压（consumer lag），积压即告警
   - 事件schema变更需要兼容性检查

### 关键决策点

1. **选什么消息队列？**
   - 高吞吐+事件流+回放 → Kafka
   - 低延迟+路由灵活+ACK确认 → RabbitMQ
   - 国内生态+事务消息 → RocketMQ
   - 多租户+Geo复制 → Pulsar

2. **事件格式怎么设计？**——CloudEvents标准 vs 自定义格式。推荐CloudEvents标准，提供统一的id/source/type/time字段

3. **事件要不要存储？**——如果需要回放能力，必须持久化（Kafka自带，RabbitMQ需要额外配置Event Store）

4. **同步还是异步？**——核心链路（如支付）保持同步，非核心链路（如通知）改为异步

### 何时不用

- 强一致性要求——金融账户余额变更等需要实时一致的场景不适合纯事件驱动
- 简单的CRUD应用——事件驱动增加的复杂度不值得
- 团队没有消息队列运维经验——消息队列的问题（积压/丢失/重复）处理需要经验
- 事件链路超过3层——事件链路过长导致难以理解和调试，考虑合并服务或简化流程
- **最重要**：不要为了"解耦"而事件驱动——如果调用方只有1个且不会增加，直接调用更简单

## 四、CQRS + Event Sourcing

### 架构图

```
                          命令（写入）
                             │
                             ▼
                    ┌─────────────────┐
                    │    命令模型       │
                    │   （写模型）      │
                    │                 │
                    │  聚合根         │
                    │  业务规则校验    │
                    │  产生领域事件    │
                    └────────┬────────┘
                             │
                    领域事件  │
                             │
                    ┌────────▼────────┐
                    │   事件存储       │
                    │  (Event Store)  │
                    │                 │
                    │  事件1: 订单已创建      │
                    │  事件2: 订单已支付      │
                    │  事件3: 订单已发货      │
                    │  ...                   │
                    └────────┬────────┘
                             │
                    事件回放  │
                             │
                    ┌────────▼────────┐
                    │    查询模型       │
                    │   （读模型）      │
                    │                 │
                    │  针对查询优化的   │
                    │  物化视图/读表   │
                    └─────────────────┘
                             │
                          查询（读取）
                             │
                             ▼
                        前端/API

  核心思想：
  - 写操作通过命令修改聚合根，产生事件
  - 事件是唯一的事实来源（Source of Truth）
  - 读操作从物化视图查询，与写模型解耦
  - 任意时刻可通过回放事件重建当前状态
```

### 适用场景

- 需要完整审计追踪（金融交易、医疗记录、法律文档）
- 业务规则复杂且频繁变化（聚合根需要从事件重建状态来校验规则）
- 读写负载差异巨大（写少读多，如商品详情页）
- 需要时间旅行查询（"这笔订单在某个时间点的状态是什么"）
- 需要事件回放能力（修复bug后重建读模型、数据迁移）

### 优势

| 优势 | 说明 |
|------|------|
| **完整审计** | 所有状态变更都以事件形式记录，可回溯任意时间点的状态 |
| **读写独立优化** | 读模型可以针对不同查询场景做不同优化，不影响写模型 |
| **业务规则精确** | 聚合根从事件流重建状态，保证业务规则校验的准确性 |
| **时间旅行** | 可以查询任意历史时间点的状态，支持后悔药和合规审查 |
| **自然集成事件驱动** | 事件天然成为集成机制，与其他系统通过事件解耦 |
| **数据修复** | 修复bug后可以回放事件重建正确的状态 |

### 代价

| 代价 | 说明 |
|------|------|
| **复杂度极高** | 这是最复杂的架构模式之一，入门门槛极高 |
| **最终一致性** | 写模型和读模型之间有延迟，用户可能看到过期数据 |
| **事件版本管理** | 事件schema变更需要向后兼容，否则回放会失败 |
| **存储成本** | 所有事件永久存储，数据量只增不减 |
| **调试困难** | 当前状态分散在多个事件中，调试时需要回放才能看到 |
| **学习曲线** | 团队需要深入理解DDD、聚合根、领域事件等概念 |

### 复杂度警告

```
  ⚠️  CQRS + Event Sourcing 的复杂度警告  ⚠️

  不要低估这个模式的复杂度。常见低估的场景：

  1. 事件版本迁移：当业务规则变化导致事件格式变化时，
     需要编写upcaster将旧事件转换为新格式。
     10个事件类型就有10个upcaster，100个就有100个。

  2. 快照（Snapshot）：事件流过长时，每次从0回放太慢，
     需要定期生成快照。快照+增量事件的组合增加了复杂度。

  3. 读模型一致性：用户提交命令后立即查询，
     读模型可能还没更新完成，需要"读自己写"的特殊处理。

  4. 事件重放：每次部署新版本的读模型处理器时，
     需要从头消费事件流重建读模型，大数据量下耗时很长。

  5. 事件删除/GDPR：用户要求删除个人数据时，
     事件是不可变的，"删除"需要特殊处理（加密/截断/标记）。

  复杂度评估：
  ┌─────────────────┬───────────────┐
  │ CQRS（仅命令查询分离）│ 中等复杂度      │
  │ CQRS + 读写不同DB  │ 中高复杂度      │
  │ CQRS + Event Sourcing │ 极高复杂度  │
  │ + 多个读模型          │ 地狱级复杂度 │
  └─────────────────┴───────────────┘

  经验法则：如果你不能清晰解释为什么需要Event Sourcing，
  那你大概率不需要它。
```

### 关键决策点

1. **是否真的需要Event Sourcing？**——大多数场景只需要CQRS（读写分离），不需要Event Sourcing（事件溯源）。先问自己："我需要回放事件来重建状态吗？"如果不需要，只要CQRS就够了

2. **读写分离到什么程度？**——最轻量级：同库不同表；中等：不同数据库实例；最重：不同数据库类型（写用PostgreSQL，读用Elasticsearch/Redis）

3. **事件存储选什么？**——EventStoreDB（专用）、Kafka（事件流+持久化）、PostgreSQL（append-only表+通知机制）

4. **读模型一致性延迟可接受吗？**——大多数业务可接受1秒以内的延迟，但有些场景（如支付后查余额）需要同步处理

### 何时不用

- 简单CRUD应用——增加10倍复杂度解决不存在的问题
- 团队没有DDD经验——CQRS+ES需要深厚的领域建模能力
- 不需要审计追踪——如果只为了"读写分离"，用CQRS即可，不需要Event Sourcing
- 事件类型>50个——事件版本管理的复杂度将失控
- **最重要**：不要因为"高级"而选择CQRS+ES——这是解决特定复杂问题的方案，不是通用的架构升级

## 五、分层架构（Layered Architecture）

### 架构图

```
┌──────────────────────────────────────────────────┐
│                  表现层（Presentation）             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ REST API │  │  GraphQL │  │   gRPC   │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└──────────────────────┬───────────────────────────┘
                       │ 仅向下调用
┌──────────────────────▼───────────────────────────┐
│                  业务逻辑层（Business）              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ 用户服务   │  │ 订单服务  │  │ 商品服务  │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└──────────────────────┬───────────────────────────┘
                       │ 仅向下调用
┌──────────────────────▼───────────────────────────┐
│                  数据访问层（Data Access）           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ 用户DAO  │  │ 订单DAO  │  │ 商品DAO  │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└──────────────────────┬───────────────────────────┘
                       │
                ┌──────▼──────┐
                │   数据库     │
                └─────────────┘

  规则：上层可以调用下层，下层不能调用上层
  禁止：跨层调用、下层依赖上层、业务逻辑泄露到表现层
```

### 适用场景

- 大多数业务应用的默认选择
- 团队习惯经典的MVC/N层架构
- 业务逻辑需要与数据访问和展示解耦
- 需要替换表现层（如从REST API改为GraphQL）或数据访问层（如从MySQL改为PostgreSQL）
- 中小型项目，不需要更复杂的架构模式

### 优势

| 优势 | 说明 |
|------|------|
| **简单直观** | 每层职责清晰，新人容易理解 |
| **关注点分离** | 表现/业务/数据各层独立变化 |
| **可测试** | 业务逻辑层可以独立于数据库和API做单元测试 |
| **可替换** | 替换数据库只需要改DAO层，替换API只需要改表现层 |
| **广泛认知** | 几乎所有开发者都理解分层架构，沟通成本低 |
| **框架支持** | Spring/ASP.NET/Django等主流框架都内置分层支持 |

### 代价

| 代价 | 说明 |
|------|------|
| **层次泄露** | 业务逻辑经常泄露到表现层（Controller变胖）或数据访问层（SQL写业务逻辑） |
| **过度抽象** | 简单CRUD也要经过三层，增加不必要的接口和转换 |
| **性能开销** | 层间数据转换（DTO↔Entity↔DO）增加CPU开销和代码量 |
| **贫血模型** | Entity退化为数据容器，业务逻辑散落在Service层，违反OOP |
| **事务脚本** | Service层变成过程式的事务脚本，缺乏领域建模 |

### 关键决策点

1. **分几层？**——经典三层（表现/业务/数据）适合大多数场景。四层（加应用服务层）在复杂业务中有价值。五层以上通常是过度设计

2. **层间数据如何传递？**——DTO vs 直接传递Entity。DTO更干净但转换代码多。推荐：简单场景用Entity，复杂场景用DTO

3. **业务逻辑放哪里？**——业务逻辑必须属于业务逻辑层。如果发现Controller里有业务逻辑（if/else/计算），说明层次泄露了

4. **是否需要领域模型？**——如果业务逻辑复杂，考虑从贫血模型转向充血模型（DDD战术设计），或者直接采用六边形架构

### 何时不用

- 业务逻辑极简的CRUD应用——直接用框架脚手架生成即可，不需要严格的分层
- 业务逻辑极复杂的领域——分层架构容易沦为贫血模型+事务脚本，考虑DDD/六边形架构
- 需要极致性能——层间转换的开销不可接受时，考虑更扁平的结构
- **最重要**：分层架构是起点不是终点——当分层架构开始"痛"的时候，再考虑更复杂的模式

## 六、六边形架构/端口适配器（Hexagonal Architecture / Ports & Adapters）

### 架构图

```
                    ┌───────────────────────────────┐
                    │                               │
                    │   ┌───────────────────────┐   │
                    │   │                       │   │
     REST API  ────┤   │      领域模型          │   │ ──── MySQL
     (驱动适配器)   │   │                       │   │  (被驱动适配器)
                    │   │   ┌───────────────┐   │   │
                    │   │   │  业务实体      │   │   │
     消息队列  ────┤   │   │  领域服务      │   │   │ ──── Redis
     (驱动适配器)   │   │   │  领域事件      │   │   │  (被驱动适配器)
                    │   │   │  端口接口      │   │   │
     定时任务  ────┤   │   └───────────────┘   │   │ ──── 外部API
     (驱动适配器)   │   │                       │   │  (被驱动适配器)
                    │   └───────────────────────┘   │
                    │                               │
                    │          六边形核心             │
                    └───────────────────────────────┘

  核心原则：
  - 领域模型在中心，不依赖任何外部技术
  - 端口（Port）= 接口，定义领域需要什么能力
  - 适配器（Adapter）= 实现，将外部技术对接到端口
  - 驱动端口 = 领域被外部调用的入口（左侧）
  - 被驱动端口 = 领域调用外部的出口（右侧）
  - 依赖方向永远从外向内
```

### 适用场景

- 业务核心逻辑需要与技术实现完全解耦
- 需要支持多种输入渠道（REST/gRPC/MQ/CLI）调用同一业务逻辑
- 需要替换外部依赖（如从MySQL迁移到PostgreSQL，从Sync改为Async）
- 业务规则复杂，需要在无任何外部依赖的情况下测试
- DDD战术设计的自然搭档

### 优势

| 优势 | 说明 |
|------|------|
| **领域纯净** | 核心业务逻辑不依赖任何框架/数据库/外部服务 |
| **技术无关** | 替换数据库/API框架/消息队列只需要写新适配器 |
| **可测试性极强** | 领域逻辑可以用纯内存适配器测试，无IO依赖 |
| **业务聚焦** | 开发者先设计领域模型，再考虑技术实现，避免技术驱动设计 |
| **多渠道复用** | REST/gRPC/MQ/CLI都通过同一端口调用同一业务逻辑 |
| **架构边界强制** | 依赖方向单向（外→内），编译器可强制检查 |

### 代价

| 代价 | 说明 |
|------|------|
| **接口膨胀** | 每个端口都需要接口+实现，代码量增加30-50% |
| **学习曲线** | 团队需要理解端口/适配器/驱动/被驱动的概念 |
| **过度设计风险** | 简单CRUD场景下增加大量接口和转换，ROI低 |
| **间接层开销** | 每次调用多一层间接，调试时跳转更多层 |
| **数据转换** | 领域模型↔适配器模型之间的转换代码量大 |

### 关键决策点

1. **端口粒度怎么定？**——一个用例一个端口（细粒度）vs 一个聚合根一个端口（粗粒度）。推荐：从粗粒度开始，按需细化

2. **领域模型是否共享？**——适配器是否可以直接使用领域模型？推荐：驱动端（入站）用DTO转换，被驱动端（出站）可以用领域模型

3. **如何强制依赖方向？**——ArchUnit/Checkstyle等工具可以编译时检查依赖方向。没有工具强制，六边形架构会退化为分层架构

4. **是否需要AOP/依赖注入？**——六边形架构需要DI框架来注入适配器实现。Spring的DI天然适配六边形架构

### 何时不用

- 简单CRUD应用——六边形的接口和转换成本不值得
- 团队不理解DDD——六边形架构的价值在于保护领域模型，没有领域模型就没有保护对象
- 项目周期极短——六边形的前期设计成本高，短期项目收益来不及回收
- 只有一种输入渠道和一种存储——没有多适配器的需求，六边形就失去了核心价值
- **最重要**：六边形架构的目的是保护领域模型——如果你的项目没有需要保护的复杂领域逻辑，六边形架构就是过度设计

## 通用建议

### 架构模式不是互斥的

实际项目中，通常是多种模式的组合：

| 常见组合 | 说明 |
|----------|------|
| 微服务 + 事件驱动 | 服务间通过事件通信，解耦+异步 |
| 微服务 + 六边形 | 每个微服务内部采用六边形架构保护领域 |
| 六边形 + CQRS | 领域模型纯净（六边形），读写独立优化（CQRS） |
| 分层 + 事件驱动 | 分层架构内部用事件驱动解耦模块 |

### 架构演进原则

1. **从简单开始**：能用单体解决的不要用微服务，能用分层解决的不要用六边形
2. **痛点驱动**：架构变更应该由痛点驱动，而不是由趋势驱动
3. **渐进演进**：架构变更应该渐进式进行，每一步都可验证和回滚
4. **记录决策**：每次架构变更都用ADR记录决策理由和权衡
5. **定期审视**：每季度审视架构是否还适合当前的业务阶段
