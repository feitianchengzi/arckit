# Decision Framework

## 简介

提供三套递进式思考工具，帮助在产品决策、战略规划、方案评审等场景中提升决策质量。

**核心能力：**
1. **第一性原理拆解** - 找方向（从零重构问题）
2. **产品价值评估** - 判价值（定量评估方案）
3. **苏格拉底追问** - 拷逻辑（六层质疑验证）

## 迁移后定位

本 skill 是过程型 thinking 能力，不是角色 skill，也不直接维护 `arckit/spec/`、`arckit/tech/`、`arckit/interaction/`、`arckit/visual/` 或项目治理文档。

- 独立使用时，可以产出完整决策分析文件或对话结论。
- 嵌入其他工作流时，只输出 `decision_handoff`，由下游结果型 skill 决定如何沉淀。
- 不要把决策模板中的过程叙事原样写入结果型文档。
- 原始 `all` 迁入的模板和脚本保留为参考资源，使用前先确认是否适合当前场景。

## 使用模式

### 独立决策工作流

当用户的主要目标是判断、比较、评估、拷问或选择时，作为主 skill 使用。默认创建决策工作文件，并在对话中逐章节引导填写。

独立决策文件是过程型分析产物，统一保存到目标项目的 `arckit/decision-analysis/`。它不是稳定产品、交互、视觉或技术事实；若结论影响 Goal、Task、Roadmap、取舍承诺或外部推进事项，先交给 `arckit-pending`，再由当前项目可用的外部治理方式处理。

典型请求：

- "帮我评估这个方案"
- "这个需求值不值得做"
- "帮我想想这个问题"
- "拷问一下这个方案"
- "我们是不是问错问题了"
- "帮我从第一性原理分析"
- "用苏格拉底式提问帮我梳理"
- "做一个产品价值评估"

### 嵌入式辅助方法

当另一个主工作流已经负责最终产物，但中途出现价值判断、方案选择、假设脆弱性、风险拷问或继续/调整/停止判断时，作为辅助方法使用。

嵌入式使用规则：
- 不创建独立决策文件，除非用户明确要求。
- 不接管主工作流的目录、模板或最终产物。
- 只输出可回填的轻量 `decision_handoff`：`method`、`question`、`assumptions`、`judgement`、`risks`、`validation_next`。
- 结论交回主工作流，由主工作流写入自己的文档、记录、任务或规格。

## 工作流程

### 第一步：场景识别

根据用户描述，自动判断应该用哪个工具：

```
问题定义不清晰？
    ↓ 是
第一性原理拆解
    ↓ 否
有多个候选方案？
    ↓ 是
产品价值评估
    ↓ 否
有具体方案需要验证？
    ↓ 是
苏格拉底追问
```

**判断依据（参考 references/all-usage-guide.md 第五章）：**

| 用户场景 | 优先工具 |
|---------|---------|
| 全新领域/业务、战略转型 | 第一性原理 |
| "竞品都在做，要不要跟？" | 第一性原理 |
| 需求评审、立项决策 | 产品价值评估 |
| 有 2+ 个候选方案要选择 | 产品价值评估 |
| 方案评审、风险识别 | 苏格拉底追问 |
| 方案反复被否、团队陷入僵局 | 苏格拉底追问 → 第一性原理 |

---

### 第二步：工具选择确认

**自动路由：**
- 根据场景判断，告诉用户"这个场景适合用【工具名】"，并用 1-2 句话说明理由。
- 场景清晰、用户目标明确时直接继续，不为了选择工具而停顿。
- 只有在独立决策工作流且方法选择不确定，或用户明确要求完整文件化产物时，才询问用户是否确认工具。

**示例回复：**
> 根据你的描述，这是一个"竞品都在做，要不要跟"的场景。我会先用**第一性原理拆解**，从零重新审视这个问题，避免盲目跟随。

**手动指定（用户明确要求）：**
- "用第一性原理帮我分析" → 直接用第一性原理
- "评估一下值不值得做" → 直接用产品价值评估
- "拷问这个方案" → 直接用苏格拉底追问

---

### 第三步：创建工作文件或输出片段

嵌入式辅助方法跳过本步骤，直接输出轻量 `decision_handoff`。

独立决策工作流默认创建填写文件；嵌入式辅助方法不创建文件。

工作区结构：

```text
arckit/decision-analysis/
  INDEX.md
  YYYY/
    YYYY-MM-DD-topic-tool.md
```

**文件命名规则：**
```
arckit/decision-analysis/{YYYY}/{YYYY-MM-DD}-{topic}-{tool}.md

工具简称：
- first-principles (第一性原理)
- value-assessment (产品价值评估)
- socratic (苏格拉底追问)

示例：
- arckit/decision-analysis/2026/2026-06-15-live-commerce-first-principles.md
- arckit/decision-analysis/2026/2026-06-15-ai-assistant-value-assessment.md
- arckit/decision-analysis/2026/2026-06-15-dev-efficiency-socratic.md
```

**操作：**
1. 询问用户项目/主题名称（如果用户已在问题中提及，直接提取）。
2. 确保 `arckit/decision-analysis/INDEX.md` 和年份目录存在。
3. 从本 skill 目录下的 `templates/` 复制对应模板到目标项目 `arckit/decision-analysis/{YYYY}/`：
   ```bash
   cp <skill-dir>/templates/{模板名}.md \
      arckit/decision-analysis/{YYYY}/{YYYY-MM-DD}-{topic}-{tool}.md
   ```
   其中 `<skill-dir>` 是当前 skill 文件夹，安装后可能位于 Codex、Claude、Cursor 或项目本地 agent skills 目录；不要写死源码仓库路径。
4. 更新 `arckit/decision-analysis/INDEX.md`，记录分析文件、方法、状态、日期和一句话问题描述。
5. 告知用户文件路径，开始引导填写。

`INDEX.md` 使用简表即可：

```markdown
# Decision Analysis

| Date | Topic | Method | Status | File | Summary |
| --- | --- | --- | --- | --- | --- |
| 2026-06-15 | AI 助手 | first-principles | drafting | [2026-06-15-ai-assistant-first-principles.md](2026/2026-06-15-ai-assistant-first-principles.md) | 判断是否跟进竞品都在做的 AI 助手方向。 |
```

---

### 第四步：逐章节引导填写

**引导原则：**
- 不要一次性抛出整个模板。
- 逐章节引导，每次只问当前章节的问题。
- 用对话式语言，不要照搬模板里的表格结构。
- 根据用户回答，帮助梳理并填入对应位置。
- 必要时参考 `references/all-usage-guide.md` 的案例和组合模式。

**引导流程：**

#### 如果用第一性原理拆解：

1. **〇、原始问题**
   - "先说说你最初想解决的问题或想做的事？"
   - 用户回答后，追问："这个问题背后隐含了哪些假设？比如'必须用某种方式'或'某个前提成立'？"
   - 填入 0 章

2. **一、拆解到最底层**
   - "我们往上追问：为什么要做这个？解决了能得到什么？"（连续追问 3-4 层，直到找到核心目标）
   - "有哪些物理/经济/人性规律是绕不过去的？"
   - "我们手上有哪些基本材料/工具可以直接用？"（人力/技术/数据/渠道/资金）
   - 填入第一章 1.1-1.3

3. **二、重新定义问题**
   - 基于刚才的拆解，帮用户重新表述问题
   - 区分硬约束（必须满足）和软约束（行业惯例，可挑战）
   - 填入第二章

4. **三、从零重构方案**
   - "基于刚才的底层真相，我们能组装出哪些可能的路径？"
   - 引导用户想出 3-5 个路径，每个都要说清楚核心思路
   - 粗估每个路径的成本和可行性
   - 填入第三章

5. **四、挑战行业假设**
   - "这个领域有哪些'大家都这么做'的惯例？"
   - 逐个拆解：为什么大家都这么做？这个假设还成立吗？
   - 填入第四章

6. **五、极简验证**
   - 从第三章的路径中，提取最关键的假设
   - "用什么最低成本的方式验证这些假设？"
   - 填入第五章

7. **六至八章快速过**
   - 帮用户整理对比矩阵、评估壁垒、输出建议
   - 填入第六至八章

#### 如果用产品价值评估：

1. **一、基本信息** → 直接填写
2. **二、核心假设句式**
   - 逐个引导填写六个模块（用户/场景/问题/现有方案/核心方法/价值）
   - 最后汇总成完整句式
3. **三、六维拆解**
   - 逐维度引导（3.1 用户 → 3.2 场景 → 3.3 痛点 → ...）
   - 每个维度填完后让用户自评（聚焦度/场景价值/痛点强度等）
4. **四、商业验证**
   - 如果是立项前，说明这部分 MVP 后再填
   - 如果已上线，引导用户回填实际数据
5. **五、综合评估**
   - 帮用户打分，计算加权总分
   - 输出建议（通过/有条件通过/需修正/推翻）

#### 如果用苏格拉底追问：

1. **〇、命题声明** → 让用户写下要拷问的命题
2. **一、澄清概念**
   - "这个命题里的关键词是什么？分别指什么？"
   - "哪些是事实？哪些是判断？"
3. **二、探查假设**（最关键）
   - "这个论点背后默认了什么？"
   - 列出所有隐含假设，逐个问："为什么可以这样默认？"
   - 评估每个假设的脆弱度
4. **三、审视证据**
   - "逻辑链条是怎样的？从 A 到 B 必然成立吗？"
   - "有没有反面证据？"
5. **四、替代观点**
   - "反对者会怎么说？"
   - "有没有中间立场或更复杂的模型？"
6. **五、检验后果**
   - "如果按这个逻辑推演下去，会产生什么后果？"
   - "有没有不可逆的风险？"
7. **六、反诘问题**（最容易被跳过，但最重要）
   - "我们是否应该问一个更根本的问题？"
   - "这个问题本身是否隐含了错误的预设？"
8. **七、综合评估**
   - 汇总各层结论
   - 判断：通过/修正/推翻/需重新定义问题

---

### 第五步：输出与后续建议

**填写完成后：**

1. **总结关键发现**（3-5 条要点）
2. **明确结论**
   - 第一性原理 → 推荐哪个路径，需要验证什么
   - 产品价值评估 → 总分多少，建议通过/修正/推翻
   - 苏格拉底追问 → 命题状态（通过/修正/推翻/重新定义）
3. **建议下一步**
   - 如果用了第一性原理，且产出了多个方案 → 建议接下来用产品价值评估选出最优
   - 如果用了产品价值评估，且方案通过 → 建议用苏格拉底追问再拷问一遍
   - 如果苏格拉底追问发现问题可能问错了 → 建议用第一性原理重新拆解
4. **存档提醒**
   - 告知决策分析文件路径
   - 建议定期复盘更新（尤其是产品价值评估的第四章"商业验证"）

**嵌入式辅助输出格式：**

```yaml
decision_handoff:
  method: first-principles | value-assessment | socratic
  question: 本次要判断的问题
  assumptions:
    - 关键假设
  judgement: 通过 | 有条件通过 | 需要修正 | 暂停 | 推翻 | 重新定义问题
  risks:
    - 关键风险
  validation_next:
    - 最小验证动作
```

嵌入式输出同时按通用 `process_handoff` 语义映射：`question`、`judgement` 和已确认判断依据进入 `accepted_facts`；`assumptions` 保持为假设；`risks` 保持为风险；`validation_next` 进入 `gaps` 或后续验证动作。需要跨回合继续时，将摘要交给 `arckit-pending`。

## 三个工具详解

### 工具 1：第一性原理拆解

**何时使用：**
- 全新领域/业务
- 战略转型
- 打破僵局（团队陷入困境）
- "竞品都在做，要不要跟？"

**核心方法：**
1. 拆解到不可改变的基本事实（物理/经济/人性规律）
2. 挑战所有"理所当然"的假设
3. 从零重构多个可能的路径

**产出：**
- 重新定义的问题
- 3-5 个候选方案
- 每个方案的成本与可行性估算

**模板位置：** `templates/first-principles-thinking.md`

---

### 工具 2：产品价值评估

**何时使用：**
- 需求评审
- 立项决策
- 有多个候选方案需要选择

**核心方法：**
六维拆解（用户/场景/痛点/现有方案/核心方法/预期价值）+ 商业定量验证（TAM/LTV/CAC）

**产出：**
- 定量评分（满分 5.0）
- 决策建议（通过/有条件通过/需修正/推翻）
- 关键风险清单

**模板位置：** `templates/product-value-assessment.md`

---

### 工具 3：苏格拉底追问

**何时使用：**
- 方案评审
- 风险识别
- 团队认知对齐
- 方案反复被否（需要找漏洞）

**核心方法：**
六层追问：概念澄清 → 假设探查 → 证据审视 → 替代观点 → 后果检验 → 反诘问题本身

**产出：**
- 命题状态判断（通过/修正/推翻/需重新定义问题）
- 待验证假设清单
- 关键风险

**模板位置：** `templates/socratic-questioning.md`

## 组合使用模式

参考 `references/all-usage-guide.md` 第二、三章。

### 模式 A：完整严谨流程（重大决策）
```
第一性原理(2-3天) → 产品价值评估(1-2天) → 苏格拉底追问(半天)
```

### 模式 B：快速验证流程（常规需求）
```
产品价值评估 → 苏格拉底追问
```

### 模式 C：深度质疑流程（打破僵局）
```
苏格拉底追问 → 第一性原理
```

## 注意事项

1. **不要教条** - 三个工具是思维框架，不是流程规范，按需选择
2. **逐章节引导** - 不要一次性抛出整个模板，用对话式逐步填写
3. **重点关注关键章节**
   - 第一性原理：第一章（拆解到底层）、第四章（挑战假设）
   - 产品价值评估：第二章（核心假设句式）、第三章（六维拆解）
   - 苏格拉底追问：第二章（探查假设）、第六章（反诘问题）
4. **必要时读取参考** - 复杂组合、实战案例和常见陷阱见 `references/all-usage-guide.md`
5. **建议后续工具** - 完成一个工具后，判断是否需要组合使用
6. **存档与迭代** - 独立决策文件位于 `arckit/decision-analysis/`，定期复盘更新对应文件

## 示例对话流程

**用户：** "帮我评估一下，要不要做直播带货？"

**Agent：**
> 这是一个"竞品都在做，要不要跟"的场景。我会先用**第一性原理拆解**，从零审视这个问题，避免盲目跟随。
>
> 先说说你最初想解决的问题或想做的事？

**用户：** "我们想提高销售额，看到竞品都在做直播带货，所以考虑要不要跟"

**Agent：**
> 明白了。这个问题背后隐含了几个假设：
> 1. 提高销售额 = 增加新的销售渠道
> 2. 直播带货是有效的销售渠道
> 3. 竞品在做 = 我们也应该做
>
> 这些假设都成立吗？我们接下来会逐个拆解。
>
> 现在往上追问：为什么要提高销售额？解决了能得到什么？

**用户：** "公司需要增长，投资人有压力"

**Agent：**
> 继续往上：增长的本质是什么？是营收数字，还是利润，还是市场份额？

**用户：** "主要是利润"

**Agent：**
> 好，找到核心目标了：**提高利润**。
>
> 那么问题就变成了：直播带货真的能提高利润吗？还是只是提高营收但利润率更低？
>
> 接下来我们拆解基本事实：有哪些物理、经济或人性规律是绕不过去的？

（继续逐步引导填写...）
