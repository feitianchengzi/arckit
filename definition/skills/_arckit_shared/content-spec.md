# ArcKit 规格文档内容规范（Skill 参考）

本文件为 arckit-spec / arckit-tech / arckit-interaction / arckit-visual 共用的规则参考，各 SKILL.md 内含精简摘要，此处提供完整细节供按需读取。

---

## 正文语体

正文使用陈述式、现在时，描述目标系统应当是什么。

**允许**：「该功能支持 X」「Token 取值为 #...」「触发 A，系统响应 B」「不支持 X」（边界约束）

**禁止**：
- 改版叙事：「本次将 X 改为 Y」「相比旧版…」「经讨论后…」
- 落地建议：「可以先简化」「MVP 阶段暂时」「后续再完善」「可先硬编码」
- 过程叙事（非 research 文档）：「我们调研了三个方案，最终选择…」

`document_scope` YAML 和 Git 历史是记录变更摘要的唯一位置，不进入文档正文。

---

## 禁止混入的内容（全域）

适用于所有 arckit/* 主文档（contracts/models YAML 除外，它们是结果规格本身）：

- 成本预估、人力估算
- 排期、里程碑、版本迭代计划
- 工程实施建议（简化实现、临时绕过、「MVP 砍 X」）
- 可运行的业务实现代码（完整函数体、类实现、服务脚本）

---

## 各文档类型内容边界

| 文档类型 | 允许 | 禁止 |
|----------|------|------|
| `spec/feature.md` | 功能行为规则、边界条件、错误态定义 | 代码、工程实施 |
| `tech/solution.md` | 架构、数据流、组件职责、短伪代码、引用契约 | 完整实现代码、成本/排期 |
| `tech/research.md` | 候选方案简洁陈述（是什么/适用场景/权衡）、选定方案及理由、未选方案的简洁说明 | 时间性过程叙述（会议流水账、「反复讨论后」）、成本/排期、落地建议 |
| `tech/decision.md` | 决策结论及理由、ADR 格式的「不选 X 原因」 | 成本/排期、简化建议 |
| `tech/_shared/contracts/*.yaml` | 接口签名、请求/响应结构、错误码 | 无限制 |
| `tech/_shared/models/*.yaml` | 实体属性、关系、约束 | 无限制 |
| `visual/tokens.yaml`、`components/*.yaml` | Token 值及语义描述、各状态视觉定义 | CSS/HTML 实现片段 |
| `interaction/interaction.md` | 触发条件与响应、状态机规则、异常态处理 | 实现代码、改版叙事 |
| `interaction/default.html` | 极简灰度线框图（HTML 本身是交付物） | 品牌色、业务逻辑脚本 |

**特例**：`tech/research.md` 允许候选方案对比与过程叙述，但必须包含明确结论节，结论节用陈述式写。

## 历史内容处置（三步决策）

文档更新时，对不再反映当前状态的旧描述：

1. 与当前规格矛盾或已过时 → 直接删除
2. 有保留价值（弃用原因、方案对照）→ 移入文档末尾固定命名的附录块（`## 历史对照` 或 `## 弃用说明`），不与主线穿插
3. 附录块内仍用陈述式，写「X 已弃用，因为 Y」

---

## 域归属判定（结构治理）

读取 INDEX 后、体积分析前，判断**新内容是否已有同域文件**：

| 判定 | 处理 |
|------|------|
| 有同域文件 | 并入——更新该文件对应章节，不新建 |
| 有同域文件但职责明显分化 | 阐明拆分理由，再新建 |
| 无同域文件 | 新建，路径按 INDEX 枝命名 |

若发现多个已有文件域重叠（同一主题分散多处）：先合并，再处理变更，合并记入 `document_scope.reorganized`。

**两个独立维度，互不替代**：
- 域归属：新内容去哪里（并入 vs 新建）
- 体积拆分：文件是否过大（如何分）
