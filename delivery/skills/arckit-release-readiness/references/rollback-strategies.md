# 回滚策略

---

## 一、代码回滚

### 方式1: git revert

| 属性 | 内容 |
|------|------|
| **适用场景** | 代码已合入main，需要撤销特定提交 |
| **操作步骤** | 1. `git log --oneline` 找到要回滚的commit<br>2. `git revert <merge-commit> -m 1`（回滚merge commit）<br>3. `git push origin main` |
| **风险** | 低。创建新commit反转变更，历史完整 |
| **时间** | 2-5分钟 |

### 方式2: Feature Flag关闭

| 属性 | 内容 |
|------|------|
| **适用场景** | 新功能通过Feature Flag控制，关闭即可回滚 |
| **操作步骤** | 1. 在配置中心将feature flag设为false<br>2. 验证功能已关闭 |
| **风险** | 极低。无需代码变更，秒级生效 |
| **时间** | <1分钟 |

### 方式3: 蓝绿部署回滚

| 属性 | 内容 |
|------|------|
| **适用场景** | 使用蓝绿部署策略，需要切回旧版本 |
| **操作步骤** | 1. 确认蓝环境（旧版本）健康<br>2. 将流量从绿环境切回蓝环境<br>3. 验证蓝环境正常 |
| **风险** | 低。旧环境一直在运行，切换即生效 |
| **时间** | 1-3分钟 |

## 二、数据库回滚

### 方式1: 向后兼容迁移

| 属性 | 内容 |
|------|------|
| **适用场景** | 迁移脚本设计为向后兼容，新旧代码都可运行 |
| **操作步骤** | 1. 部署新代码（兼容新旧schema）<br>2. 执行数据库迁移<br>3. 如需回滚，只需部署旧代码 |
| **风险** | 低。迁移是加法操作（加列/加表），不破坏旧代码 |
| **时间** | <5分钟（只回滚代码） |

**向后兼容迁移原则：**
- 加列：允许NULL或有默认值
- 加表：新表不影响旧代码
- 删列：先部署不再使用该列的代码，下个版本再删列
- 改列：先加新列，迁移数据，再删旧列（分3个版本完成）

### 方式2: 双写

| 属性 | 内容 |
|------|------|
| **适用场景** | 需要变更数据格式，新旧格式并存 |
| **操作步骤** | 1. 新代码同时写新旧格式<br>2. 迁移历史数据到新格式<br>3. 验证新格式完整<br>4. 停止写旧格式<br>5. 如需回滚，切回旧格式读取 |
| **风险** | 中。需要维护双写逻辑，回滚时需确保旧格式数据完整 |
| **时间** | 10-30分钟 |

### 方式3: 数据修复脚本

| 属性 | 内容 |
|------|------|
| **适用场景** | 数据已被错误修改，需要修复到正确状态 |
| **操作步骤** | 1. 从备份/日志分析错误范围<br>2. 编写修复脚本<br>3. 在staging验证<br>4. 在生产执行<br>5. 验证修复结果 |
| **风险** | 高。修复脚本本身可能有bug，必须先在staging验证 |
| **时间** | 30分钟-2小时（取决于数据量和复杂度） |

## 三、配置回滚

### 方式1: 版本化配置

| 属性 | 内容 |
|------|------|
| **适用场景** | 配置存储在Git/配置中心，支持版本回退 |
| **操作步骤** | 1. `git revert` 配置变更<br>2. 推送到配置仓库<br>3. 配置中心自动同步 |
| **风险** | 低 |
| **时间** | 1-3分钟 |

### 方式2: 快速切换

| 属性 | 内容 |
|------|------|
| **适用场景** | 使用Feature Flag/配置中心，可秒级切换 |
| **操作步骤** | 1. 在配置中心切换到旧值<br>2. 验证生效 |
| **风险** | 极低 |
| **时间** | <1分钟 |

### 方式3: 环境变量回滚

| 属性 | 内容 |
|------|------|
| **适用场景** | 通过环境变量控制行为 |
| **操作步骤** | 1. 修改环境变量为旧值<br>2. 重启服务（如需要） |
| **风险** | 低。需确认服务重启不影响其他功能 |
| **时间** | 2-5分钟 |

## 四、回滚决策矩阵

| 信号 | 严重程度 | 回滚级别 | 时间要求 |
|------|---------|---------|---------|
| 错误率超过基线2倍 | 🔴 Critical | 代码+配置回滚 | 立即 |
| P95延迟超过基线3倍 | 🔴 Critical | 代码回滚 | 立即 |
| 核心业务指标下降>5% | 🔴 Critical | 代码回滚 | 15分钟内未恢复则回滚 |
| 数据不一致 | 🔴 Critical | 代码回滚+数据修复 | 立即回滚 + 数据修复 |
| 新功能异常但核心流程正常 | 🟡 Medium | Feature Flag关闭 | 30分钟内 |
| 非核心功能性能退化 | 🟢 Low | 下一版本修复 | 不回滚 |
| UI小问题 | 🟢 Low | 下一版本修复 | 不回滚 |

## 五、回滚后处理

### 1. 通知

```
回滚通知模板：

【紧急回滚】{项目名} v{version}

回滚原因：{简述问题}
影响范围：{受影响的功能/用户}
回滚时间：{开始时间} - {结束时间}
当前状态：✅ 已回滚到 v{previous_version}

下一步：
1. {排查根因的计划}
2. {预计修复时间}
3. {是否需要停机窗口}
```

### 2. 复盘

回滚后24小时内完成复盘：

```markdown
# 事故复盘

## 时间线
| 时间 | 事件 |
|------|------|
| HH:MM | 发布 v{version} |
| HH:MM | 发现异常（{信号}） |
| HH:MM | 决定回滚 |
| HH:MM | 回滚完成 |
| HH:MM | 确认服务恢复 |

## 根因
[为什么会出问题？]

## 为什么没有在发布前发现？
[测试/review/监控的盲点]

## 修复计划
- [ ] {修复方案}
- [ ] {防止同类问题再发生的措施}

## 经验教训
- {教训1}
- {教训2}
```

### 3. 修复计划

- 回滚≠问题解决，只是止血
- 48小时内提交修复PR
- 修复PR必须包含：问题复现测试 + 修复代码 + 回归测试
- 修复PR的review标准不低于正常PR
