# ArcKit Debug Diagnosis

## 目标

把未知根因的问题收敛成可复现、可观察、可验证的必要修复。先分析代码逻辑能否完整解释问题表现；只有逻辑推演结果与问题表现 100% 匹配时，才直接确认根因。其余情况通过输出到 `.log` 的运行时日志埋点采集真实路径证据，确认根因后再修复。

## 适用边界

- 适用：已经出现异常、回归或不确定根因，需要定位和修复。
- 不适用：用户明确要求正向开发、架构设计、功能规划或重构方案。
- 不隐式联动：不要自动调用、路由或直接修改其他 skill。
- 可接收证据：可以接收用户、运行日志、测试输出、实现结果或外部 adapter handoff 作为诊断证据，但必须重新验证关键假设。
- 不沉淀执行：如果发现可沉淀为正向开发规则，只在最终输出中给出建议，不直接修改任何 skill。

## 硬约束

- 先做逻辑匹配判断：代码逻辑推演必须能完整解释问题的触发条件、实际表现、数据或状态变化、发生位置及时序，且与已知反例不冲突，才算与问题表现 100% 匹配。
- 逻辑分析结果与问题表现未达到 100% 匹配时，必须进入 `.log` 埋点诊断，不得直接确认根因或开始修复。错误栈、测试失败、现有日志和 diff 可以帮助设计埋点，但不能替代这一匹配门禁。
- `.log` 埋点诊断是在关键执行边界添加临时日志，让应用、测试或脚本在复现问题时把真实运行事件输出到 `arckit/debug/<issue-slug>.log`。日志应包含时间、唯一标记、事件、关键状态、关联 ID 和必要结果，足以验证逻辑推演并区分竞争假设。
- 运行复现路径并读取 `.log` 后，根据观测结果更新分析；直到日志证据能够确认实际执行路径与根因，才进入必要修复。
- `arckit/debug/<issue-slug>.log` 承载运行时埋点数据；症状、假设、分析结论、修复说明和清理状态由最终输出或项目已有记录载体承载。
- 临时代码埋点必须范围受控、可搜索、可删除。新增 `console.*`、临时日志、debug flag 或 trace marker 时，使用唯一标记（如 `ARC_DEBUG:<issue-slug>`），避免敏感信息，避免改变业务行为。
- 修复完成后必须清理临时埋点代码。最终响应前检查 diff 或搜索唯一标记，确认临时 `console.*`、临时日志、debug flag、trace marker 已删除；若保留长期结构化日志，必须说明它的长期诊断价值和脱敏边界。
- 不允许把“看起来像某段代码的问题”当作已定位。只有观测结果能排除关键竞争假设时，才能进入必要修复。

## 诊断循环

1. 区分现象和解释：先把用户看到的问题表现整理为可逐项匹配的事实。
2. 推演代码逻辑：建立输入、数据流、状态变化、边界和时序链，判断它能否完整解释每一项问题表现。
3. 识别身份和新鲜度：同一个资源、请求、记录或状态，可能身份没变但内容完整度、时间戳、缓存层级或派生结果已经不同。
4. 关注边界和时序：异步补全、失败回退、缓存命中、取消、重试、并发、环境差异，经常比单点代码更接近根因。
5. 执行匹配门禁：只有逻辑推演与问题表现 100% 匹配时才确认根因；否则进入 `.log` 埋点诊断。
6. 设计并添加日志埋点，运行复现路径采集 `.log`，用真实执行数据验证逻辑并排除竞争假设。
7. 做必要修复：只改被证据指向的根因路径和必要防护。
8. 回到原始路径验证：确认用户报告的症状在同样条件下消失。
9. 清理临时代码埋点及其临时 `.log` 输出；只有具备长期运维价值时才保留产品自身的结构化日志。

## 关键问题

定位时先回答这些问题，再决定检查顺序：

- 这个问题发生在用户路径的哪一步？
- 这一刻系统依赖哪些输入、状态、缓存、配置、外部返回或异步任务？
- 哪些数据是 source of truth，哪些只是派生、展示、缓存或 fallback？
- 最近变化中，哪一处最可能改变了身份、新鲜度、时序或边界条件？
- 应在哪些关键执行边界添加 `.log` 埋点，才能验证逻辑分析并区分竞争假设？
- 修复应该落在哪个合适边界，才能避免扩散到无关模块？

## 执行流程

1. 保护现场：先看工作区状态、失败输出、最近变更和用户已有改动；不要回滚未确认的改动。
2. 收敛症状：写清期望行为、实际行为、首次出现时间、影响范围、稳定复现还是偶发、严重程度。
3. 建立复现：记录稳定路径、输入数据、账号/环境、设备/系统、网络/缓存/时间条件；没有稳定复现时先布置 `.log` 埋点，等待问题路径再次触发并采集数据。
4. 判断逻辑匹配度：逐项对照代码逻辑推演与问题表现；只有触发条件、表现、状态变化、位置和时序全部匹配，才能直接确认根因。
5. 对照变更：检查相关 diff、最近提交、依赖升级、配置变化、数据迁移、远端接口或环境变化。
6. 切分问题面：按当前证据区分状态、数据、请求、缓存、异步时序、环境、配置、渲染、性能或构建等可能方向。
7. 设计 `.log` 观察点：未达到 100% 匹配时，在能区分假设的执行边界选择日志字段，优先覆盖数据流、边界条件、异步时序、缓存身份、并发和错误路径。
8. 添加日志埋点：在被诊断代码中加入带唯一标记的临时埋点，把运行时事件输出到 `arckit/debug/<issue-slug>.log`；记录必要状态和关联 ID，避免敏感信息和业务行为变化。
9. 运行并确认：执行问题复现路径，读取 `.log`，逐项核对实际执行数据与逻辑分析；观测仍不能确认根因时，调整埋点并继续采集。
10. 必要修复：只改根因路径和必要防护；避免顺手重写无关结构。
11. 回归验证：跑相关测试，补缺失测试或手测路径；覆盖成功、失败、空数据、重试、边界和回退场景。
12. 清理临时埋点：删除临时 `console.*`、debug log code、debug flag 或 trace marker，并删除不再需要的临时 `.log`；搜索唯一标记确认无残留，把清理结果写入最终输出或项目已有记录载体。
13. 沉淀建议：如果这次根因暴露出正向开发规则缺口，只在最终输出中写“建议沉淀”，并说明建议内容和理由。

## 修复原则

- 先判断逻辑分析能否与问题表现 100% 匹配；不能完全匹配时，通过 `.log` 日志埋点确认根因后再修改。
- 优先改根因，不用大面积重构掩盖问题。
- 对偶发问题重点检查时序、取消、缓存、并发、幂等和环境差异。
- 对数据问题区分“数据不存在”“数据过期”“数据身份相同但完整度变化”“数据存在但 UI 未刷新”。
- 对外部依赖问题保留失败降级和可观测错误，不把错误吞成空白状态。
- 修复后删除临时噪声日志，保留有长期诊断价值的结构化观察点。
- 不直接套固定清单；先从当前项目的结构和证据推导检查顺序。

## 输出结构

完成诊断或阶段性停顿时，按需输出：

- 症状：用户看到的问题和期望行为。
- 复现：稳定路径；如果无法稳定复现，说明已布置或还需要布置哪些 `.log` 埋点。
- 逻辑匹配：代码逻辑分析是否与问题表现 100% 匹配；未完全匹配时说明需要验证的差异。
- `.log` 埋点：埋点位置、观测字段、`arckit/debug/<issue-slug>.log` 输出路径和唯一标记；明确日志由哪个运行命令或复现路径产生。其他 trace、测试或状态快照作为补充证据列出。
- 证据：日志、测试、diff、状态快照、请求响应、截图或指标。
- 假设：当前最强假设，以及已经排除的假设。
- 修复：实际改动范围、根因路径和必要防护。
- 验证：运行过的测试、手测路径或仍需观察的指标。
- 临时埋点清理：删除了哪些临时代码埋点；是否搜索过唯一标记；如保留长期日志，说明原因。
- 沉淀建议：如果值得沉淀，说明建议优化哪类正向开发规则、为什么、依据来自哪次定位；只给建议，不直接修改任何 skill。

## 参考和脚本

- 复杂或多轮失败的诊断，可读取 `references/debugging-methodology.md`，其中保留了原始 `all/06-sre/operate` 的系统化调试方法论。
- 需要生成结构化调试报告草案时，可使用 `scripts/debug_report.py`。该脚本只生成报告文本，不证明根因，也不替代证据验证。

## 完成标准

- 给出明确症状、复现条件或无法复现时的观察计划。
- 明确逻辑分析是否与问题表现 100% 匹配；未完全匹配时必须完成 `.log` 日志埋点、运行复现和证据分析。
- 标明最可能的问题域和被排除的关键假设。
- 修复范围受证据约束，并说明根因路径。
- 修复后说明临时埋点清理结果，或说明保留长期结构化日志的理由。
- 提供测试、手测或线上观察验证方式。
- 如发现正向规则缺失，只输出可执行的优化建议，不执行 skill 修改。
