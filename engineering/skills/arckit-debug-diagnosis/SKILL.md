# ArcKit Debug Diagnosis

## 目标

把未知根因的问题收敛成可复现、可观察、可验证的最小修复。先把症状拆成事实，再根据项目上下文提出假设，用证据证明问题在哪里，最后做必要修改。

## 适用边界

- 适用：已经出现异常、回归或不确定根因，需要定位和修复。
- 不适用：用户明确要求正向开发、架构设计、功能规划或重构方案。
- 不隐式联动：不要自动调用、路由或直接修改其他 skill。
- 可接收证据：可以接收 `arckit-runtime-operations`、`arckit-verify-implementation` 或用户提供的 handoff 作为诊断证据，但必须重新验证关键假设。
- 不沉淀执行：如果发现可沉淀为正向开发规则，只在最终输出中给出建议，不直接修改任何 skill。

## 硬约束

- 根因解释必须由证据支持。除非现有错误栈、失败测试、运行日志、可复现路径或明确 diff 已经足以证明根因，否则禁止直接按代码猜测修复。
- 证据不足时必须先补最小观察点。只要存在两个以上竞争假设，或只能从代码阅读推测根因，就先增加日志、console、trace、状态快照、请求/响应记录、断点、测试或指标来区分假设。
- ArcKit 项目中的诊断过程优先记录到 `arckit/debug/<issue-slug>.log`，至少写明症状、假设、观察点、临时埋点位置、观测结果、结论和清理状态。
- 临时代码埋点必须最小、可搜索、可删除。新增 `console.*`、临时日志、debug flag 或 trace marker 时，使用唯一标记（如 `ARC_DEBUG:<issue-slug>`），避免敏感信息，避免改变业务行为。
- 修复完成后必须清理临时埋点代码。最终响应前检查 diff 或搜索唯一标记，确认临时 `console.*`、临时日志、debug flag、trace marker 已删除；若保留长期结构化日志，必须说明它的长期诊断价值和脱敏边界。
- 不允许把“看起来像某段代码的问题”当作已定位。只有观测结果能排除关键竞争假设时，才能进入最小修复。

## 诊断循环

1. 区分现象和解释：用户看到的是现象，根因解释必须等待证据支持。
2. 建立事实链：输入是什么、数据从哪里来、在哪里变形、什么时候进入展示/接口/存储/缓存、什么时候没有按预期更新。
3. 识别身份和新鲜度：同一个资源、请求、记录或状态，可能身份没变但内容完整度、时间戳、缓存层级或派生结果已经不同。
4. 关注边界和时序：异步补全、失败回退、缓存命中、取消、重试、并发、环境差异，经常比单点代码更接近根因。
5. 设计可证伪观察点：每个假设都要有一个能改变判断的最小观察点。
6. 证据不足时先埋点或记录观察结果，直到能排除关键竞争假设。
7. 做最小修复：只改被证据指向的根因路径和必要防护。
8. 回到原始路径验证：确认用户报告的症状在同样条件下消失。
9. 清理临时代码埋点，保留必要的诊断记录或长期结构化日志。

## 关键问题

定位时先回答这些问题，再决定检查顺序：

- 这个问题发生在用户路径的哪一步？
- 这一刻系统依赖哪些输入、状态、缓存、配置、外部返回或异步任务？
- 哪些数据是 source of truth，哪些只是派生、展示、缓存或 fallback？
- 最近变化中，哪一处最可能改变了身份、新鲜度、时序或边界条件？
- 能用哪个最小观察点区分两个竞争假设？
- 修复应该落在哪个最窄的边界，才能避免扩散到无关模块？

## 执行流程

1. 保护现场：先看工作区状态、失败输出、最近变更和用户已有改动；不要回滚未确认的改动。
2. 收敛症状：写清期望行为、实际行为、首次出现时间、影响范围、稳定复现还是偶发、严重程度。
3. 建立复现：记录最短路径、输入数据、账号/环境、设备/系统、网络/缓存/时间条件；没有复现时先补观察点。
4. 判断证据充分性：明确现有证据是否已经能证明唯一根因；如果不能，列出竞争假设和缺口，不进入修复。
5. 对照变更：检查相关 diff、最近提交、依赖升级、配置变化、数据迁移、远端接口或环境变化。
6. 切分问题面：按当前证据区分状态、数据、请求、缓存、异步时序、环境、配置、渲染、性能或构建等可能方向。
7. 设计观察点：选择最小日志、断点、测试、trace、请求响应、状态快照、缓存内容或性能指标；观察点要能证伪假设。
8. 增加临时观测：证据不足时添加最小埋点或 `arckit/debug/<issue-slug>.log` 记录，写清唯一标记、文件位置、观测字段、预期能区分的假设和清理计划。
9. 验证假设：每次只验证一个根因假设；优先验证数据流、边界条件、异步时序、缓存身份、并发和错误路径。
10. 最小修复：只改根因路径和必要防护；避免顺手重写无关结构。
11. 回归验证：跑相关测试，补缺失测试或手测路径；覆盖成功、失败、空数据、重试、边界和回退场景。
12. 清理临时埋点：删除临时 `console.*`、debug log code、debug flag 或 trace marker；搜索唯一标记确认无残留，把清理结果写入诊断记录或最终输出。
13. 沉淀建议：如果这次根因暴露出正向开发规则缺口，只在最终输出中写“建议沉淀”，并说明建议内容和理由。

## 修复原则

- 先证明，再修改；不能证明时先加最小观察点或临时埋点。
- 优先改根因，不用大面积重构掩盖问题。
- 对偶发问题重点检查时序、取消、缓存、并发、幂等和环境差异。
- 对数据问题区分“数据不存在”“数据过期”“数据身份相同但完整度变化”“数据存在但 UI 未刷新”。
- 对外部依赖问题保留失败降级和可观测错误，不把错误吞成空白状态。
- 修复后删除临时噪声日志，保留有长期诊断价值的结构化观察点。
- 不直接套固定清单；先从当前项目的结构和证据推导检查顺序。

## 输出结构

完成诊断或阶段性停顿时，按需输出：

- 症状：用户看到的问题和期望行为。
- 复现：稳定路径；如果无法复现，说明缺什么观察点。
- 证据充分性：现有证据是否足以证明唯一根因；不足时列出竞争假设。
- 观察点/埋点：新增或使用的日志、console、trace、测试、状态快照、`arckit/debug/<issue-slug>.log` 路径和唯一标记。
- 证据：日志、测试、diff、状态快照、请求响应、截图或指标。
- 假设：当前最强假设，以及已经排除的假设。
- 修复：实际改动范围和为什么是最小修复。
- 验证：运行过的测试、手测路径或仍需观察的指标。
- 临时埋点清理：删除了哪些临时代码埋点；是否搜索过唯一标记；如保留长期日志，说明原因。
- 沉淀建议：如果值得沉淀，说明建议优化哪类正向开发规则、为什么、依据来自哪次定位；只给建议，不直接修改任何 skill。

## 参考和脚本

- 复杂或多轮失败的诊断，可读取 `references/debugging-methodology.md`，其中保留了原始 `all/06-sre/operate` 的系统化调试方法论。
- 需要生成结构化调试报告草案时，可使用 `scripts/debug_report.py`。该脚本只生成报告文本，不证明根因，也不替代证据验证。

## 最低交付标准

- 给出明确症状、复现条件或无法复现时的观察计划。
- 明确证据是否足以证明唯一根因；不足时必须先补最小观察点或临时埋点。
- 标明最可能的问题域和被排除的关键假设。
- 修复范围最小，并说明根因路径。
- 修复后说明临时埋点清理结果，或说明保留长期结构化日志的理由。
- 提供测试、手测或线上观察验证方式。
- 如发现正向规则缺失，只输出可执行的优化建议，不执行 skill 修改。
