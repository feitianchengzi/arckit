# 问题背景

## 时代变化

AI 时代的软件研发发生了基础生产关系变化。过去，软件研发的核心生产者主要是人，工具负责辅助人完成编辑、构建、测试、部署和协作。现在，Codex 类 Agent 开始具备理解意图、读取项目、修改文件、运行命令、生成文档、维护行为约束、验证结果和持续协作的能力。

这种变化使软件研发从“人使用工具生产软件”扩展为“人、单 Agent 和多 Agent 自动化平台共同生产软件”。人在其中不再只是在 IDE 中逐行编写代码，也会通过 prompt 把目标、约束、判断、纠错和决策传递给 Agent，由 Agent 执行一部分产品、工程、验证和沉淀工作；在自动化程度更高的场景中，多 Agent + Loop 平台会持续推进日常研发任务，并在遇到无法可靠解决的问题时交还人类接手。

传统研发流程通常通过团队角色分工来承载复杂性。产品、设计、研发、测试、运维和运营分别维护自己的产物、判断和协作边界。AI Agent 进入后，单个人可以和 Agent 覆盖过去需要多个角色连续协作才能推进的链路，但这些角色背后的问题并没有消失。

本质变化包括：

- 执行主体变化：代码、文档、测试、发布准备和行为约束维护不再只能由人手工完成，Agent 可以直接参与生产。
- 交互界面变化：自然语言 prompt 成为重要研发入口，替代或补充传统 issue、PRD、设计稿、任务单和代码编辑。
- 团队边界变化：一部分跨角色协作被压缩到“人 + Agent”的对话中，也可能被组织到多 Agent 自动化平台中；只有在需要外部确认、专业判断、组织授权或人类价值判断时才回到人类。
- 产物范围变化：最终产物不再只可能是 code，也可能是 prompt、workflow patch、行为约束文件或其他能稳定改变 Agent 行为的对象。
- 过程载体变化：讨论、判断、纠错和执行经验如果不被系统沉淀，就会随着上下文窗口、聊天记录和单次任务结束而丢失。
- 事实源关系变化：实现事实可以被 Agent 快速改变，预期事实、过程事实和工作流事实必须同步建立边界，否则实现会脱离预期，或当前实现会被误认为正确预期。

Codex 类 Agent 的采用动因来自能力和效率的同时成立。Agent 看起来能做很多事，并且实际已经能把大量软件研发活动完成得足够好；业界发展趋势也表明，这类能力会继续变强、变稳、变便宜、变普及。

在人面对真实软件研发目标时，只要 Agent 能在一部分场景中显著降低成本、缩短反馈周期、扩大个人可完成范围，使用 Agent 就会成为理性选择。人没有理由因为传统流程存在，就拒绝一个能更快完成探索、实现、验证、文档和维护工作的协作者。

因此，问题不是人要不要使用 Codex 类 Agent，而是人在使用之后，如何避免把真实软件研发压缩成零散 prompt、一次性代码修改和不可追踪的聊天上下文。Agent 越能干、越高效，越需要明确的协作操作层来承接阶段、产物、事实源、验证和记忆。

人之所以需要和 Codex 类 Agent 或多 Agent 自动化平台做全流程协同，不是因为传统研发流程不再存在，而是因为传统流程中的大量活动被重新压缩到人、Agent 和自动化 loop 的持续交互中。系统必须重新建立阶段、产物、事实源、职责、交接和记忆机制，才能让这种压缩后的协作仍然保持真实软件研发所需的严肃性。

## AI 能力演进与版本化推导

当前问题背景基于当下可用的 Codex 类 Agent 能力做判断。这个判断不是永久静态前提。AI 和 Agent 能力会持续变化，变化内容包括模型理解能力、上下文容量、工具调用稳定性、代码修改可靠性、长程任务能力、成本、速度、安全策略、组织授权和行业合规边界。

当这些能力发生关键变化时，Arckit 需要回到本问题背景重新推导：哪些研发活动仍需要人主导，哪些活动可以交给 Agent，哪些阶段可以合并，哪些事实源需要新增，哪些验证门禁可以放宽或必须加强，哪些能力单元需要拆分、合并或被新的能力替代。

不同国家、公司、行业、业务、成本结构和安全要求会允许使用不同水平的 AI 能力。一个组织可以使用强 Agent、长上下文和高权限工具链，另一个组织可能只能使用低权限、低成本或受监管的 Agent。二者对应的 Arckit 方案不一定是一新一旧，也不一定一对一错，而是基于不同 AI 能力假设推导出的不同版本。

因此，Arckit 的产品决策需要保留版本化推导能力。每个方案都应能说明它基于什么 Agent 能力、成本约束、权限边界和组织环境成立。当 AI 能力继续发展时，系统通过重新推导问题背景和解决思路来演进，而不是简单在旧方案上堆叠规则。

## 场景定义

Arckit 面向的基础场景是：人基于 Codex 类 Agent 推进真实软件研发活动。这里的 Agent 不是简单问答工具，而是具备读取项目、修改文件、运行命令、生成文档、维护行为约束和持续协作能力的软件研发执行体。

Arckit 同时面向更高自动化的场景：多 Agent + Loop 平台围绕同一个软件项目持续推进日常研发任务。该平台可以包含规划、实现、诊断、验证、审查、治理等不同 AI 角色，也可以调用外部工具、CI、部署系统、设计系统或项目管理系统。Arckit 不等同于这个平台本身，而是为平台和人类提供共享的项目事实、任务状态、handoff、pending、workflow memory 和接手协议。

这两个场景使用同一套 Arckit 语义。区别在于执行调度由谁承担：在人机协作中，人和 Codex 类 Agent 直接对话；在自动化平台中，loop 和多 Agent 角色持续推进；当自动化平台遇到无法可靠处理的审美、商业、组织授权、权限、外部依赖或高风险技术判断时，人类可以基于 Arckit 记录直接接手。

该场景中的“软件研发”不只包括写代码。它包括产品概念形成、价值判断、需求定义、交互设计、视觉规则、技术方案、任务拆解、实现、验证、review、发布、运行观察、运营素材和工作流沉淀。

该场景中的最终产物通常是能改变系统行为的产物。传统项目中最终产物主要是代码；在 Agent 时代，最终产物也可以是 prompt、workflow patch、配置、脚本、行为约束文件或其他能稳定改变 Agent 或软件系统行为的对象。

## 人发 Prompt 的目的

人在该场景中发 prompt 的目的不是单一的“下命令”。Prompt 是人把软件研发意图传递给 Agent 的低摩擦界面。

一个 prompt 可以承担以下目的：

- 表达真实软件目标。
- 提供产品想法或价值假设。
- 请求讨论、分析或判断。
- 指定做事原则或流程偏好。
- 提供约束、素材、上下文或背景。
- 请求任务拆解或分发。
- 请求具体实现。
- 请求验证、review 或发布准备。
- 纠正 Agent 对需求、事实、流程、沉淀方式或术语的理解。
- 把暂时说不清的灵感、担忧或直觉交给 Agent 处理。

因此，prompt 是多本体表达。它可能同时包含需求、决策、流程、代码细节、产品价值、未决问题和工作流纠偏。

## Prompt 的局限

Prompt 的天然局限是表达压缩。人无法也不应该在每次输入中完整说清软件研发所需的全部背景。

Prompt 通常存在以下问题：

- 上下文省略：人默认 Agent 理解项目背景、历史讨论和隐含目标。
- 本体混合：需求、判断、流程、实现和纠错混在同一段话中。
- 粒度不稳定：一句话可能指向真实产品，也可能只指向一个局部细节。
- 阶段不明确：用户可能要概念澄清、任务拆解、实现、验证或沉淀，但没有显式说明阶段。
- 目标演化：用户可能在说话过程中才逐步找到真正想要的东西。
- 正确性不确定：用户表达可以是已确认决策，也可以是待验证假设。
- 产物不明确：用户可能期待 code、行为约束文件、spec、pending、运营素材或只期待聊天中的判断。

系统不能要求用户先把 prompt 结构化成专业研发文档。系统应该承担语义识别、阶段判断和产物路由。

## Codex 类 Agent 的限制

Codex 类 Agent 具备强执行能力，但默认存在研发协作限制。

主要限制包括：

- 上下文窗口有限，无法天然保留完整项目历史。
- 单轮执行受时间、成本、工具权限和环境状态限制。
- Agent 倾向把用户输入压缩成当前可执行任务，尤其是代码实现任务。
- Agent 可以修改文件，但不天然知道哪些过程思考应该沉淀。
- Agent 可以生成文档，但不天然区分正式事实、候选判断和临时讨论。
- Agent 可以使用可复用能力，但容易按关键词触发单个能力，而不是编排完整研发流程。
- Agent 可以观察实现事实，但不天然知道实现事实是否应该反向修正预期事实。
- Agent 可以执行验证，但不天然知道验证结果如何回流到产品、技术或工作流记忆。

这些限制不是 Agent 错误，而是人机协同产品必须显式解决的系统问题。

## 真实软件预期与单轮交付之间的差距

人在软件研发中通常指向真实世界的真实软件。真实软件包含真实用户、真实场景、真实运行环境、真实上线、真实运营和真实迭代。

Agent 单轮协作通常只能推进一个阶段产物。阶段产物可以是概念澄清、价值假设、需求草案、正式规格、任务拆解、局部实现、验证报告、发布判断或运行观察。

有限交付本身不是问题。问题在于系统把阶段产物误当成真实软件目标已经完成，或把真实软件预期压缩成一次性实现任务。

## 事实源混乱

软件研发中至少存在两类同等重要的事实：

- 预期事实：软件应该是什么。
- 实现事实：软件现在是什么。

预期事实通常由 spec、interaction、visual、tech、产品价值和已确认决策表达。实现事实通常由代码、prompt、workflow patch、行为约束文件、配置、运行结果、日志和测试结果表达。

只保留预期事实会得到无法证明已经实现的愿望。只保留实现事实会把当前实现误当成正确预期，丢失为什么做、应该做到什么程度和偏离时如何判断。

软件研发还存在过程事实和工作流事实。过程事实表达探索、分析和未决状态；工作流事实表达 Agent 未来应该如何工作。没有这些中间事实源，系统会在信息丢失和正式事实污染之间反复摇摆。

## 根问题

Arckit 要解决的根问题是：

> 在人、Codex 类 Agent 和多 Agent 自动化平台共同完成真实软件研发时，如何把人的高压缩、多义、持续演化的自然语言表达，以及 Agent 执行过程中的证据、失败和接力需求，转换成当前可执行的研发动作、正确类型的阶段产物、可验证的最终产物、可恢复的 handoff，以及可持续回流的多类事实源。

该问题不能通过单个能力单元、固定流程或某个全自动平台内部实现解决。它需要一个面向软件开发 Agent 的协作与接力协议层，持续完成语义编译、阶段判断、产物路由、能力编排、handoff、验证比较、事实回流和人类接手准备。
