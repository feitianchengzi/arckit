# 解决思路

## 定位

解决思路承接问题背景，回答 Arckit 为什么应该以“软件开发 Agent 协作与接力协议层”的方式解决问题。

本文件不定义完整产品概念清单，不描述产品结构，也不涉及具体执行载体。它只描述从问题到方案之间的推导原则。

## 将协作和接力问题前置

人、Codex 类 Agent 和多 Agent 自动化平台协作的根问题不是某个单点能力缺失，而是完整软件研发活动被压缩进自然语言对话和自动化 loop 后，阶段、产物、事实、责任、交接和记忆失去稳定边界。

因此，系统首先解决协作组织和接力恢复问题，而不是直接解决某个局部执行问题。

该思路要求系统先判断当前输入或自动化执行状态处于什么研发语境，再决定要推进什么产物、交给哪个能力、何时停止以及如何让人类或另一个 Agent 接手。

## 支持人机协作和自动化平台同构

Arckit 不绑定单一运行形态。它既服务人类在 Codex 类 Agent 中完成日常软件项目开发，也服务多 Agent + Loop 的全自动化 AI 平台推进日常研发任务。

两种形态共享同一套项目事实、case 状态、pending、handoff、workflow memory 和 agent context。差异在于执行调度层：Codex 类协作由人和单 Agent 对话推进；自动化平台由多 Agent 角色和 loop 推进。Arckit 不负责替代平台调度、队列、权限、运行环境和通知系统，而负责让这些执行体围绕同一套软件项目语义工作。

## 保留人类接手能力

自动化平台遇到无法可靠解决的问题时，人类必须能随时接手继续操作。接手不应依赖阅读完整聊天记录或猜测 Agent 状态，而应依赖可恢复的项目状态、case record、事实源、handoff、验证结果、pending 和开放问题。

因此，Arckit 的沉淀目标不是“让 Agent 永远自动做完”，而是让任何一轮执行后都能说明：当前目标是什么、已确认事实在哪里、做了什么、没验证什么、哪些判断需要人类、下一步该由谁继续。

## 将 Prompt 视为语义材料

Prompt 不被视为单一任务命令，而是高压缩、多义、上下文依赖的语义材料。

系统需要从 prompt 中识别真实目标、当前阶段、约束、价值判断、实现请求、纠错、证据和未决信息。

该思路允许人保持自然表达，同时把结构化责任交给系统承担。

## 不以表达清晰度评价研发意图

系统不把用户描述是否清晰作为判断软件研发活动是否严肃、合理或应该推进的依据。

人在真实研发中可以用模糊表达提出重要问题，也可以用精确表达提出错误、冲突或不完整的要求。表达清晰只影响系统能直接使用多少显式约束、证据和验收口径，不决定当前阶段本身。

系统应识别的是目标、约束、证据、风险、冲突和待确认项，而不是对用户表达做主观清晰度评分。

## 区分真实目标和当前阶段

真实软件预期通常大于单轮协作能完成的范围。

系统需要持续区分：

- 用户最终想达成的真实软件目标。
- 本轮在当前上下文、成本、权限、证据和确认程度下应该推进到哪里。

该区分使系统既能支持快速执行，也能避免把阶段产物误表达为真实软件已经完成。

## 区分产物角色

系统需要区分最终产物、过程产物和阶段产物。

最终产物稳定改变系统行为。过程产物服务于最终产物的生产、验证、协作和复盘。阶段产物表达本轮推进结果。

该区分避免两个错误：

- 把所有软件研发活动都压缩成实现。
- 把所有讨论和过程材料都误写成正式预期。

## 建立多事实源协同

系统不寻找一个全局唯一事实源。

软件研发至少需要区分：

- 预期事实：应该是什么。
- 实现事实：现在是什么。
- 过程事实：如何判断、探索和未决。
- 工作方式事实：人和 Agent 如何协作。
- Agent 启动上下文：Agent 或自动化平台进入项目时应先知道的仓库规则、读取顺序和事实路由。

预期事实和实现事实同等重要。验证活动负责比较二者，回流机制负责判断差异应该进入修正、确认、待定还是重新推导。

## 使用低承诺沉淀

系统承认自身不能总是可靠判断哪些信息应该成为正式事实。

因此，沉淀机制不做简单的存或不存二分，而是提供低承诺空间承接未确认但可能有价值的信息。

低承诺沉淀用于避免信息丢失，也用于保护正式事实源不被模糊判断污染。

## 用场景化预期验证方案

真实软件研发中，人经常会表达真实使用场景、协作场景和预期结果，但这些表达不一定已经是需求。

系统需要允许这类场景先作为评测资产存在，用来验证产品方案、实现产物或能力单元是否覆盖真实活动。

场景化预期不自动写入正式规格，也不自动变成任务。它先承接真实输入、期望表现、验收重点和暴露风险，再根据验证结果决定是否提升为规格、未决项、任务、测试或工作方式更新。

## 让工作方式可演化

人和 Agent 的协作方式会随着项目、组织约束、AI 能力、成本和用户偏好变化而变化。

系统需要把执行经验、用户纠错和能力变化视为演化输入。它不把当前流程写死为永久模板，而是保留重新推导和版本化方案的能力。

该思路使 Arckit 能在不同 AI 能力假设、行业约束和组织环境下形成不同但清晰的方案版本。

## 解决思路验收口径

解决思路成立时，系统表现为：

- 不要求用户先结构化表达。
- 不把 prompt 默认压缩成实现任务。
- 不把表达清晰度作为阶段判断或价值判断的主轴。
- 能说明真实目标和当前阶段的差异。
- 能说明当前产物在研发链路中的角色。
- 能区分预期事实、实现事实、过程事实和工作方式事实。
- 能说明人类、单 Agent 和多 Agent 平台如何共享同一套项目事实和 handoff。
- 能说明自动化失败或不确定时，人类如何基于 Arckit 记录接手。
- 能在不确定时使用低承诺空间。
- 能把真实使用场景作为评测资产，而不是直接混同为需求。
- 能根据用户纠错和 AI 能力变化重新推导工作方式。
