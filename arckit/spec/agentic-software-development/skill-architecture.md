# Skill 架构

## 架构定位

Arckit 的 skill 体系是 Agent 软件研发操作层的能力实现。每个 skill 是一个可复用的软件研发能力单元。Skill 不只是提示词素材，也是 AI Agent 时代的行为代码。

Skill 体系的目标不是让单个 skill 在关键词触发时孤立工作，而是让多个 skill 根据当前研发阶段、产物类型、事实源类型和用户目标组合成可迭代工作流。

本规格描述预期架构，不描述当前仓库实现完成度。实际 skill 可以逐步向该架构收敛。

## Skill 类型

Arckit skill 分为以下类型。类型用于编排，不要求每个 skill 只能属于一个类型。

- 入口编译型：识别场景、编译 workflow frame、选择能力组合。
- 后续适配型：处理用户补充、纠错、目标变化和暂停。
- 过程思考型：生成分析、比较、草案、模型和 handoff。
- 结果维护型：维护长期事实源。
- 项目治理型：维护计划、责任、迭代、任务和 roadmap。
- 工程执行型：生产最终产物或指导工程动作。
- 质量验证型：验证实现是否符合预期并识别风险。
- 评测场景型：维护真实场景预期和验收样本，用于检查产品方案、实现产物和 skill 行为覆盖。
- 交付运行型：处理发布、上线、运营和运行观察。
- 记忆演化型：沉淀 workflow 经验和执行记录。
- Skill 产物型：创建、验证、同步和治理 skill 自身。

## 编排原则

Skill 编排遵循以下规则：

- 入口层先判断问题背景、当前阶段和产物类型，再选择 skill。
- 用户点名某个 skill 时，系统使用该 skill，但仍检查是否存在其他事实源影响。
- 入口层识别显式约束、证据、风险和冲突，不把用户表达是否清晰作为主观流程判断。
- 过程型 skill 输出 handoff，不能直接污染结果事实源。
- 结果型 skill 只维护稳定事实。
- 实现型动作完成后，系统判断是否需要验证和回流。
- 用户表达真实场景预期时，系统先判断其应进入评测集、规格、pending、任务还是测试，不默认写成需求。
- workflow memory 只记录工作方式，不记录产品事实。
- skill 作为最终产物时，使用类似代码研发的创建、验证、治理和发布流程。

## 入口与适配

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `using-arckit` | Agent 软件研发操作层入口编译器 | 建立 runtime situation，区分真实软件预期和本轮阶段产物，选择 skill 组合，触发事实回流 | 语义输入层、生命周期判断层、产物路由层 |
| `arckit-turn-adaptation` | 后续消息适配器 | 区分补充、纠错、目标变化、事实纠正、暂停和澄清，调整当前 frame | 人的纠错多义性、单轮有限交付 |
| `arckit-role-orchestration` | 显式多角色协作编排能力 | 定义角色、责任和交接，不替代普通单 Agent 编排 | 真实团队生命周期、任务分发阶段产物 |

入口型 skill 不直接替代专门 skill。它负责判断当前任务应由哪些专门 skill 组合完成。

## 记忆与沉淀

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-intake` | 原始输入记录能力 | 保存用户提供的原始材料，不直接提升为需求或决策 | 沉淀不确定性、信息丢失防护 |
| `arckit-pending` | 低承诺未决空间 | 保存尚未确认的想法、风险、开放问题、过程 handoff 和候选事实 | 候选事实、过程产物 |
| `arckit-workflow-memory` | Agent 工作方式过程记忆 | 维护 workflow signal、candidate patch、accepted patch 和 execution record | 工作流事实源、记忆与演化层 |

记忆与沉淀 skill 保护正式事实源。它们允许系统保存不确定信息，而不是在丢失信息和污染规格之间二选一。

## 产品概念与价值

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-idea` | 长期产品创意和商机记录能力 | 保存产品创意或商机，不替代探索和决策 | 意图种子、价值假设 |
| `arckit-idea-explore` | 产品创意深度探索能力 | 梳理目标用户、核心假设、MVP 方向和线框 | explore 阶段、value-discovery |
| `arckit-market-research` | 外部市场证据输入能力 | 获取竞品、趋势、渠道和用户反馈证据 | 产品价值验证、运营素材来源 |
| `arckit-decision-framework` | 横向决策方法能力 | 支持第一性原理、产品价值评估、苏格拉底追问，输出 decision handoff | decide 阶段、价值假设、过程产物 |

产品概念与价值 skill 处理“为什么做”和“是否值得做”。它们不要求价值在开工前绝对正确，但要求价值假设可被表达、验证和修正。

## 定义事实源

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-spec` | 产品预期事实源维护能力 | 维护产品行为、功能边界、验收口径和稳定规则 | 预期事实源、define 阶段 |
| `arckit-interaction` | 交互预期事实源维护能力 | 维护页面流程、状态、线框和交互规则 | design 阶段、预期事实源 |
| `arckit-visual` | 视觉预期事实源维护能力 | 维护视觉风格、tokens、主题和组件视觉规则 | design 阶段、预期事实源 |
| `arckit-tech` | 技术方案事实源维护能力 | 维护架构说明、技术方案、数据模型和 API 契约 | 技术预期事实源、最终产物生产约束 |

结果维护型 skill 只维护稳定事实。草案、猜测、过程分析和未确认判断先进入过程事实源或 pending。

## 思考与方案 Handoff

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-draft-spec` | 需求和规格草案生成能力 | 将模糊输入整理成候选需求、用户故事、验收口径和规格草案 | 从 prompt 到候选预期事实 |
| `arckit-domain-modeling` | 领域模型分析能力 | 梳理实体、值对象、状态、不变量、事件和边界 | 过程事实源、模型桥 |
| `arckit-architecture-decision` | 架构决策分析能力 | 比较架构方案、约束、风险和验证方式 | 技术方案前的过程判断 |
| `arckit-explore-product-design` | 产品设计方向探索能力 | 探索页面方案、状态表达、交互风险和候选方向 | explore 阶段、design 前置产物 |

过程型 skill 输出 handoff。handoff 是候选输入，不自动成为正式规格或技术事实。

## 项目治理与计划

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-project-governance-workflow` | 项目执行控制层 | 管理 Backlog、Goal、Scope、Iteration、Task、Review、Decision 和 Roadmap | plan 阶段、任务拆解过程产物 |
| `arckit-iteration-planning` | 迭代边界规划能力 | 定义当前、下一轮和后续迭代分别解决什么 | 当前可达阶段、单轮有限交付 |
| `arckit-team-responsibility` | 团队职责和 owner 判断能力 | 维护职责范围、决策权和 owner 推断规则 | 人机协同扩展到多人协作 |
| `arckit-work-item-discovery` | 产品级后续工作发现能力 | 阅读项目文档和源码，推荐后续开发事项 | 实现事实与预期事实的差距发现 |

治理与计划 skill 把真实软件预期拆成可推进的阶段目标。它们不替代实现，也不把全部规划压成一个不可维护文档。

## 实现、诊断与质量

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| 普通实现工作流 | 最终产物生产能力 | 修改 code、skill、prompt、workflow patch、配置或脚本 | implement 阶段、最终产物、实现事实源 |
| `arckit-debug-diagnosis` | 问题诊断能力 | 从症状、日志、测试和代码证据定位根因 | 实现事实源调查、预期与实现偏差 |
| `arckit-code-review` | 实现质量审查能力 | 识别 bug、风险、行为回归和缺失测试 | review 阶段、实现事实源 |
| `arckit-verify-implementation` | 实现符合预期的验证能力 | 验证实现是否符合规格、验收口径和质量要求 | 预期事实源与实现事实源比较 |
| 评测场景维护能力 | 真实场景预期和验收样本维护能力 | 保存、分类和更新真实场景评测集，判断场景是否应提升为规格、pending、任务、测试或工作方式更新 | 评测集、过程产物、场景化预期 |

实现、诊断与质量 skill 管理实现事实源。实现完成后，系统判断实现结果是否需要回流到预期事实源、过程事实源或 workflow memory。

评测场景维护能力不替代验证 skill。它维护验收样本；验证 skill 使用规格、实现证据和评测样本判断覆盖情况。

## 交付与运行

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arckit-release-readiness` | 发布准备判断能力 | 判断发布 gate、风险、回滚、灰度和上线准备 | deliver 阶段、真实环境门控 |
| `arckit-runtime-operations` | 运行期观察能力 | 分析运行状态、健康、指标和告警，将反馈转为研发输入 | operate 阶段、实现事实源、运营反馈回流 |
| `app-store-listing-assets` | 应用商店素材与运营交付能力 | 将产品价值、功能和视觉资产转化为运营素材 | 运营产物、产品价值到运营素材 |

交付与运行 skill 把软件带入真实环境。运行期反馈可以改变产品价值、规格、技术方案和后续迭代。

## Skill 产物生命周期

| Skill | 定位 | 作用 | 对应产品概念 |
|---|---|---|---|
| `arcforge-skill-creator` | skill 作为软件产物的创建和维护能力 | 将真实任务、用户纠错和验证反馈转化成可复用 skill 能力单元 | skill 是最终产物、从纠错到稳定能力 |
| `arcforge-skill-first` | skill 产物隔离验证能力 | 用真实任务验证目标 skill 是否能被另一个 Agent 正确执行 | Agent 行为驱动产物的实现事实 |
| `arcforge` | skill 生命周期治理能力 | 扫描、审计、同步、漂移检查、正式化和共享 skill | skill 产物交付与治理 |

当 Arckit 维护 skill 包时，skill 不是辅助材料，而是项目的核心最终产物。Skill 的创建、验证、发布和回流应具备与代码研发类似的工程纪律。

当评测集用于验收 skill 时，真实场景预期相当于代码项目中的验收样本。Skill 修改完成后，系统应能选择相关场景验证该 skill 是否按预期改变 Agent 行为。

## 编排验收口径

Skill 架构满足规格时，系统表现为：

- 不因用户提到某个 skill 就只维护该 skill。
- 能识别该 skill 变更是否还影响产品概念、工作流、事实源或 pending。
- 能区分过程型 skill handoff 和结果型 skill 事实维护。
- 能在实现型任务完成后触发验证和事实回流判断。
- 能在 skill 作为最终产物时使用创建、验证、治理和发布能力。
- 能让多个 skill 根据对话自迭代组合，而不是被入口 skill 写死。
