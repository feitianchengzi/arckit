# 首批覆盖发现

## 结论

基于首批真实研发活动场景，当前产品方案的大方向成立：它能解释 prompt 多义性、阶段判断、最终产物不只代码、多事实源、低承诺沉淀、评测集和工作方式演化。

但当前体系存在两个核心风险：一是场景没有按最终产物类型分开时，skill 类项目和 code 类项目的验收重点会混在一起；二是 code 类场景如果只覆盖模糊输入或只覆盖明确输入，都会漏掉真实项目中的另一半能力要求。

这些发现是评测结论，不自动等于正式产品规格。是否进入 `arckit/spec/` 或具体 skill，需要后续逐项确认。

## 主要错漏风险

| 编号 | 风险 | 典型场景 | 影响 |
|---|---|---|---|
| F001 | skill 类和 code 类场景混在一起 | SP 全部、CP 全部 | 最终产物不同，验收口径不同，容易得出错误覆盖结论。 |
| F002 | 关键词触发单个 skill，缺少完整 workflow 编排 | SP001、SP004 | 只改直接相关 skill，遗漏产品决策、工作方式和验收规则。 |
| F003 | 评测材料和产品规格边界不清 | SP005、SP006、CP015 | 场景样本被误写进 spec，污染正式产品方案。 |
| F004 | 任务拆解阶段容易被推进成实现 | CP012 | 用户只想计划或分发任务时，Agent 可能过早修改最终产物。 |
| F005 | 模糊功能开发缺少需求边界 | CP002、CP011 | Agent 可能直接实现局部 UI 或接口，遗漏字段、状态、权限、异常和验收口径。 |

## 对产品方案的验证判断

现有产品方案可以解释以下内容：

- 人的 prompt 是多本体表达，不应被默认视为单一任务命令。
- 当前阶段可能只是澄清、计划、验证或沉淀，不一定是实现。
- 当前项目的最终产物可以是 skills，而不是传统代码。
- 普通软件项目的最终产物仍然可以是 code 或可运行系统。
- 过程产物和最终产物都重要，但角色不同。
- 预期事实和实现事实同等重要，需要验证比较。
- 低承诺空间能避免信息丢失和正式事实污染。

现有产品方案仍需要通过后续讨论确认以下内容：

- 是否需要在产品方案中明确“评测资产”与 spec、skill、pending、workflow memory 的关系。
- 是否需要分别定义 skill 作为最终产物、code 作为最终产物时的验收生命周期。
- 是否需要把 artifact impact scan 作为入口编排的固定要求。
- 是否需要为“只做评测、不改产物”的用户意图建立明确阶段。

## 对 skill 体系的优化建议

优先级最高的是入口和编排能力，而不是单个垂直 skill。

建议先验证并优化：

- 入口 skill 是否能识别真实活动和当前阶段。
- 入口 skill 是否能在用户点名某个 skill 时继续做影响扫描。
- turn adaptation 是否能处理用户对落盘位置、事实源边界和停止条件的纠正。
- skill 创建维护流程是否要求配套验收场景。
- 结果型 skill 是否只接收稳定事实，过程型输出是否先进入 handoff 或 pending。
- 发布类入口是否能把 final_goal 和 current_phase 分开，并在选择 git-branching 后继承 Git-only stop condition。

## 后续使用方式

每次修改重要 skill 前，可以先选择 2 到 5 个相关场景作为验收样本。

每次真实试用 skill 后，可以把用户输入和实际偏差追加成新场景，或更新已有场景的评测状态。

当某类场景连续暴露同一问题时，再把问题提升为产品规格、skill 架构或具体 skill 的修订任务。
