# 真实研发活动场景总览

## 定位

本目录维护两类真实研发活动场景。

第一类是 **skill 作为最终产物** 的场景。当前项目本身属于这一类：仓库维护的核心产物是 reusable skills，类似传统软件项目中的代码。

第二类是 **code 作为最终产物** 的场景。它对应更常见的软件开发项目：最终交付对象是应用、服务、库、脚本、配置和可运行系统。

当前项目存在的目的不是只维护当前 skill 仓库，而是为了形成一种能支持后续开发 skill 类项目和 code 类项目的 Agent 软件研发方法。因此，两类场景都必须被评测集覆盖。

## 为什么分开

两类项目共享同一套底层产品概念：prompt 多义性、真实软件预期、当前阶段、阶段产物、最终产物、过程产物、多事实源、低承诺空间和评测集。

但两类项目的验收重点不同。

| 最终产物类型 | 最终产物是什么 | 实现事实来自哪里 | 主要验收重点 |
|---|---|---|---|
| skill 类项目 | `SKILL.md`、workflow 规则、prompt 约束、安装/同步配置 | Agent 行为、skill 文件、真实任务试跑结果 | skill 是否稳定改变 Agent 行为，是否被正确触发和组合 |
| code 类项目 | App、服务、库、脚本、配置、部署产物 | 代码、运行结果、测试、日志、用户行为 | 软件是否按预期运行，是否满足真实用户和业务场景 |

如果不分开，容易出现两个问题：

- 用 code 项目的验收习惯评估 skill，忽略 skill 的真实效果是 Agent 行为变化。
- 用 skill 项目的编排问题解释 code 项目，忽略代码实现、运行、测试、发布和线上反馈的工程事实。

## 文件

- `skill-product-scenarios.md`：当前项目维护场景，最终产物是 skills。
- `code-product-scenarios.md`：标准软件开发场景，最终产物是 code 或可运行软件系统。
- `coverage-findings.md`：基于两类场景汇总产品方案和 skill 体系的覆盖问题。

## 使用方式

评估 Arckit 当前项目时，优先使用 `skill-product-scenarios.md`。

评估 Arckit 是否能支持普通软件项目时，优先使用 `code-product-scenarios.md`。

评估产品方案是否具有通用性时，必须同时使用两类场景。只有同时覆盖 skill 类项目和 code 类项目，才能说明当前项目沉淀的方法具有通用 Agent 软件研发价值。
