# Code 类最终产物场景

## 定位

本文件维护标准软件开发项目的真实研发活动场景。

这类项目的最终产物是 code 或可运行软件系统，包括 App、Web 服务、库、脚本、配置、部署产物和运行环境。它们用于验证 Arckit 是否不仅能维护 skill 项目，也能支撑普通软件项目的完整研发流程。

真实 code 项目的入口覆盖从模糊到明确的完整光谱。高频输入通常是模糊功能开发、bug 现象、用户反馈、体验不对、性能变慢、数据异常、线上告警、第三方集成失败或只是一句“这里不对”；同时，真实用户也会给出明确字段、复现步骤、技术约束、验收标准和禁止范围。

因此，本文件同时维护约束较少的样本和约束较多的输入变体。前者用于检验意图识别、证据补齐、必要确认和范围限制；后者用于检验 Agent 是否尊重显式约束、减少无意义追问、控制范围并按标准验收。约束显式程度不是对用户表达是否清晰的主观评分。

## CP001 模糊产品目标启动

用户输入样本：我要做一个待办软件。

约束显式程度：少。

明确输入变体：我要做一个本地优先的待办 App，第一版只支持任务标题、完成状态、今天/全部两个列表和本地持久化，不需要账号和同步。

真实研发活动：人用一句高度压缩的话表达真实软件目标。

本轮真实目标：识别这句话背后的真实软件预期，并判断当前应该做目标澄清、需求草案、原型实现、任务拆解还是直接实现。

期望能力组合：入口编排、需求草案、决策分析、项目治理或实现能力按用户后续选择组合。

期望产物：当前任务处境、可选推进路径、必要确认点；若用户要求继续，则形成对应阶段产物。

期望沉淀路径：未确认目标进入低承诺空间；确认后的需求进入产品规格；工作方式偏好进入工作方式事实。

验收重点：Agent 不应默认立即写完整软件，也不应强制用户先写完整 PRD。

当前风险或疑问：入口 skill 可能仍倾向选择一个最直接执行路径，而不是呈现阶段选择。

评测状态：待试跑。

## CP002 模糊功能开发请求

用户输入样本：这个列表页能不能加个筛选？现在东西多了有点不好找。

约束显式程度：少到中。

明确输入变体：列表页加状态和创建时间筛选，默认全部状态、最近 30 天；筛选条件要体现在 URL query，刷新和分享链接后保持筛选结果。

真实研发活动：人基于使用感受提出功能改进，但没有给出完整需求、字段、交互和验收标准。

本轮真实目标：识别这是功能开发入口，先判断是否需要补需求、交互、数据模型或直接做小范围实现。

期望能力组合：入口编排、产品规格草案、交互判断、实现能力、验证。

期望产物：筛选能力边界、默认筛选维度、交互规则、实现变更或待确认问题。

期望沉淀路径：确认后的筛选行为进入 spec 或 interaction；实现进入代码；未确认筛选维度进入 pending。

验收重点：Agent 不应直接按自己猜测加一个筛选控件后结束，应识别缺失字段、默认状态、空结果、组合筛选和验收口径。

当前风险或疑问：功能开发入口容易跳过需求边界，直接实现局部 UI。

评测状态：待试跑。

## CP003 模糊体验问题

用户输入样本：这个页面用起来怪怪的，感觉操作不顺。

约束显式程度：少。

明确输入变体：编辑页保存后不要直接返回列表，先停留在当前页并显示保存成功；失败时保留用户输入并把接口错误展示在表单顶部。

真实研发活动：人报告体验问题，但没有明确 bug 或具体需求。

本轮真实目标：把模糊体验反馈拆成可观察问题，判断是交互流程、视觉层级、性能、文案、状态反馈还是信息架构问题。

期望能力组合：入口编排、产品设计探索、交互维护、实现能力或 pending。

期望产物：体验问题假设、需要观察的用户路径、候选改进方向、待确认项或小范围修改。

期望沉淀路径：稳定交互规则进入 interaction；不确定体验判断进入 pending 或 evaluation；实现变更进入代码。

验收重点：Agent 不应把“怪怪的”直接翻译成某个自认为合理的 UI 改动。

当前风险或疑问：模糊体验反馈容易被过度设计或过早实现。

评测状态：待试跑。

## CP004 用户反馈驱动改进

用户输入样本：有几个用户说找不到导出入口，你看看是不是需要改一下。

约束显式程度：中。

明确输入变体：把导出入口从更多菜单移到列表右上角，按钮文案为“导出”，仅管理员可见；点击后保留原 CSV 导出逻辑。

真实研发活动：外部用户反馈进入产品改进流程。

本轮真实目标：判断反馈代表真实可用性问题、信息架构问题、文案问题还是用户教育问题，并给出处理路径。

期望能力组合：入口编排、用户反馈分析、产品规格或交互判断、实现能力、评测场景维护。

期望产物：反馈归因、候选改进、验收场景、必要实现变更。

期望沉淀路径：反馈事实进入 intake 或过程事实；确认后的改进入 spec/interaction；场景进入 evaluation；实现进入代码。

验收重点：Agent 不应只说“加个更明显的按钮”，也不应无视用户反馈来源和重复程度。

当前风险或疑问：用户反馈和产品需求之间的转换门禁不清。

评测状态：待试跑。

## CP005 模糊 bug 报告

用户输入样本：登录有时候会失败，但我也不知道怎么复现。

约束显式程度：少。

明确输入变体：用户在 iOS 18.5、弱网环境下用手机号验证码登录，点击提交后偶尔停在 loading，服务端日志没有收到 verify 请求。

真实研发活动：人报告偶发 bug，缺少稳定复现步骤。

本轮真实目标：收集症状、环境、日志和可证伪假设，定位最可能问题或形成后续排查计划。

期望能力组合：debug 诊断、运行观察、日志分析、pending。

期望产物：症状摘要、假设列表、需要采集的证据、最小修复或待复现记录。

期望沉淀路径：已知事实进入 bug 诊断记录；未复现事项进入 pending；确认修复进入代码和验证结果。

验收重点：Agent 不应在没有证据时随意改登录逻辑，也不应因为不能复现就结束。

当前风险或疑问：偶发 bug 容易被压成猜测式修复。

评测状态：待试跑。

## CP006 具体 bug 修复请求

用户输入样本：点保存后页面提示成功，但刷新后数据没变。

约束显式程度：中。

明确输入变体：编辑客户资料时接口返回 200，response 里 name 是新值，但刷新页面仍显示旧值；Network 里后续 GET /customers/:id 返回旧值。

真实研发活动：人报告可观察的功能不一致，通常需要定位前端、接口、缓存或后端持久化问题。

本轮真实目标：判断预期行为和实际行为差异，定位根因并完成最小修复。

期望能力组合：debug 诊断、实现能力、实现验证、必要时技术方案维护。

期望产物：根因判断、最小修复、回归验证、剩余风险。

期望沉淀路径：修复进入代码；验证结果进入过程事实；若发现规格缺失，回流 spec。

验收重点：Agent 应先证明数据没变的原因，不应只改提示文案或盲目刷新状态。

当前风险或疑问：实现事实和预期事实的比较门禁需要足够明确。

评测状态：待试跑。

## CP007 数据异常

用户输入样本：后台统计数和列表里的数量对不上，你帮我看看。

约束显式程度：少到中。

明确输入变体：统计卡片显示“待处理订单 42”，列表在相同筛选条件下只有 37 条；统计接口是 /stats/orders，列表接口是 /orders?status=pending。

真实研发活动：人发现数据一致性问题，但不确定是统计逻辑、筛选条件、权限、缓存还是数据源差异。

本轮真实目标：明确两个数字各自的定义、来源和计算口径，定位差异原因。

期望能力组合：debug 诊断、领域模型分析、技术方案、实现验证。

期望产物：数据口径说明、差异根因、修复或规格修订建议。

期望沉淀路径：稳定数据口径进入 spec 或 tech；修复进入代码；未确认口径进入 pending。

验收重点：Agent 不应只让两个数字显示一致，而应澄清它们是否本来就应该一致。

当前风险或疑问：数据异常场景容易把实现修复和口径决策混在一起。

评测状态：待试跑。

## CP008 性能问题

用户输入样本：最近首页打开有点慢，你看看是不是哪里写得不太好。

约束显式程度：少。

明确输入变体：首页首屏从 1 秒变成 4 秒，Chrome performance 显示主要耗时在 /api/dashboard，最近合并了统计接口聚合逻辑。

真实研发活动：人基于感知报告性能问题，但没有指标、设备、网络和复现条件。

本轮真实目标：建立性能基线，定位瓶颈，区分前端渲染、接口、数据库、资源加载或环境因素。

期望能力组合：debug 诊断、运行观察、实现能力、验证。

期望产物：性能证据、瓶颈判断、优化方案、优化实现或后续采集计划。

期望沉淀路径：性能基线进入过程事实；稳定性能要求进入 spec 或 tech；优化进入代码。

验收重点：Agent 不应凭感觉改代码，也不应只做无证据的微优化。

当前风险或疑问：性能场景需要先采证，和普通 bug 修复不同。

评测状态：待试跑。

## CP009 第三方集成失败

用户输入样本：支付回调好像没生效，订单一直没变成已支付。

约束显式程度：中。

明确输入变体：支付宝回调能打到 /payment/alipay/callback，验签通过，日志显示 update order affectedRows=0，订单状态仍是 pending。

真实研发活动：外部系统集成出现异常，涉及接口、签名、网络、幂等、状态机和日志。

本轮真实目标：明确回调是否到达、是否验签通过、是否更新状态、是否被幂等或异常吞掉。

期望能力组合：debug 诊断、技术方案、实现能力、验证。

期望产物：集成链路检查结果、根因、修复、回归验证和监控建议。

期望沉淀路径：稳定集成约束进入 tech；订单状态预期进入 spec；修复进入代码；未确认外部平台问题进入 pending。

验收重点：Agent 不应只改订单状态更新代码，而要检查完整集成链路。

当前风险或疑问：外部依赖问题容易被误判为本地代码问题。

评测状态：待试跑。

## CP010 线上运行异常回流研发

用户输入样本：线上这两天报错变多了，你先看看是不是某个版本引入的问题，整理后续处理建议。

约束显式程度：中。

明确输入变体：7 月 1 日发布 v1.8.3 后 Sentry 中 CheckoutView 的 NullPointer 增加到每天 200 次，主要发生在 Android 14。

真实研发活动：真实运行反馈触发研发回流。

本轮真实目标：分析运行证据，判断是否存在回归、风险和后续处理路径。

期望能力组合：runtime operations、debug 诊断、项目治理、pending。

期望产物：运行观察、假设、证据、建议任务或待确认项。

期望沉淀路径：运行事实进入运行观察；确认后的 bug 进入任务或实现修复；不确定判断进入 pending。

验收重点：Agent 应区分运行事实、诊断假设和修复动作。

当前风险或疑问：运行期观察是否会被直接压成代码修改。

评测状态：待试跑。

## CP011 需求变更但影响范围不清

用户输入样本：用户现在需要一个批量操作，应该不难吧？

约束显式程度：少到中。

明确输入变体：订单列表增加批量取消，只有 pending 状态可选；一次最多 50 条；失败项要单独提示；操作需要二次确认并写入审计日志。

真实研发活动：人提出看似简单的功能变更，但实际可能影响权限、选择状态、接口、错误处理、性能和审计。

本轮真实目标：做影响扫描，识别功能边界和风险，再决定是否实现。

期望能力组合：入口编排、产品规格、技术方案、实现能力、验证。

期望产物：影响范围、关键问题、实现方案或任务拆解。

期望沉淀路径：确认后的批量操作规则进入 spec/interaction/tech；实现进入代码；未决项进入 pending。

验收重点：Agent 不应被“应该不难”诱导成直接改代码。

当前风险或疑问：影响扫描在普通功能开发中容易缺失。

评测状态：待试跑。

## CP012 用户只想拆解和分发任务

用户输入样本：这个功能先不要实现，你帮我拆清楚哪些要我自己看，哪些可以让其他人处理，哪些后续再说。

约束显式程度：多。

明确输入变体：不要改代码，只输出三组事项：我确认的产品问题、需要设计确认的交互问题、后端需要评估的接口问题；每项带阻塞原因。

真实研发活动：人处于计划和协作分发阶段，不要求生产最终产物。

本轮真实目标：形成任务分解、责任边界、后续跟进和暂缓项。

期望能力组合：入口编排、项目治理、团队责任、pending。

期望产物：任务清单、owner 或确认口径、暂缓项、后续入口。

期望沉淀路径：稳定计划进入治理文档；未定事项进入 pending；不写入产品规格主线。

验收重点：Agent 应在计划阶段停止，不应擅自写代码。

当前风险或疑问：入口 workflow 是否能识别“只拆解不实现”的停止条件。

评测状态：待试跑。

## CP013 代码实现后的质量审查

用户输入样本：这个功能已经写完了，先别加新功能，帮我 review 有没有明显 bug、遗漏和多余实现。

约束显式程度：多。

明确输入变体：只 review 本次 diff，不做重构；重点看保存流程、权限判断和缺失测试；输出 findings-first，按严重程度排序。

真实研发活动：人要求对 code 最终产物做质量审查，而不是继续开发。

本轮真实目标：识别实现风险、行为回归、缺失测试和不必要复杂度。

期望能力组合：代码审查、实现验证、必要时 debug 诊断。

期望产物：按严重程度排序的 findings、证据位置、剩余风险。

期望沉淀路径：审查结论作为过程事实；确认后的修复进入任务或代码；稳定规则可进入 tech 或 workflow memory。

验收重点：Agent 应以 findings-first 输出，不应默认直接重构。

当前风险或疑问：review 阶段是否会被误推进为实现阶段。

评测状态：待试跑。

## CP014 发布前检查

用户输入样本：这个版本准备上线，帮我判断还能不能发，有什么风险，怎么回滚。

约束显式程度：中。

明确输入变体：只做发布前检查，不执行发布；检查 v2.3.0 相比 v2.2.9 的数据库 migration、登录流程和支付回调风险，并给出回滚步骤。

真实研发活动：人要求发布 gate 和风险判断。

本轮真实目标：检查上线准备、风险、回滚、灰度和阻断项。

期望能力组合：release readiness、验证、项目治理、必要时运行观察。

期望产物：go/no-go 判断、风险清单、回滚准备、阻断项。

期望沉淀路径：发布判断作为过程事实；稳定发布规则进入 spec、tech 或 workflow memory。

验收重点：Agent 不应把发布准备等同于立即执行上线。

当前风险或疑问：发布判断与实际发布动作的边界需要验证。

评测状态：待试跑。

## CP015 只做评测不改代码

用户输入样本：你先别改代码，拿几个真实用户场景验证一下现在的方案有没有漏。

约束显式程度：多。

明确输入变体：不要改代码，也不要改 spec；只新增或更新 evaluation 场景，并输出当前方案覆盖不足清单。

真实研发活动：人要求先做验收分析，而不是修改最终产物。

本轮真实目标：生成或执行评测场景，输出覆盖结论。

期望能力组合：评测场景维护、决策分析、必要时 pending。

期望产物：场景样本、覆盖分析、问题清单、后续优化建议。

期望沉淀路径：场景进入 evaluation；稳定结论经确认后再进入 spec、任务或测试。

验收重点：Agent 应先分析材料性质和落盘位置，再决定是否修改正式文档或代码。

当前风险或疑问：Agent 容易把评测材料误写成产品方案，或直接开始改代码。

评测状态：待试跑。
