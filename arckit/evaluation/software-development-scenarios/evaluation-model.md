# 评测模型

## 评测对象

本评测集的主要对象是 Arckit 面向两类最终产物的研发支撑能力。

在当前项目中，skills 是核心最终产物，类似传统软件项目中的代码。与此同时，当前项目沉淀的方法需要支持普通 code 项目。评测场景用于检查 Arckit 是否能同时支撑 skill 类项目和 code 类项目。

评测对象包括：

- 入口路由和工作流编排是否正确。
- 单个 skill 的职责边界是否清晰。
- 多个 skills 是否能按阶段组合。
- 过程型 handoff 是否被正确交给结果型事实源。
- 结果型文档是否只承接稳定事实。
- skill 作为最终产物时，是否经过创建、验证、发布和回流。
- code 作为最终产物时，是否经过定义、实现、测试、发布和运行反馈回流。

## 评测材料是什么

评测材料是场景化验收样本。

它们不是产品需求本身，而是用来检验产品方案和 skill 实现是否成立的样本。它们的作用类似端到端验收测试和 benchmark corpus。

一个场景不是“用户点击了什么”，而是“人在真实研发活动中说了什么、想推进什么、Agent 应该怎么判断、应该产生什么、应该沉淀到哪里”。

## 评测维度

| 维度 | 检查问题 |
|---|---|
| 约束显式程度 | 场景输入给出了多少显式约束、证据和验收口径；Agent 是否正确使用这些信息。 |
| 意图识别 | Agent 是否识别用户输入背后的真实研发活动，而不是只按关键词触发单个 skill。 |
| 阶段判断 | Agent 是否判断当前应推进概念、决策、任务、实现、验证、发布、运行还是沉淀。 |
| 能力组合 | Agent 是否选择并组合了必要 skills，而不是孤立使用直接相关 skill。 |
| 产物区分 | Agent 是否区分最终产物、阶段产物和过程产物。 |
| 事实路由 | Agent 是否把信息写入正确事实源，或在不确定时进入低承诺空间。 |
| 实现验证 | 当存在最终产物变更时，Agent 是否验证实现是否符合预期。 |
| 回流判断 | 实现、验证、纠错和运行反馈是否反向更新预期事实、过程事实或工作方式事实。 |
| 终止条件 | Agent 是否在正确位置停止，并给出清楚的结果、剩余风险和后续入口。 |

## 覆盖状态

| 状态 | 含义 |
|---|---|
| 通过 | 当前产品方案和 skill 体系能完整解释并执行该场景。 |
| 部分通过 | 能推进主要目标，但存在阶段、组合、沉淀、验证或回流缺口。 |
| 未通过 | 容易误解用户目标、遗漏关键产物、只触发单个 skill 或污染事实源。 |
| 待试跑 | 场景已定义，但尚未用真实 Agent 执行验证。 |

## 场景模板

每个场景使用以下结构维护：

- 场景编号
- 场景名称
- 最终产物类型
- 约束显式程度
- 用户输入样本
- 明确输入变体
- 真实研发活动
- 本轮真实目标
- 当前项目最终产物视角
- 期望能力组合
- 期望产物
- 期望沉淀路径
- 验收重点
- 当前风险或疑问
- 评测状态

## 判定原则

一个场景通过，不要求 Agent 一次完成真实世界中的全部软件研发目标。

一个场景通过，要求 Agent 能正确识别真实目标和当前阶段的差异，给出合理推进边界，并把本轮结果沉淀到正确位置。

约束较少的输入通过时，Agent 需要补齐缺失信息、提出必要确认点或限制实现范围。

约束较多的输入通过时，Agent 需要尊重已给出的约束、减少无意义追问、不擅自扩大范围，并按明确验收标准验证。

约束显式程度不是对用户表达是否清晰的主观评分，也不是阶段判断主轴。

当场景暴露 skill 内容不够硬、不够具体或职责过宽时，优先修订对应 skill。当场景暴露入口编排、事实源路由或阶段判断错误时，优先修订入口和工作流类能力。
