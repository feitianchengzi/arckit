# Skill 类最终产物场景

## 定位

本文件维护当前项目的核心验收场景。

当前项目的最终产物是 reusable skills。Skill 在这里不是辅助文档，而是能稳定改变 Agent 行为的最终产物，类似传统软件项目中的代码。

这些场景用于验证 Arckit 是否能支持 skill 的创建、维护、验证、发布、治理和持续演化。

## SP001 试用某个 skill 后提出流程纠错

用户输入样本：刚才我实际试用 xxx skill，运行流程 xxx，里面有很多不应该出现的步骤。我的预期是这个 skill 只做分支规范和 tag 规范，优先根据用户 prompt 推荐配置，确认后直接操作，如果要发布版本，就只通过 push 到远程仓库触发 Xcode Cloud workflow。

真实研发活动：人基于真实试用反馈纠正一个最终产物 skill 的职责边界和工作流。

本轮真实目标：维护目标 skill，同时识别这段话里包含的产品决策、流程约束和可复用工作方式。

期望能力组合：入口编排、skill 创建维护、workflow 纠偏、产品规格或决策沉淀、必要时 pending。

期望产物：目标 skill 变更、职责边界说明、相关产品决策记录、必要的工作方式信号。

期望沉淀路径：稳定产品决策进入 spec；skill 行为进入 skill 文件；未确认平台扩展进入 pending；工作方式纠错进入 workflow memory。

验收重点：Agent 不应只维护被点名的 skill，而忽略这段反馈对产品概念、工作流和验收标准的影响。

当前风险或疑问：现有体系容易关键词触发单个 skill，缺少 artifact impact scan。

评测状态：待试跑。

## SP002 发布意图触发分支和远端 workflow

用户输入样本：我要发一个版本，按当前项目规则处理分支和 tag，发布构建让远端 workflow 触发，本地不要构建。

真实研发活动：人表达发布意图，但具体执行边界只包含分支、tag 和远端触发。

本轮真实目标：按项目发布规则选择分支和 tag，确认后通过 push 触发远端 workflow。

期望能力组合：入口编排、git 分支发布能力、必要确认、release readiness 仅作为判断输入。

期望产物：分支或 tag 操作、push 结果、远端 workflow 触发说明。

期望沉淀路径：项目级发布工作方式进入 workflow memory；稳定发布规则进入对应 skill 或 spec。

验收重点：Agent 应遵守“只通过 push 触发远端 workflow”的边界，不越权做本地构建、打包、上传或商店发布。

当前风险或疑问：发布相关 skill 可能混入不该出现的步骤。

评测状态：待试跑。

## SP003 把反复纠错沉淀成新 skill

用户输入样本：以后遇到这种需求，你都应该先判断是不是只需要沉淀文档，不要直接实现。这个规则能不能做成 skill？

真实研发活动：人把重复工作方式提炼成新的 Agent 行为产物。

本轮真实目标：创建或更新 skill，使其稳定改变 Agent 后续行为。

期望能力组合：skill 创建维护、workflow memory、验证场景生成、arcforge 治理。

期望产物：新 skill 或已有 skill 修改、验收场景、安装或同步建议。

期望沉淀路径：skill 文件进入仓库；验证场景进入 evaluation；工作方式信号进入 workflow memory。

验收重点：Agent 不应只把规则写成聊天总结，应形成可验证的最终产物。

当前风险或疑问：skill 创建后是否必须配套真实任务验证。

评测状态：待试跑。

## SP004 维护单个 skill 时影响系统架构

用户输入样本：这个入口 skill 不应该写死工作流，多个阶段和多个 skill 的组合应该基于对话自迭代。

真实研发活动：人纠正一个 skill 的架构原则，同时影响整体产品方案。

本轮真实目标：更新入口 skill 的职责边界，并判断是否需要修订产品架构或 skill 架构。

期望能力组合：skill 维护、产品规格、架构决策、workflow 纠偏。

期望产物：入口 skill 调整、产品架构或 skill 架构更新、影响说明。

期望沉淀路径：稳定架构原则进入 spec；具体行为进入 skill；开放问题进入 pending。

验收重点：Agent 应做影响扫描，不应只改入口 skill 文本。

当前风险或疑问：现有入口路由是否会吞掉系统级架构反馈。

评测状态：待试跑。

## SP005 只做评测，不改 skill

用户输入样本：你先别改 skill，拿几个真实场景模拟一下，看现在这个方案能不能覆盖。

真实研发活动：人要求先做验收分析，而不是修改最终产物。

本轮真实目标：生成或执行评测场景，输出覆盖结论。

期望能力组合：评测场景维护、决策分析、必要时 pending。

期望产物：场景样本、覆盖分析、问题清单、后续优化建议。

期望沉淀路径：场景进入 evaluation；稳定结论经确认后再进入 spec 或 skill。

验收重点：Agent 应先分析材料性质和落盘位置，再决定是否修改正式文档。

当前风险或疑问：Agent 容易把评测材料误写成产品方案。

评测状态：本场景由当前任务触发，待后续复盘。

## SP006 后续对话纠正落盘位置

用户输入样本：我不希望放在 arckit/spec 下面，你感觉呢？

真实研发活动：人纠正产物归属和事实源边界。

本轮真实目标：重新判断材料性质，必要时迁移或重建产物位置。

期望能力组合：turn adaptation、artifact routing、spec 维护或评测资产维护。

期望产物：新的归属判断、文件迁移或重建、旧引用清理。

期望沉淀路径：稳定归属规则可进入 workflow memory；本次评测材料进入 evaluation。

验收重点：Agent 不应机械维护已写错的 spec，而应回到材料本质重新判断。

当前风险或疑问：后续消息对当前 workflow frame 的影响是否被识别。

评测状态：本场景由当前任务触发，待后续复盘。

## SP007 不确定想法只应低承诺保存

用户输入样本：我突然想到一个可能有用的点，但不确定对不对，先记一下，别写进正式方案。

真实研发活动：人提供不稳定但可能有价值的想法。

本轮真实目标：忠实保存，不提升为正式产品事实。

期望能力组合：intake 或 pending，必要时入口编排。

期望产物：未决项或原始输入记录。

期望沉淀路径：进入 pending 或 intake；不进入 spec 主线。

验收重点：Agent 应避免过度分析后直接修改产品方案。

当前风险或疑问：Agent 可能为了“有产出”而污染正式规格。

评测状态：待试跑。

## SP008 AI 能力变化要求重新推导

用户输入样本：如果以后模型上下文足够长、能稳定执行长任务，那现在这些阶段和沉淀规则是不是要重新想？

真实研发活动：人基于 AI 能力演进重新审视产品方案和 skill 工作方式。

本轮真实目标：识别这是能力假设变化，不是普通需求变更。

期望能力组合：产品规格、决策分析、workflow memory 或 pending。

期望产物：能力假设差异、需要重新推导的问题、暂不改动或改动建议。

期望沉淀路径：稳定能力假设进入 spec；远期不确定项进入 pending。

验收重点：Agent 不应把未来能力变化简单当成当前立即改规则。

当前风险或疑问：当前产品方案是否支持多 AI 能力版本评测。

评测状态：待试跑。
