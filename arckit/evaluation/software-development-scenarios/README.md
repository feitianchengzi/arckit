# 软件研发场景评测集

## 定位

本目录是一组可维护的真实研发活动样本，用于验证 Arckit 是否能支撑两类项目：以 skills 为最终产物的项目，以及以 code 为最终产物的标准软件项目。

当前项目本身的最终产物是 skills，但当前项目存在的目的，是为了后续能更好地开发 skill 类项目和 code 类项目。因此评测集必须同时覆盖这两条轨道。

它类似传统软件项目中的验收测试、端到端测试和基准样本集。区别是这里的验收对象既可以是代码行为，也可以是 Agent 在一组 skills 约束下的研发协作行为。

## 不是什么

本目录不是产品规格。产品规格仍由 `arckit/spec/` 维护。

本目录不是交互流程。交互流程关注界面、状态和操作路径；这里关注真实软件研发活动如何被 Agent 理解、拆解、执行和沉淀。

本目录不是一次性分析记录。场景需要持续维护、补充、修正和归档。

本目录不是普通单点测试。它不只检查某个 skill 是否被触发，而是检查多个 skills 是否能围绕真实目标形成有效工作流。

## 为什么需要

人在真实软件研发中发出的 prompt 往往是高压缩、多义、阶段不明确的。一个 prompt 可能同时包含产品目标、纠错、工作方式偏好、实现请求、沉淀要求和未决问题。

如果只验证单个 skill 能否响应关键词，就会出现当前已经观察到的问题：直接相关的 skill 会工作，但完整研发流程没有被组合起来，过程思考、事实回流和产品决策会丢失。

本目录通过模拟真实场景来检验：

- 用户输入背后的真实研发活动是否被识别。
- 当前阶段是否被正确判断。
- 应该组合的 skills 是否被组合。
- 过程产物、最终产物和事实源是否被正确区分。
- skill 类项目是否具备类似传统代码项目的验收纪律。
- code 类项目是否能被同一套 Agent 软件研发操作层稳定支撑。
- 场景暴露的问题是否能反向驱动产品方案和 skill 内容优化。

## 文档结构

- `evaluation-model.md` 定义评测对象、维度、状态和场景模板。
- `scenarios.md` 维护两类场景的总览和使用方式。
- `skill-product-scenarios.md` 维护以 skills 为最终产物的当前项目场景。
- `code-product-scenarios.md` 维护以 code 为最终产物的标准软件开发场景。
- `coverage-findings.md` 汇总基于首批样本发现的产品方案和 skill 体系风险。

## 维护规则

新增场景时，优先记录真实或近似真实的用户输入，不要只写抽象类型。

每个场景至少包含：用户输入样本、真实活动、本轮目标、期望能力组合、期望产物、沉淀路径、验收重点和暴露风险。

场景可以先保留为“待验证”。经过实际试跑后，再补充实际表现、偏差和后续优化建议。

当某个场景反复暴露同类问题时，再把结论提升到 `arckit/spec/`、具体 skill 内容或 workflow memory 中。
