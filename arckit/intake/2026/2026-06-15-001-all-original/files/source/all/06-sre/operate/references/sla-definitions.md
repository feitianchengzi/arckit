# SLA定义与告警规则

---

## 一、核心概念

| 概念 | 全称 | 含义 | 示例 |
|------|------|------|------|
| **SLI** | Service Level Indicator | 衡量服务的具体指标 | P95延迟=320ms |
| **SLO** | Service Level Objective | 对SLI的目标承诺 | P95延迟<500ms |
| **SLA** | Service Level Agreement | 违反SLO的后果约定 | 可用性<99.9%则退款10% |

**关系：** SLI是事实 → SLO是目标 → SLA是承诺

```
事实(SLI)   →  目标(SLO)  →  承诺(SLA)
P95=320ms   →  P95<500ms  →  超过则退款
```

---

## 二、常见SLI指标

### 可用性

| 指标 | 计算方式 | 数据来源 |
|------|---------|---------|
| 服务可用率 | 成功请求数 / 总请求数 × 100% | 负载均衡/网关日志 |
| 功能可用率 | 功能可用的分钟数 / 总分钟数 × 100% | 合成监控（定时探测） |

**可用性等级：**

| 可用性 | 年度停机时间 | 适用场景 |
|--------|------------|---------|
| 99% | 3天15小时 | 内部工具 |
| 99.9% | 8小时46分钟 | 一般业务系统 |
| 99.95% | 4小时23分钟 | 重要业务系统 |
| 99.99% | 52分钟 | 核心交易系统 |
| 99.999% | 5分钟 | 生命安全/基础设施 |

### 延迟

| 指标 | 计算方式 | 数据来源 |
|------|---------|---------|
| P50延迟 | 50%请求的响应时间 | APM/访问日志 |
| P95延迟 | 95%请求的响应时间 | APM/访问日志 |
| P99延迟 | 99%请求的响应时间 | APM/访问日志 |

**注意：** 延迟SLO应基于服务端处理时间，不包含用户网络延迟。但面向用户的SLO应包含端到端时间。

### 吞吐

| 指标 | 计算方式 | 数据来源 |
|------|---------|---------|
| QPS | 每秒查询数 | 网关日志 |
| RPS | 每秒请求数 | 网关日志 |
| 并发数 | 同时处理的请求数 | 连接池/APM |

### 错误率

| 指标 | 计算方式 | 数据来源 |
|------|---------|---------|
| 5xx率 | 5xx响应数 / 总请求数 | 网关日志 |
| 4xx率 | 4xx响应数 / 总请求数 | 网关日志 |
| 业务错误率 | 业务异常数 / 总请求数 | 应用日志 |

### 数据持久性

| 指标 | 计算方式 | 典型SLO |
|------|---------|---------|
| 数据丢失率 | 丢失对象数 / 总对象数 | <0.000000001% (11个9) |
| 数据恢复时间 | RPO/RTO | RPO<1min, RTO<4h |

## 三、SLO设定方法

### 方法1: 基于历史数据

```bash
# 取最近30天的P95延迟
curl -s 'api.example.com/metrics?name=latency_p95&period=30d'

# 结果：P50=280ms, P95=350ms, P99=800ms, Max=3200ms
# 设定SLO：P95<500ms（留约30%缓冲）
```

**规则：** SLO = 历史P95 × 1.3（30%缓冲）

### 方法2: 基于用户期望

| 用户场景 | 期望延迟 | 推荐SLO |
|---------|---------|---------|
| 搜索/查询 | "秒出" | P95<200ms |
| 提交/保存 | "不卡" | P95<500ms |
| 报表生成 | "能等" | P95<3s |
| 批量处理 | "别太慢" | P95<30s |

### 方法3: 基于竞品对比

| 指标 | 我们 | 竞品A | 竞品B | SLO建议 |
|------|------|-------|-------|---------|
| P95延迟 | 350ms | 200ms | 400ms | <300ms（持平或超过A） |

## 四、告警规则设计

### 静态阈值告警

| 级别 | 延迟告警 | 错误率告警 | 可用性告警 |
|------|---------|-----------|-----------|
| P0 | P95>3s | >5% | <99% |
| P1 | P95>1s | >1% | <99.9% |
| P2 | P95>500ms | >0.5% | <99.95% |
| P3 | P95>300ms | >0.1% | <99.99% |

### 动态基线告警

```
基线 = 过去7天同一时间段（同 weekday + 同 hour）的P50值

告警条件：
  当前值 > 基线 × 2   → P2
  当前值 > 基线 × 3   → P1
  当前值 > 基线 × 5   → P0

适用场景：
  - 流量有明显周期性（工作日vs周末、白天vs夜间）
  - 静态阈值容易误报
```

### 组合条件告警

```
条件1：错误率 > 1%
  AND
条件2：持续 > 5分钟
  AND
条件3：非部署窗口（排除部署导致短暂波动）

→ P1告警

优点：减少瞬时波动导致的误报
```

### 告警分级

| 级别 | 触发条件 | 响应时间 | 通知方式 | 升级 |
|------|---------|---------|---------|------|
| **P0-Critical** | 服务不可用/数据丢失 | <5分钟 | 电话+短信+IM | 15分钟未响应升级到总监 |
| **P1-High** | 核心功能异常 | <30分钟 | 短信+IM | 1小时未响应升级 |
| **P2-Medium** | 性能退化/非核心异常 | <4小时 | IM | 下一工作日跟进 |
| **P3-Low** | 潜在风险/容量预警 | <24小时 | 邮件/工单 | 下周回顾 |

## 五、告警疲劳治理

### 问题诊断

| 症状 | 原因 | 治理方法 |
|------|------|---------|
| 告警太多，开始忽略 | 阈值过灵敏 | 提高阈值，添加持续时间条件 |
| 大量重复告警 | 同一问题触发多个告警 | 告警聚合/去重 |
| 半夜告警频繁 | 非关键告警夜间触发 | 按时段分级：夜间只通知P0/P1 |
| 不知道该不该处理 | 告警级别不清晰 | 严格分级，明确每级的响应要求 |
| 告警后不知道做什么 | 缺少runbook | 每个告警配套处理手册 |

### 治理策略

**1. 告警降噪**

```bash
# 目标：告警数量减少80%
# 手段：
- 合并同一问题的多源告警（如CPU+内存+磁盘同时告警→1条）
- 添加持续时间条件（错误率>1%持续5分钟才告警）
- 排除已知维护窗口
- 抑制级联告警（下游服务因上游故障告警→抑制下游告警）
```

**2. 告警分级**

```
问3个问题：
1. 用户是否直接受影响？→ 是=至少P1
2. 数据是否面临风险？→ 是=P0/P1
3. 是否可等到工作时间处理？→ 是=P2/P3
```

**3. 升级规则**

```
P0 → 5分钟未响应 → 自动升级到oncall组长
P0 → 15分钟未响应 → 自动升级到总监
P1 → 30分钟未响应 → 自动升级到oncall组长
P2 → 4小时未处理 → 转为工单
P3 → 24小时未处理 → 下周回顾讨论
```

## 六、事故响应流程

### P0/P1响应流程

```
发现异常
  ↓
确认影响范围（哪些功能/多少用户）
  ↓
通知团队（事故频道+oncall）
  ↓
止血（回滚/降级/限流）
  ↓
根因调查
  ↓
修复
  ↓
确认恢复
  ↓
24小时内复盘
```
