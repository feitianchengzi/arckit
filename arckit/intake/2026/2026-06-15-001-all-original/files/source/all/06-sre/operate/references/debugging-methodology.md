# 系统化调试方法论

---

## 一、四阶段流程详解

### 阶段1: 根因调查

**核心原则：不要猜测，要观察。**

#### 1.1 仔细读错误信息

```
❌ 错误做法：看到"500 Internal Server Error"就开始猜
✅ 正确做法：读完整个错误堆栈，找到最内层的异常消息和行号
```

**检查清单：**
- [ ] 完整阅读错误消息（不只是第一行）
- [ ] 识别异常类型（NullPointerException / TimeoutException / ...）
- [ ] 定位到具体文件和行号
- [ ] 检查是否有内嵌异常（Caused by）
- [ ] 确认错误发生的环境（dev/staging/prod）

#### 1.2 稳定复现

| 复现难度 | 策略 |
|---------|------|
| 稳定复现 | 直接调试，观察变量状态 |
| 偶尔复现 | 添加更多日志，等待下次触发 |
| 无法复现 | 模拟极端条件（高并发/大数据量/边界输入） |

**复现检查清单：**
- [ ] 记录最小复现步骤
- [ ] 确认复现环境与报错环境一致
- [ ] 排除环境差异（依赖版本/配置/数据）

#### 1.3 检查近期变更

```bash
# 查看最近一天的提交
git log --since="1 day ago" --oneline

# 查看特定文件的变更历史
git log --follow -p -- path/to/file

# 二分法定位引入bug的commit
git bisect start
git bisect bad HEAD
git bisect good <known-good-commit>
```

#### 1.4 逐层加诊断

```
外部（用户操作）
  → 网关层（请求是否到达？参数是否正确？）
    → 服务层（逻辑是否正确执行？）
      → 数据层（SQL是否正确？数据是否符合预期？）
        → 基础设施（网络/DNS/磁盘是否正常？）
```

**在每层添加日志/断点，确认数据流到哪一层开始出错。**

#### 1.5 追踪数据流

```
输入 → 校验 → 转换 → 处理 → 存储 → 输出

问自己：
- 输入是否正确到达？
- 校验是否通过？
- 转换是否改变了数据？
- 处理逻辑是否正确？
- 存储的数据与预期一致吗？
- 输出是否被正确序列化？
```

### 阶段2: 模式分析

#### 2.1 找正常案例

| 问题 | 做什么 |
|------|--------|
| 这个功能有正常工作的时候吗？ | 找到一个成功的case |
| 正常case和异常case有什么不同？ | 逐项对比 |
| 什么时候开始不正常的？ | 时间线 + 变更记录 |

#### 2.2 对比差异

```markdown
| 维度 | 正常Case | 异常Case |
|------|---------|---------|
| 输入参数 | ... | ... |
| 用户状态 | ... | ... |
| 数据量 | ... | ... |
| 时间/并发 | ... | ... |
| 环境/配置 | ... | ... |
```

#### 2.3 理解依赖

- 这个bug依赖特定数据吗？
- 依赖特定时间/顺序吗？
- 依赖特定并发条件吗？
- 依赖外部服务状态吗？

---

### 阶段3: 假设与测试

#### 3.1 形成单一假设

```
❌ "可能是缓存问题，也可能是数据库问题，或者并发bug"
✅ "假设1：缓存未失效导致读取了旧数据"
```

**一次只测一个假设。** 同时测多个假设=做实验不控制变量=不可靠的结论。

#### 3.2 最小变更测试

```
假设：缓存未失效
测试：手动清除缓存，观察问题是否消失
  → 消失 → 假设成立，修复缓存失效策略
  → 未消失 → 假设不成立，回到阶段1形成新假设
```

#### 3.3 ⚠️ 3次失败规则

| 修复尝试 | 应该做的 |
|---------|---------|
| 第1次失败 | 回到阶段1，检查是否遗漏了线索 |
| 第2次失败 | 换一个角度（从数据层→从服务层→从网络层） |
| 第3次失败 | **停止修复**。质疑架构 |

**3次失败后质疑架构的清单：**
- [ ] 这个模块的职责是否清晰？
- [ ] 是否存在循环依赖？
- [ ] 数据一致性模型是否正确？
- [ ] 是否需要引入 `arch` 重新设计？
- [ ] 局部修复是否在累积技术债？

---

### 阶段4: 实现

#### 4.1 先写失败测试

```typescript
// 复现bug的测试
test('订单取消后不应允许再次发货', async () => {
  const order = await createOrder();
  await cancelOrder(order.id);
  
  await expect(shipOrder(order.id)).rejects.toThrow('订单已取消');
  // 修复前：不抛异常，允许发货 → bug复现
  // 修复后：正确抛异常 → 测试通过
});
```

#### 4.2 最小修复

- 只修bug，不重构
- 不添加"顺便优化"的改动
- 修复后运行全量测试确认无回归

## 二、调试工具箱

### 日志分析

```bash
# 实时跟踪日志
tail -f app.log | grep --line-buffered "ERROR"

# 按时间范围查日志
awk '/2026-06-10 14:3[0-9]/' app.log

# 统计错误类型分布
grep "ERROR" app.log | awk -F']' '{print $3}' | sort | uniq -c | sort -rn

# 查看错误前后的上下文
grep -B5 -A5 "NullPointerException" app.log

# 追踪请求ID的完整链路
grep "req-12345" app.log
```

### 性能分析

```bash
# CPU profiling (Node.js)
node --prof app.js
node --prof-process isolate-*.log

# 火焰图
perf record -g node app.js
perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg

# 内存分析
node --inspect app.js
# Chrome DevTools → Memory → Take Heap Snapshot

# 慢查询分析
EXPLAIN ANALYZE SELECT ...
```

### 网络分析

```bash
# 抓包
tcpdump -i eth0 -w dump.pcap port 443

# HTTP请求分析
curl -w "@curl-format.txt" -o /dev/null -s https://api.example.com/health

# DNS解析
dig example.com
nslookup example.com

# 连接状态
netstat -an | grep ESTABLISHED | wc -l
ss -s
```

### 内存分析

```bash
# 进程内存
ps aux --sort=-%mem | head -10

# 系统内存
free -h
vmstat 1 5

# 堆dump (Java)
jmap -dump:format=b,file=heap.hprof <pid>

# 堆dump (Node.js)
node --inspect app.js
# Chrome DevTools → Memory → Take Heap Snapshot
```

## 三、常见Bug模式速查

| # | 症状 | 典型根因 | 检查方向 |
|---|------|---------|---------|
| 1 | 间歇性500错误 | 竞态条件/连接池耗尽 | 并发测试+连接池监控 |
| 2 | 内存持续增长 | 事件监听器未清理/闭包引用 | Heap Snapshot对比 |
| 3 | 首次请求慢/后续快 | 缓存冷启动/连接池初始化 | 预热策略 |
| 4 | 凌晨定时出错 | 定时任务冲突/批处理超时 | cron配置+批处理日志 |
| 5 | 部分用户功能异常 | 数据不一致/缓存脏数据 | 特定用户数据审查 |
| 6 | 修改后功能未生效 | 缓存/CDN未刷新/旧代码 | 确认部署版本+清缓存 |
| 7 | 上传文件失败 | 文件大小超限/磁盘满/权限 | 检查nginx配置+磁盘空间 |
| 8 | 第三方API超时 | 网络问题/限流/对方故障 | curl直接测试+状态页 |
| 9 | 数据库查询突然变慢 | 索引失效/锁等待/数据量突增 | EXPLAIN+慢查询日志 |
| 10 | 跨环境行为不一致 | 配置差异/时区/数据差异 | 对比环境配置 |
| 11 | 高并发时数据错乱 | 乐观锁/原子操作缺失 | 并发测试+数据库约束 |
| 12 | 序列化/反序列化失败 | 字段类型变更/版本不兼容 | 对比接口契约 |
| 13 | 支付金额对不上 | 浮点精度/舍入规则/并发扣减 | 金额用整数(分) |
| 14 | 搜索结果不完整 | ES索引延迟/分词问题 | 直接查ES+对比DB |
| 15 | 证书过期 | SSL/TLS证书未续期 | 检查证书有效期 |
| 16 | Docker容器OOM | 内存限制过小/内存泄漏 | 调整内存限制+内存分析 |
| 17 | 定时任务重复执行 | 多实例同时触发 | 分布式锁 |
| 18 | 日志丢失 | 磁盘满/日志级别不对/缓冲未刷 | 检查磁盘+日志配置 |
| 19 | 请求偶尔返回旧数据 | 缓存不一致/读从库延迟 | 关键路径直查主库 |
| 20 | 服务雪崩 | 级联故障/无熔断 | 熔断器+超时+降级 |

## 四、调试报告模板

```markdown
# 调试报告

## 问题
[一句话描述现象]

## 影响
- 影响范围：[用户数/功能范围]
- 严重程度：P0/P1/P2/P3
- 开始时间：[首次发现时间]

## 根因
[根本原因，不是表面现象]

## 证据链
1. 错误日志显示：{具体日志}
2. 数据流追踪发现：{具体发现}
3. 对比正常案例发现：{具体差异}
4. 假设验证：{假设}→{结果}

## 修复方案
[最小修复，不是重构]

## 验证
- [ ] 修复后测试通过
- [ ] 无回归
- [ ] 根因已消除（不是症状被掩盖）

## 预防措施
[如何防止同类问题再次发生]
```
