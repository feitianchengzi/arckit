# TDD反合理化表

> 每个借口都是绕过纪律的尝试。以下是11种常见借口及事实反驳。

## 1. "太简单了不需要测试"

| | 内容 |
|---|------|
| **事实** | 简单代码也会出bug。`a + b`在处理大数溢出时就会错。测试只需30秒，修bug可能要3小时 |
| **真实案例** | `isAdult(age)` 返回 `age >= 18`，看起来简单。但`age`为负数、小数、字符串时呢？ |
| **正确做法** | 越简单的代码测试越快写——没有理由跳过 |

## 2. "先写代码再补测试"

| | 内容 |
|---|------|
| **事实** | 后补的测试倾向于验证实现而非行为。"代码做了X"≠"代码应该做X"。TDD的核心是先想清楚"什么是对的" |
| **真实案例** | 先写了排序算法再补测试，测试会按算法内部步骤写，而非按排序的数学性质写。换算法时测试全废 |
| **正确做法** | RED→GREEN→REFACTOR。先写失败测试定义期望行为，再写最小实现 |

## 3. "测试写了但跑不通，先跳过"

| | 内容 |
|---|------|
| **事实** | 跑不通=隐藏问题。通常是设计问题（耦合太重/接口不清），不是测试问题 |
| **真实案例** | 测试依赖数据库连接跑不通→说明被测代码直接耦合了数据库，而非通过接口。这是设计缺陷信号 |
| **正确做法** | 跑不通时停下来重构设计，而非skip测试。引入接口/mock/依赖注入 |

## 4. "这个逻辑没法测试"

| | 内容 |
|---|------|
| **事实** | 没法测试=耦合太重。这是重构信号，不是跳过测试的理由 |
| **真实案例** | "发送邮件的逻辑没法测试"→因为直接调用SMTP。提取`EmailSender`接口，测试时用mock |
| **正确做法** | 没法测试的代码先重构到可测试，再写测试。常见手法：提取接口/依赖注入/纯函数 |

## 5. "重构太危险，别动"

| | 内容 |
|---|------|
| **事实** | 有测试保护的重构是安全的。不敢重构=测试不够 |
| **真实案例** | 3年前的"祖传代码"没人敢动，因为0测试。每次改都是盲改，出bug概率>50% |
| **正确做法** | 先给要重构的代码补测试（ characterization test），确认行为基线，再重构 |

## 6. "这个改动太小不需要走TDD"

| | 内容 |
|---|------|
| **事实** | 小改动出大bug的概率不比大改动低。改一行配置可能影响全链路。5分钟的TDD不会浪费 |
| **真实案例** | 改了一个默认超时从30s→60s，看似小事。但下游有个重试3次的逻辑，总超时变成180s，触发雪崩 |
| **正确做法** | 改动大小与风险大小无关。任何行为变更都应有对应测试 |

## 7. "时间紧，先赶进度"

| | 内容 |
|---|------|
| **事实** | 没有测试的代码是技术债。还债的时间是写测试的3-10倍。短期赶的进度，长期会加倍偿还 |
| **真实案例** | Sprint 1赶进度不写测试。Sprint 3开始，每次改功能都要花2小时手动验证回归，效率反而更低 |
| **正确做法** | TDD的初始投入约+15%时间，但3个Sprint后效率反超。长期看是加速而非减速 |

## 8. "UI没法TDD"

| | 内容 |
|---|------|
| **事实** | UI可以测试行为（点击/输入/断言），而非测试实现（DOM结构/CSS类名） |
| **真实案例** | "点击提交按钮后显示成功提示"——这是行为测试，可以用Playwright/Testing Library写 |
| **正确做法** | 用行为驱动的方式写UI测试：用户看到什么→用户做什么→用户期望看到什么。不测DOM结构 |

## 9. "这是原型，不需要测试"

| | 内容 |
|---|------|
| **事实** | 原型变生产的概率>80%。"以后补测试"=永远不会补。从一开始就写测试 |
| **真实案例** | "先做个原型验证想法"→3个月后原型变成线上产品，0测试，线上出bug无法快速定位 |
| **正确做法** | 原型也要写关键路径的测试。如果原型要扔掉，测试也一起扔。如果原型要留下，测试已经在了 |

## 10. "测试覆盖率100%太贵"

| | 内容 |
|---|------|
| **事实** | 关键路径100%，非关键路径>60%。0%不可接受。追求100%的边际成本确实高，但>60%的投入性价比很好 |
| **真实案例** | 支付流程100%覆盖，管理后台的报表导出50%覆盖。前者出bug损失大，后者出bug影响小 |
| **正确做法** | 按风险分级：核心域100%→支撑域80%→通用域60%。用风险矩阵决定覆盖策略 |

## 11. "集成测试比单元测试更有价值"

| | 内容 |
|---|------|
| **事实** | 两者互补，不是替代。单元测试定位问题（精确到哪个函数），集成测试验证协作（模块间是否正确配合） |
| **真实案例** | 集成测试失败了：下单接口返回500。是支付模块的问题？还是库存模块的问题？单元测试能在1秒内定位 |
| **正确做法** | 测试金字塔：单元测试(多)→集成测试(中)→E2E测试(少)。比例约70:20:10 |

## 速查：如何回应这些借口

当你听到任何一种借口时，不要争论，而是问一个问题：

| 借口 | 反问 |
|------|------|
| "太简单不需要测试" | "如果这行代码出bug，修复要多久？" |
| "先写代码再补测试" | "你怎么知道代码做的是对的事？" |
| "跑不通先跳过" | "跑不通说明什么设计问题？" |
| "没法测试" | "如果没法测试，用户怎么能正确使用它？" |
| "重构太危险" | "如果没有测试保护，你怎么确保任何修改不出错？" |
| "改动太小" | "这个改动会影响哪些下游？" |
| "时间紧" | "3个Sprint后，手动回归验证要花多少时间？" |
| "UI没法TDD" | "用户会怎么操作这个界面？" |
| "原型不需要" | "这个原型上线的概率有多大？" |
| "100%太贵" | "这个模块出bug的损失有多大？" |
| "集成测试更好" | "集成测试失败时，你怎么1分钟内定位到具体函数？" |
