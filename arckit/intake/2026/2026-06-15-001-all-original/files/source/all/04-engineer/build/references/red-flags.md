# 编码红灯信号

> 当你看到这些信号时，停下来重新思考。继续写下去只会制造更多技术债。

---

## 一、代码异味信号

### 1. 过长函数（>30行）

| | 内容 |
|---|------|
| **描述** | 一个函数超过30行，通常意味着它做了不止一件事 |
| **为什么危险** | 难以理解、难以测试、难以复用。修改一处可能影响其他逻辑 |
| **怎么修** | 提取子函数，每个子函数只做一件事，名字揭示意图 |

### 2. 过深嵌套（>3层）

| | 内容 |
|---|------|
| **描述** | if/for/try嵌套超过3层 |
| **为什么危险** | 认知负荷指数级增长。边界条件容易被遗漏 |
| **怎么修** | 提前返回（guard clause）、提取函数、使用函数式操作（map/filter/reduce） |

### 3. 过多参数（>4个）

| | 内容 |
|---|------|
| **描述** | 函数参数超过4个 |
| **为什么危险** | 调用方容易传错顺序。新增参数时所有调用方都要改 |
| **怎么修** | 将相关参数封装为对象/结构体（如 `SearchQuery`、`PaginationOptions`） |

### 4. 神类（God Class）

| | 内容 |
|---|------|
| **描述** | 一个类超过500行或持有超过10个字段 |
| **为什么危险** | 违反单一职责。修改任何功能都可能影响其他功能 |
| **怎么修** | 按职责拆分。每个类只有一个变更的理由（SRP） |

### 5. 特征嫉妒（Feature Envy）

| | 内容 |
|---|------|
| **描述** | 一个类频繁访问另一个类的数据 |
| **为什么危险** | 逻辑分散在错误的地方。数据和行为应该在一起 |
| **怎么修** | 将行为移到数据所属的类中（Move Method） |

### 6. 魔法数字/字符串

| | 内容 |
|---|------|
| **描述** | 代码中直接使用 `30`、`"active"` 等字面量 |
| **为什么危险** | 含义不明，修改时容易遗漏 |
| **怎么修** | 提取为命名常量：`const SESSION_TIMEOUT_SECONDS = 30` |

### 7. 重复代码（DRY违反）

| | 内容 |
|---|------|
| **描述** | 相同或高度相似的代码出现在3+个地方 |
| **为什么危险** | 修改一处时容易遗漏其他处，导致不一致 |
| **怎么修** | 提取公共函数/基类/组合。注意：只在第3次重复时才抽象（Rule of Three） |

### 8. 注释解释代码

| | 内容 |
|---|------|
| **描述** | 需要注释解释"这段代码在做什么" |
| **为什么危险** | 注释会过时，代码不会。注释与代码不一致比没有注释更危险 |
| **怎么修** | 重构代码使其自解释：改善命名、提取函数、使用意图明确的类型 |

### 9. 过度使用可变状态

| | 内容 |
|---|------|
| **描述** | 大量 `let`/变量重新赋值，对象被多处修改 |
| **为什么危险** | 状态变更的时序难以追踪，并发时必出bug |
| **怎么修** | 优先使用 `const`/不可变数据结构。函数返回新值而非修改旧值 |

### 10. 布尔参数控制行为

| | 内容 |
|---|------|
| **描述** | `processData(data, true, false, true)` |
| **为什么危险** | 调用方无法理解每个布尔值的含义。新增行为时参数越来越多 |
| **怎么修** | 用枚举/选项对象/拆分为独立函数替代 |

### 11. 异常被吞掉

| | 内容 |
|---|------|
| **描述** | `catch(e) { /* ignore */ }` 或只打印日志不处理 |
| **为什么危险** | 错误被静默忽略，下游代码基于错误状态继续运行，产生更难诊断的bug |
| **怎么修** | 要么处理异常（恢复/降级），要么向上传播（throw/Result类型），绝不静默忽略 |

### 12. 全局状态/单例滥用

| | 内容 |
|---|------|
| **描述** | 大量使用全局变量或单例模式 |
| **为什么危险** | 隐式依赖，测试时无法隔离，并发时状态竞争 |
| **怎么修** | 依赖注入。将依赖作为参数/构造函数参数传入 |

## 二、设计异味信号

### 1. 过度抽象

| | 内容 |
|---|------|
| **描述** | 为只有一个实现创建了接口/抽象类 |
| **为什么危险** | 增加间接层但不增加灵活性。YAGNI——在第二个实现出现之前不要抽象 |
| **怎么修** | 删除抽象，直接使用具体类。第二个实现出现时再提取接口 |

### 2. 抽象不足

| | 内容 |
|---|------|
| **描述** | 3+个地方有相似的if/switch判断同一条件 |
| **为什么危险** | 新增类型时需要修改所有判断点，容易遗漏 |
| **怎么修** | 提取策略模式/多态。让每种类型自己处理自己的逻辑 |

### 3. 职责模糊

| | 内容 |
|---|------|
| **描述** | 无法用一句话说清楚一个类/模块的职责 |
| **为什么危险** | 边界不清→到处耦合→改一处动全局 |
| **怎么修** | 重新定义职责边界。如果一句话说不清，说明需要拆分 |

### 4. 循环依赖

| | 内容 |
|---|------|
| **描述** | A依赖B，B又依赖A |
| **为什么危险** | 无法独立理解/测试/复用任一模块。修改一个必须同时改另一个 |
| **怎么修** | 提取共享部分到C，让A和B都依赖C而非互相依赖 |

### 5. 依赖具体而非接口

| | 内容 |
|---|------|
| **描述** | 直接依赖具体实现类（如 `new MySQLDatabase()`） |
| **为什么危险** | 替换实现需要修改所有使用方。无法在测试中使用mock |
| **怎么修** | 依赖接口/抽象。通过构造函数/参数注入具体实现 |

## 三、流程异味信号

### 1. 跳过测试

| | 内容 |
|---|------|
| **描述** | "这次先不写测试了" |
| **为什么危险** | 技术债的起点。1次跳过→2次跳过→永远不写测试 |
| **怎么修** | 严格执行TDD。没有失败测试就不能写产品代码 |

### 2. 未review就合入

| | 内容 |
|---|------|
| **描述** | 代码直接push到main，不经过review |
| **为什么危险** | 第二双眼睛的价值在于发现作者因上下文盲区而遗漏的问题 |
| **怎么修** | 所有代码必须经过PR review。紧急hotfix可以缩短review时间，但不能跳过 |

### 3. 未理解需求就动手

| | 内容 |
|---|------|
| **描述** | 看到任务就开始写代码，没有确认理解是否正确 |
| **为什么危险** | 方向错了，代码越快越浪费。返工成本是初始成本的3-10倍 |
| **怎么修** | 动手前复述需求，与需求方确认理解一致。至少写一个测试来锚定期望行为 |

### 4. 大批量提交

| | 内容 |
|---|------|
| **描述** | 一次提交涉及10+个文件、500+行变更 |
| **为什么危险** | review困难、回滚困难、bisect困难。问题定位的时间与批量大小成正比 |
| **怎么修** | 小步提交。每个逻辑变更一个commit。每个commit可独立review和回滚 |

### 5. 在main上直接开发

| | 内容 |
|---|------|
| **描述** | 直接在main分支上写代码，不用功能分支 |
| **为什么危险** | 未完成的代码影响其他人。无法选择性合入。无法回滚 |
| **怎么修** | 使用功能分支开发。完成并review后再合并到main |

## 速查：看到信号时该做什么

| 信号 | 第一步 |
|------|--------|
| 过长函数 | 识别函数做了几件事，拆分 |
| 过深嵌套 | 找到guard clause机会 |
| 过多参数 | 分组为对象 |
| 神类 | 列出职责，按职责拆分 |
| 特征嫉妒 | 把行为移到数据旁边 |
| 魔法数字 | 提取常量 |
| 重复代码 | Rule of Three后提取 |
| 注释解释代码 | 重命名/提取函数 |
| 跳过测试 | 停下来，先写测试 |
| 未review就合入 | 创建PR，请求review |
| 未理解就动手 | 复述需求，确认理解 |
| 大批量提交 | 拆分为小步提交 |
