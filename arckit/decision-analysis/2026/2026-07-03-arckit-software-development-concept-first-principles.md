## 原始问题

本次要探讨的不是“Arckit 应该怎么实现”或“如何让某个 skill 被触发”这个局部问题，而是一个更上层的产品概念：

> 人在基于 Codex 类 Agent 完成软件研发全流程时，如何让不精确、混合、多义的自然语言 prompt，被系统性地拆解、路由、执行、沉淀，并持续回流到预期、实现、过程和工作流记忆中。

这意味着先要回答“Agent 参与软件研发时，人的表达、阶段产物、最终产物、事实来源和纠错到底是什么”。Arckit 相关内容只作为讨论中的参考材料，用来帮助暴露问题、举例说明或校验概念是否覆盖真实场景。

## 你实际表达的核心判断

### 1. Prompt 是多种本体的自然语言表达

人发给 Agent 的 prompt 不是一种单一对象。它可能同时指代：

- 需求。
- 讨论。
- 指导原则。
- 做事方式。
- 做事流程。
- 代码架构。
- 语言版本。
- 具体代码写法。
- 抽象想法。
- 不确定的灵感、试探或呓语。
- 对实现结果的纠错。
- 对 Agent 工作方式的纠错。

判断：这些表达都是真实输入，不能被系统预设为“只是一条任务指令”。

产品含义：这类产品必须先做 prompt 语义拆解，而不是直接把输入当作单一任务指令。

### 2. 最终产物从“代码”扩展为“能驱动系统运行或行为的产物”

从没有 AI Agent 的传统软件开发来看，软件研发活动的最终产物基本就是代码。没有代码，软件无法运行；代码是把需求、设计、架构和任务拆解转化为可执行系统的最终承载物。这是理解软件研发的根。

在 Codex 类 Agent 参与软件研发之后，这个判断不是被推翻，而是被扩展了：最终产物仍然可能是代码，但也可能是一组能驱动 Agent 行为、改变系统能力或稳定复用的软件化产物。

最终产物可能是：

- code。
- skill。
- prompt。
- workflow patch。

spec、tech solution、任务拆解、验证结果、运营素材等不应直接混入最终产物列表。它们是最终产物生产过程中需要的过程产物。过程产物不代表不重要，反而是让最终产物可被正确生产、验证、协作和复盘的关键中间物。

判断：传统软件研发的最终产物以代码为核心；AI Agent 时代的软件研发最终产物范围扩大，可以包括 skill、prompt、workflow patch 等能稳定驱动 Agent 或系统行为的产物。过程产物需要被明确区分，不应因为重要就被误归类为最终产物。

产品含义：这类产品需要从传统软件开发的“代码是最终运行物”出发，扩展出 AI Agent 时代的“行为驱动产物”概念；同时明确最终产物和过程产物的边界。

### 3. 人预期的阶段产物可能不是最终产物

人在一次 prompt 中期待的产物可能只是阶段性成果，例如：

- 理清楚需求。
- 形成产品决策。
- 做任务迭代拆解。
- 分发任务。
- 明确谁后续跟进。
- 只保留一个 pending 问题。
- 做一次验证或复盘。

判断：一次 prompt 可能对应一个阶段，也可能同时跨多个阶段。

产品含义：这类产品必须有清晰的研发阶段划分和阶段产物定义；系统应识别当前阶段，而不是默认进入实现。

### 4. 预期和实现是两类不同事实

你提出的关键区分是：

- `arckit/spec` 表达形成决策后的预期：要做什么、为什么做、验收口径是什么。
- code 表达当前实现现状：现在实际做到了什么。

判断：code 不能天然代表结果或预期；实现不一定满足预期。

产品含义：这类产品需要明确“预期事实源”和“实现事实源”的关系。实现完成后，如果过程思考和决策没有回流到预期文档，软件活动没有真正完成。

补充判断：预期事实源和实现事实源不是上下级关系，也不是谁取代谁的关系。它们是软件研发闭环里的两个必要参照：预期事实源提供方向和验收标准，实现事实源提供当前状态和可观察行为，验证与迭代发生在两者的比较和调整之间。

### 5. 真实来源不是单点，而是预期事实源和实现事实源的协同

如果把范围扩大到“软件本身”，软件的最终事实是它在目标环境中的可观察运行行为；代码、配置、依赖、环境、数据、prompt、skill 等共同生成这种运行行为。

但软件研发活动不只需要知道“现在实际运行成什么样”，还必须知道“本来预期应该是什么”。因此不能简单争论谁是唯一最终事实来源，而要区分两类同等重要的事实源：

- 预期事实源：表达软件应该做什么、为什么做、验收口径是什么、行为边界是什么。
- 实现事实源：表达软件现在实际做到了什么、在目标环境中实际如何运行。

SDD 中把 spec 作为事实来源，原因不是 spec 等于软件全部事实，而是 spec 是“预期事实”的来源。它防止当前实现被误当成正确预期，也让验证和纠偏有稳定参照。

实现事实源回答的是另一类问题：

- 如果问预期、目标、验收口径，真实来源应该是 spec。
- 如果问实际现状、当前行为、已实现能力，真实来源应该是 code、runtime behavior 和验证证据。
- 如果问过程假设、未决问题或未确认思考，真实来源可能是 pending 或 decision-analysis。
- 如果问 Agent 工作方式，真实来源可能是 workflow memory 或 skill contract。

判断：预期事实源和实现事实源都不能被省略。只保留 spec，会得到无法证明已经实现的愿望；只保留 code/runtime，会把当前实现误当成正确预期，丢失为什么做、应该做到什么程度和偏离时如何判断。

产品含义：软件研发产品需要定义多事实源协同模型。验证活动的核心就是比较预期事实和实现事实：当前实现是否符合预期，若不符合，是预期需要修正、实现需要修正，还是两者之间还缺少过程判断。

### 6. 代码既可能是过程工具，也可能是最终目标

人发现实现不符合预期后，可能直接改代码。这不代表代码成了预期，而是因为人无法提前完整表达，只能通过实践寻找答案。

判断：代码在同一轮活动中可能同时承担两种角色：

- 探索工具：通过实现试出真实需求或方案。
- 最终产物：项目最终要交付的运行系统。

产品含义：完成代码开发不等于软件活动完成。实现过程中形成的思考、决策、约束和预期变化，需要回流到对应的结果文档或 pending。

### 7. 人的纠错也是多义输入

人的纠错不是一种固定类别。它可能纠正：

- 需求。
- 预期。
- 实现。
- 架构。
- 阶段判断。
- 工作流。
- 验证方式。
- skill 内容。
- Agent 的理解方式。
- 是否需要沉淀。

判断：纠错要回到 prompt 多本体表达模型处理，不能只当作“请修改刚才的代码”。

产品含义：系统需要支持纠错信号拆解，区分事实纠错、工作流纠错、目标变化、补充上下文等类型。

### 8. Arckit 是当前讨论的参考素材，不是讨论目标

用户提到 Arckit，不是为了讨论 Arckit 本身，也不是为了从 Arckit 推导产品概念，而是因为当前项目提供了可引用的上下文材料：它能帮助说明人在真实 Agent 软件研发中遇到的表达、阶段、产物、事实来源和回流问题。

判断：文档记录 Arckit 相关内容时，应把它当作“参考材料”或“例子”，不能把它写成产品概念的主体、起点、目标或必然实现方式。

产品含义：讨论顺序应该是“记录用户关键思考 -> 分析思想和概念 -> 整理成通用产品概念”。是否映射到 Arckit，是后续实现层问题，不属于本轮概念文档的目标。

### 9. 产品价值是可演化假设，不是开工前的绝对前提

当人进入产品运营阶段时，运营目的和运营素材通常来自产品价值和产品功能。但产品价值本身不一定在软件研发开始前就已经清楚。

产品价值可能有几种状态：

- 一开始就相对清楚。
- 只有模糊方向，需要通过研发和用户反馈逐步验证。
- 迭代一段时间后才被真正找到。
- 经过多个版本后从原本假设中修正出来。
- 仍处于不确定状态，只能表达为候选价值或待验证假设。

判断：软件研发不能强约定“必须先有一个绝对正确的产品价值才能开工”，否则很多探索型产品无法启动；但完全没有产品价值也不行，否则研发和运营都没有方向。

产品含义：产品价值应该被视为可演化的研发对象。它既是研发前的方向假设，也是研发中的验证对象，还是运营阶段的素材来源。系统需要表达产品价值的稳定程度，而不是把它简单分成“有”或“没有”。

### 10. 裸 Agent 会把一句话压缩成实现请求，真实团队会把它展开成生命周期问题

假设没有加载任何 skill 包，人对 Agent 说“我要做一个待办软件”，裸 Agent 很可能会直接进入常见实现路径：

- 询问少量偏好，例如平台、技术栈、是否需要登录。
- 或者直接生成一个简单 Todo App。
- 默认补齐常见功能，例如新增、完成、删除、过滤、持久化。
- 把这句话理解成“请帮我开始做一个软件”或“请写一个可运行版本”。

这种反应不一定错。对于个人实验、原型或明确希望快速产出的场景，它可能正好有效。但它把一句话压缩成了实现任务，跳过了真实软件研发中大量可能存在的阶段判断。

如果放到真实团队的真实软件研发全生命周期，“我要做一个待办软件”首先不是完整需求，而是一个产品意图种子。它可能需要被展开成：

- 为什么要做待办软件，目标用户是谁。
- 市场上已有大量待办软件，新的价值假设是什么。
- 做给个人、团队、项目管理、AI 工作流，还是某个垂直场景。
- 当前要的是探索产品价值、定义 MVP、做原型、拆任务、做技术验证，还是直接实现。
- 最小可用版本需要哪些能力，哪些明确不做。
- 预期事实源如何形成，当前实现事实如何验证。
- 运营时要传播什么价值，素材来自哪些功能、场景和验证证据。

判断：一句简单 prompt 在裸 Agent 场景中容易被当作实现请求；在真实研发全生命周期中，它可能只是 idea、goal、value hypothesis、scope seed、prototype request 或 implementation request。系统不能只根据字面动作决定阶段，而要识别“当前人真正想推进到哪个阶段产物”。

产品含义：这类产品需要在不阻碍快速实现的前提下，保留生命周期展开能力。它既要允许“先做一个原型看看”，也要能识别“这句话还没有足够产品价值、用户、范围和验收信息，当前更应该进入探索或定义”。

### 11. 真实软件预期和单轮有限交付之间存在天然张力

AI 时代的人机协同里，很多软件研发活动默认就是“人和 Agent 一起做”。只有在需要更多角色、权限、领域知识或组织协同时，人才会再找其他团队成员参与。

当人说“我要做一个待办软件”时，最终目标通常不是一个玩具回答，而是真实世界中的真实软件。即使当前只是一个模糊产品概念，背后也指向真实软件开发预期：真实用户、真实场景、真实功能、真实运行环境、真实运营和真实迭代。

但 Agent 的单轮工作受到上下文长度、时间、成本、工具权限、信息完整度和当前确认程度限制。它只能做到某个程度：可能是概念澄清、价值假设、MVP 范围、原型、任务拆解、局部实现、验证报告，或者一个可运行但不完整的版本。

判断：有限交付不是错误，模糊表达也不是错误。它们都是人机协同软件研发的基本事实。问题在于，如果系统不显式区分“真实世界最终目标”和“本轮可达阶段产物”，Agent 就容易把阶段产物误说成最终完成，或把真实软件预期压缩成一次性实现任务。

产品含义：这类产品需要同时记录两层目标：一层是真实世界的软件预期，一层是本轮在限制条件下可交付的阶段产物。每轮结束时应说明当前产物离真实软件预期还有哪些差距、哪些预期已被确认、哪些还只是候选或 pending。

### 12. 是否沉淀本身不是可靠二元判断，需要不确定性处理机制

如果有了这样的系统，并不意味着每次对话都应该机械地产生文件。每次对话都应该做“是否需要沉淀”的判断，但这个判断本身就是问题的一部分。

Agent 不应该假装自己总能分清哪些内容该存、哪些内容不该存。人的表达经常是半成型的：可能是灵感、倾向、纠错、疑问、暂时判断、过程想法，也可能后来才被证明是关键决策。如果过度沉淀，会污染正式事实源；如果完全不沉淀，会造成重要信息丢失。

因此沉淀不应该是简单的“存 / 不存”二元判断，而应是多态决策：

- 明确不存：临时闲聊、无新增信息的确认、用户明确说不要记录。
- 只在聊天中总结：有帮助但暂时不需要进入文件系统。
- 保存原始输入：先保留原话，不提升为结论。
- 存为 pending：有潜在价值但尚未确认。
- 存为 decision-analysis：需要继续分析的问题或判断。
- 存为候选 spec / tech / workflow：可能成为稳定事实，但还需要确认。
- 存为正式事实：已经形成清晰结论、规范、预期或决策。
- 请求用户确认：Agent 无法判断风险或作用域时，先问人。
- 暂时缓冲：上下文不足时先保留在本轮总结或候选区，等待后续信号。

判断：沉淀机制的目标不是让 Agent 一次性准确分类所有内容，而是在“信息丢失”和“事实污染”之间建立可追踪、可回滚、可提升的中间层。

产品含义：产品概念里必须包含沉淀不确定性模型。正式事实源要保持干净，但系统也要有低承诺空间承接模糊信息，后续再通过确认、重复出现、验证结果或用户纠正逐步提升。

## 重新定义后的产品问题

这类产品不应被定义为“技能集合”或“任务路由器”。更准确的问题定义是：

> 如何构建一个面向 Codex 类 Agent 软件研发活动的语义编译和事实沉淀系统：把人的自然语言 prompt 拆解成真实世界软件预期、当前可达阶段、产物类型、事实来源、价值假设、执行动作、验证要求和沉淀目标，并用可追踪、可回滚、可提升的沉淀机制，在最终产物、阶段产物、产品价值和运营素材之间形成闭环。

## 底层概念模型

### Prompt 表达层

人输入的是自然语言，不保证清晰、完整、稳定或单义。

需要支持的信号类型：

- 意图种子信号。
- 真实软件预期信号。
- 当前可达阶段信号。
- 目标信号。
- 需求信号。
- 价值假设信号。
- 约束信号。
- 决策信号。
- 实现信号。
- 验证信号。
- 过程信号。
- 纠错信号。
- 工作流偏好信号。
- 未确认想法信号。
- 沉淀意图信号。
- 沉淀不确定性信号。

### 研发阶段层

同一 prompt 可能落入一个或多个阶段：

- intake：记录原始输入。
- explore：探索想法和问题。
- decide：形成判断或方向。
- define：形成预期和验收口径。
- design：形成交互、视觉或技术方案。
- plan：拆解任务、迭代和分发。
- implement：产出最终或实验性实现。
- verify：验证实现和预期的关系。
- review：审查质量、风险和偏差。
- value-discovery：通过实现、反馈、使用数据或运营观察发现和修正产品价值。
- deliver：发布、同步或交付。
- operate：围绕产品价值和产品功能组织运营目标、素材、反馈和运行观察。
- remember：沉淀事实、pending 或 workflow memory。

### 产物层

产物至少分为：

- 最终产物：code、skill、prompt、workflow patch、交付包等能让软件或 Agent 系统实际运行、执行或稳定改变行为的产物。
- 过程产物：spec、tech solution、任务拆解、验收口径、目标、决策、decision-analysis、draft、architecture handoff、exploration 等生产最终产物所需的中间产物。
- 预期产物：spec、验收口径、目标、决策等表达“希望做成什么”的过程产物。
- 实现产物：代码、配置、skill 文件、脚本等实际承载实现的产物；其中一部分可能就是最终产物。
- 未决产物：pending。
- 价值产物：产品价值假设、价值判断、价值验证证据、运营定位。
- 工作流产物：workflow signal、candidate、accepted patch。
- 验证产物：测试结果、review 结果、验证报告。
- 运营产物：运营目标、运营素材、用户反馈、转化证据。
- 沉淀产物：原始输入记录、conversation digest、候选事实、低承诺 pending、正式事实。

### 事实来源层

不同问题对应不同 source of truth：

| 问题类型 | 真实来源 |
|---|---|
| 预期是什么 | 预期文档，例如 `arckit/spec` |
| 当前实现是什么 | code 或对应实现 artifact |
| 技术方案是什么 | 技术方案文档，例如 `arckit/tech` |
| 交互或视觉预期是什么 | 交互/视觉预期文档，例如 `arckit/interaction`、`arckit/visual` |
| 产品价值是什么 | 价值判断、产品规格、决策分析或运营定位文档，取决于稳定程度 |
| 产品价值是否成立 | 用户反馈、使用数据、运营结果、验证记录 |
| 运营素材来自哪里 | 已确认或候选的产品价值、产品功能、用户场景和使用证据 |
| 还没决定什么 | pending 空间，例如 `arckit/pending` |
| 为什么做这个判断 | decision-analysis 或治理 Decision |
| Agent 以后应该怎么做 | workflow memory 或 skill contract |
| 本轮实际做了什么 | execution record、git diff、验证记录 |

## 关键产品决策草案

### 决策 1：核心输入不是任务，而是 prompt 中的多本体信号

系统不能把用户输入先简化成“要执行什么任务”。它应先拆解 prompt 中可能同时存在的目标、预期、事实、实现、纠错、过程和工作流信号。

### 决策 2：最终产物从代码扩展到 AI Agent 行为驱动产物

传统软件研发的最终产物以代码为核心，因为软件依靠代码运行。AI Agent 时代，最终产物可能仍是代码，也可能是 skill、prompt、workflow patch 等能稳定驱动 Agent 行为的产物。spec、任务拆解、验证结果等是生产最终产物的过程产物，不应与最终产物混淆。

这类产品的系统价值在于维护从过程产物到最终产物，再从实现结果回流到预期、价值和事实来源的闭环。

### 决策 3：Spec 表达预期，code 表达现状

预期文档不应被 code 替代。code 可以帮助探索预期，也可以实现预期，但不能天然代表预期。实现和预期之间必须允许存在偏差，并通过验证和回流机制处理。

### 决策 3.1：预期事实源和实现事实源同等重要，不能只保留其一

软件研发不是寻找一个全局唯一事实源，而是建立多事实源的分工和协同。预期事实源负责表达应该是什么，实现事实源负责表达现在是什么。验证、纠错和迭代依赖两者的比较。只保留预期会失去实现证据，只保留实现会丢失目标和判断标准。

### 决策 4：实现完成后仍需要回流判断

当实现活动完成时，系统必须判断是否有新的预期、决策、约束、验证结论或 pending 需要沉淀。否则软件活动只是完成了局部执行，没有完成研发闭环。

### 决策 5：Skill 是软件产物

维护 skill 应进入与代码类似的软件研发流程：需求理解、决策、实现、验证、治理、同步和回流。不能把 skill 维护降级成单文件编辑。

### 决策 6：当前文档只沉淀产品概念，不讨论具体承载方式

在产品概念层面，目标是定义“Agent 软件研发活动的语义编译与事实回流系统”。当前文档只负责记录、分析和整理这个概念；Arckit、skills、workflow memory、spec/tech/pending 等都只是讨论中可能出现的参考材料或后续实现选项。

### 决策 7：产品价值应作为可演化对象参与研发闭环

产品价值不是开工前必须完全正确的静态前提，也不是可以完全缺失的空白项。它应以不同稳定程度存在于研发活动中：从模糊方向、候选假设、待验证价值，到逐步确认的产品定位。研发、验证和运营都可能反过来修正产品价值。

### 决策 8：运营不是研发之外的独立末端

运营目的和运营素材来自产品价值、产品功能、用户场景和验证证据。产品运营会反过来提供反馈，帮助判断价值是否成立、功能是否匹配预期，以及下一轮研发应该调整什么。因此运营应被纳入软件研发全流程的反馈和事实回流模型。

### 决策 9：简单产品意图不能默认等同于实现任务

“我要做一个待办软件”这类表达在不同上下文中可以指向不同阶段：产品探索、价值假设、MVP 定义、原型、任务拆解、技术验证或直接实现。系统应先判断用户当前期待的阶段产物；当用户明确要快速产出时可以直接实现，但不能把所有产品意图都默认压缩成写代码。

### 决策 10：每轮协作必须区分真实软件预期和本轮阶段产物

人的最终目标通常指向真实世界的真实软件，但 Agent 单轮协作只能在上下文、时间、成本和工具边界内推进到某个阶段。系统应把真实软件预期作为长期目标保留，同时把本轮产物标注为阶段性结果，并说明它与真实软件预期之间的差距和下一步。

### 决策 11：沉淀机制必须承认 Agent 无法总是分清该存什么

系统不能把沉淀设计成简单的存/不存分类器。对明确事实可以写入正式事实源，对明确无价值内容可以不存；大量模糊内容应进入低承诺空间、候选区、pending、decision-analysis 或等待用户确认。沉淀的核心能力是支持信息从模糊到稳定的逐步提升，而不是一次性判断正确。

## 需要继续拷问的问题

1. `arckit/spec` 是否只表达产品预期，还是也表达 skill 系统本身的预期？
2. skill/prompt/workflow patch 的预期应写入 `spec`、`tech`，还是需要独立 artifact domain？
3. “实现探索”产生的代码如何标记为探索工具，而不是已接受实现？
4. 什么时候必须把代码实践回流到 spec，什么时候只需要 pending？
5. workflow memory 和 skill contract 的边界是什么？
6. 人的纠错中，哪些属于项目事实纠错，哪些属于 workflow 纠错？
7. “软件研发活动完成”的最小判断标准是什么？
8. 这类产品是否需要一个统一的 artifact ontology，还是由运行时 frame 动态判断即可？
9. 讨论中引用的具体项目、工具或实现材料，哪些只是例子，哪些表达了更通用的产品概念？
10. 产品价值处于模糊、候选、待验证、已确认状态时，分别允许进入哪些研发阶段？
11. 没有绝对正确产品价值时，什么样的最小价值假设足以支持开工？
12. 运营素材应该从哪些阶段产物中生成，运营反馈又应该回流到哪里？
13. 当用户只说“我要做一个 X 软件”时，系统应如何判断当前阶段是探索、定义、原型、任务拆解还是实现？
14. 如何在不阻碍快速原型的前提下，避免把所有产品意图都误判为直接实现任务？
15. 每轮 Agent 协作结束时，如何表达“这只是阶段产物，不等于真实软件已经完成”？
16. 在上下文、时间和成本限制下，如何选择当前最有价值的阶段产物？
17. 沉淀判断不确定时，哪些内容应进入低承诺保存，哪些必须请求用户确认？
18. 如何防止过度沉淀污染事实源，同时避免重要思考丢失？
19. 候选信息通过什么条件提升为正式事实：用户确认、重复出现、验证证据，还是实现结果？

## 当前建议

先接受以下产品概念作为讨论起点：

> 要探讨的产品概念是“Agent 软件研发活动的语义编译与事实回流系统”。它将人的自然语言 prompt 拆解为真实软件预期、当前可达阶段、产物类型、事实来源、价值假设、执行动作、验证关系和沉淀目标，并通过低承诺保存、候选提升和正式事实源协同，支持 code、skill、prompt、运营素材等不同产物在同一套研发生命周期中被处理。

下一步不应直接讨论 Arckit 怎么实现或直接改全部 skill，而应继续围绕你的关键思考拆清楚三类结果：

1. 产品概念：定义 Agent 软件研发中的真实软件预期、当前可达阶段、最终产物、过程产物、产品价值状态、沉淀不确定性和 source of truth 边界。
2. 系统模型：定义 prompt-signal decomposition、artifact ontology、fact routing、价值验证和回流闭环。
3. 参考材料处理规则：说明讨论中出现的 Arckit、skills、code、spec 等内容只是用于举例和校验概念，不能提前变成讨论目标。
